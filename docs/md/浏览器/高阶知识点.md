# 浏览器高阶知识点

## 面试定位

这一章是 **加分项**，不是三年岗开场必答。基础渲染、缓存、事件循环答稳后，用 1～2 个点体现「理解浏览器整体架构」，并 **绑定项目场景**（卡顿、首屏、安全、跨站登录）。

切忌空泛堆术语：多进程、V8、HTTP/3 各说一两句即可，除非面试官深挖。

---

## 面试回答

### 总答：你怎么理解「现代浏览器」而不只是「跑 JS 的环境」？

> 我会说浏览器是多进程协作的系统：Browser 进程管窗口和调度，Renderer 进程负责页面解析、JS、布局和绘制，Network/GPU 等分工处理请求和合成。
>
> 但 **JavaScript 仍在渲染进程主线程上单线程执行**，所以「浏览器多进程」和「JS 单线程」不矛盾——卡顿说的是主线程被长任务占满。
>
> 性能上除了减 JS，还可以用 preload/preconnect 优化关键资源；安全上有站点隔离和沙箱限制渲染进程越权。这些我都会结合项目讲，而不是单独背概念。

一句话总结：

> 浏览器 = 多进程隔离 + 渲染主线程跑 JS；高阶题答「架构目的 + 项目挂钩」。

---

### 题 1：浏览器有哪些进程？各干什么？

> 常见有 **Browser**（总控）、**Renderer**（页面、JS、layout/paint）、**Network**（请求）、**GPU**（合成加速）、**Plugin/Extension**（隔离扩展）。
>
> 好处是 **稳定性**（一个 tab 崩不全崩）、**安全**（站点隔离）、**性能**（网络/GPU 不堵主线程）。三年岗说到 Renderer 主线程即可，不必背 Chromium 全部进程类型。

---

### 题 2：站点隔离和沙箱是什么？

> **站点隔离**：尽量把不同站点放到不同 Renderer 进程，降低 Spectre 类跨站读内存风险。
>
> **沙箱**：限制渲染进程直接访问文件系统、设备等系统资源，恶意页面难以越权。
>
> 面试表达：**降低攻击面、把风险关在进程里**；不必展开内核细节。

---

### 题 3：V8 怎么执行 JavaScript？（简述版）

> 源码 **解析成 AST** → 生成 **字节码** 由解释器执行 → **热点代码** 交给优化编译器（如 TurboFan）生成机器码 → 假设失效可能 **反优化**。
>
> 结论：**不是简单逐行解释**，而是解释 + JIT 优化。这和「为什么有些代码突然变快/变慢」有关，但三年岗很少要求背 IR 细节。

```text
源码 → AST → 字节码（Ignition）
              ↓ 热点
         优化机器码（TurboFan）
              ↓ 类型变化等
           反优化回字节码
```

---

### 题 4：`preload`、`prefetch`、`preconnect` 区别？

> **`preload`**：当前页 **马上需要** 的高优先级资源（如关键 CSS、字体），会抢占带宽，别滥用。
>
> **`prefetch`**： **未来导航** 可能用的低优先级资源，空闲时预取。
>
> **`preconnect`**：提前完成 DNS + TCP + TLS，适合已知跨域 CDN 源。
>
> **`dns-prefetch`**：只做 DNS 解析，比 preconnect 更轻。

```html
<link rel="preconnect" href="https://cdn.example.com" />
<link rel="preload" href="/critical.css" as="style" />
<link rel="prefetch" href="/next-route.js" as="script" />
```

---

### 题 5：HTTP/3 和 QUIC 解决了什么问题？

> HTTP/2 多路复用仍受 **TCP 队头阻塞** 影响：一个包丢失，同连接上其他流也要等。
>
> **QUIC 基于 UDP**，在传输层实现可靠传输和多路流，**单流丢包不阻塞其他流**；还支持 **连接迁移**（如 Wi-Fi 切 4G）。
>
> 不要说「UDP 不可靠所以更快」——QUIC 在 UDP 之上做了可靠性和加密（常配合 TLS 1.3）。

---

### 题 6：第三方 Cookie 收紧对你项目有什么影响？

> 跨站跟踪和嵌入场景下，**`SameSite`**、第三方 Cookie 限制、**分区存储** 让 iframe 里默认拿不到以前能拿的 Cookie。
>
> 登录态、SSO、嵌入式 widget 可能要改 **同源代理**、**Storage Access API** 或 **token + postMessage** 方案。适合在「跨域登录 / 嵌入页」项目里主动提。

## 核心原理

### 多进程 vs JS 单线程

| 层面 | 模型 | 面试要点 |
| --- | --- | --- |
| 浏览器整体 | 多进程、多线程协作 | 隔离、稳定、安全 |
| 单个 Renderer | 主线程跑 JS、DOM、大部分 layout/paint | 长任务 → 卡顿 |
| 合成 | 部分工作可在合成线程 | `transform` 动画更顺滑 |

### V8 执行（三年岗深度）

1. **Parser**：源码 → AST  
2. **Ignition**：字节码解释执行，快速启动  
3. **TurboFan**：热点函数优化编译  
4. **Deopt**：类型或结构变化导致优化假设失败  

不必背 opcode；知道 **Hidden Class** 影响属性访问性能即可作加分：同一形状的对象属性访问更快。

### 资源提示优先级（概念）

`preload` > 正常解析发现 > `prefetch`。错误 preload 会浪费带宽、拖慢 LCP。

### HTTP 版本与队头阻塞

| 版本 | 传输 | 备注 |
| --- | --- | --- |
| HTTP/1.1 | TCP | 队头阻塞（连接/请求级） |
| HTTP/2 | TCP | 多路复用，仍受 TCP 丢包阻塞 |
| HTTP/3 | QUIC(UDP) | 流级隔离，改善丢包阻塞 |

### 隐私与存储趋势（了解）

- `SameSite=Lax/Strict` 限制跨站 Cookie 发送。  
- 第三方 Cookie 逐步淘汰 → 广告归因、跨站登录需新方案。  
- **Partitioned Storage**：第三方上下文存储按顶级站点分区。

## 高频追问

### 多进程会不会让 JS 变快？

不会直接让单线程 JS 更快；收益在隔离和并行 I/O/GPU。JS 性能仍靠算法、少阻塞、Worker 等。

### preload 和 prefetch 能同时用在同一 URL 吗？

技术上可以但不推荐；语义冲突且浪费优先级，按「当前页 vs 未来页」二选一。

### 0-RTT 是什么？有什么风险？

QUIC/TLS 1.3 可在握手完成前发数据，降低延迟；但有 **重放攻击** 风险，敏感操作不应依赖 0-RTT。
