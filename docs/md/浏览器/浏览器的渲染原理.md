# 浏览器的渲染原理

## 面试定位

渲染原理是三年岗 **必考基础**，很少单独出「背流程」题，而是和性能优化、卡顿排查、动画方案绑在一起问。

常见开场：从 HTML/CSS/JS 到屏幕像素发生了什么？CSS 和 JS 为什么会阻塞？重排、重绘、合成有什么区别？为什么动画推荐 `transform`？

答这类题的关键：**先给可口述的结论，再补链路和项目验证**，不要一上来堆 DOM/CSSOM 名词。

---

## 面试回答

### 总答：浏览器是怎么把页面画出来的？

> 我会按一条流水线来讲：解析 HTML 得到 DOM，解析 CSS 得到 CSSOM，两者合成渲染树，再 layout 算位置尺寸，paint 画像素，最后 composite 把图层合成到屏幕。
>
> 性能上我会重点说阻塞和代价：CSS 会阻塞渲染，因为没 CSSOM 就算不出最终样式；普通 script 会阻塞 HTML 解析，因为脚本可能改 DOM。优化时我尽量减首屏阻塞资源，动画优先用 `transform` 和 `opacity` 走合成，避免频繁触发 layout。
>
> 项目里如果列表卡顿，我会用 Performance 看 Long Task 和 Layout，排查是不是读写布局属性交错、DOM 批量更新没做好。

一句话总结：

> 渲染 = DOM + CSSOM → Render Tree → Layout → Paint → Composite；性能优化核心是少阻塞、少 layout、动画走合成。

---

### 题 1：重排、重绘、合成有什么区别？

> **重排（reflow/layout）** 是几何信息变了，要重新算布局，代价最高，还可能引发后续重绘。
>
> **重绘（repaint）** 是外观变了但尺寸位置不变，比如改颜色，只重新绘制，不做完整 layout。
>
> **合成（composite）** 是在已有图层上改组合方式，比如 `transform`、`opacity` 动画，通常不触发布局，由合成线程完成，主线程压力小。
>
> 所以动画我会优先 `transform`，避免改 `width`、`top` 这类触发布局的属性。读布局属性前也要避免和写样式交错，否则会强制同步 layout。

---

### 题 2：CSS 和 JavaScript 如何阻塞渲染？

> **CSS 阻塞渲染**：浏览器要等 CSSOM 才能确定最终样式，否则会出现 FOUC（无样式内容闪烁）。所以关键 CSS 要尽量小、尽量早到。
>
> **JS 阻塞解析**：默认 `<script>` 会暂停 HTML 解析去执行脚本，因为脚本可能 `document.write` 或改 DOM。非关键脚本用 `defer`（下载并行、DOM 解析完再按序执行）或 `async`（下载完立刻执行、不保证顺序）。
>
> 三年岗不用背所有细节，但要能说出：**阻塞的是主线程上的解析/渲染节奏，优化目标是缩短关键路径。**

---

### 题 3：`display: none` 和 `visibility: hidden` 有什么区别？

> `display: none` 不参与渲染树，不占布局空间，也不会触发点击。
>
> `visibility: hidden` 仍在渲染树里，占空间，只是不可见。
>
> 性能上两者都可能避免不必要的绘制，但 `display: none` 连 layout 都不参与；做切换动画时选型会影响是否触发布局。

---

### 题 4：读取 `offsetWidth` 为什么可能导致卡顿？

> 如果前面有未应用的样式修改，读 `offsetWidth`、`getBoundingClientRect()` 等布局属性会迫使浏览器**立刻完成 layout**，这叫强制同步布局（layout thrashing）。
>
> 循环里「写样式 → 读布局 → 再写」会把 layout 放大很多倍。正确做法是**先批量读、再批量写**，或用 `requestAnimationFrame` 合并 DOM 更新。

## 核心原理

### 渲染主流水线

```text
HTML 解析 → DOM
CSS 解析 → CSSOM
        ↘
         Render Tree（仅可见节点）
              → Layout（几何）
              → Paint（绘制指令）
              → Composite（图层合成 → 屏幕）
```

渲染树会跳过 `display: none` 的节点；`visibility: hidden` 仍保留在树中并占位。

### DOM 与 CSSOM

DOM 描述文档结构，CSSOM 描述样式规则、继承与层叠后的计算结果。两者都就绪后，浏览器才知道「哪些节点、以什么样式」展示。

```html
<div class="card"><h1>Hello</h1></div>
```

```css
.card { width: 200px; color: red; }
```

### 阻塞行为

| 资源 | 阻塞什么 | 常见优化 |
| --- | --- | --- |
| CSS | 渲染（需要 CSSOM） | 关键 CSS 内联/提前、减小体积 |
| 同步 script | HTML 解析 | `defer` / `async`、底部加载、拆包 |
| 图片/字体 | 一般不阻塞解析 | `preload`、合适格式、子集化 |

```html
<!-- 推荐：不阻塞解析，DOM 就绪后按序执行 -->
<script src="/main.js" defer></script>

<!-- 独立脚本、无顺序要求时用 async -->
<script src="/analytics.js" async></script>
```

### Layout / Paint / Composite

**Layout** 计算盒模型、位置。常见触发：改宽高、margin、font-size、增删 DOM、resize。

**Paint** 填充颜色、文字、阴影等。改 `color`、`background` 等可能只重绘。

**Composite** 把分层结果合成。`transform`、`opacity` 常可只走合成层；滥用 `will-change` 会增加内存，应 targeted 使用。

```css
/* 合成友好动画 */
.box {
  transform: translateX(100px);
  opacity: 0.8;
}
```

### 强制同步布局示例

```javascript
// 不推荐：读写交错，每轮都可能 layout
items.forEach((item) => {
  item.style.width = "100px";
  console.log(item.offsetWidth);
});

// 更好：先读后写
const widths = items.map((item) => item.offsetWidth);
items.forEach((item, i) => {
  item.style.width = `${widths[i] + 10}px`;
});
```

### 项目中的优化清单

- 减少首屏阻塞 CSS/JS 体积，路由级代码分割。
- 长列表虚拟滚动，降低 DOM 与 layout 规模。
- 动画用 `transform` / `opacity` + `requestAnimationFrame`。
- 用 Performance / Lighthouse 验证 LCP、Long Task、Layout Shift。

## 高频追问

### `will-change` 能随便加吗？

不能。它会提示浏览器提前分层，可能提升动画，但增加内存与合成成本；只在确有需要时使用，动画结束后移除。

### 重绘一定比重排便宜吗？

不一定。大范围重绘仍占主线程；且重排往往连带重绘。面试答「合成 < 重绘 < 重排」是合理简化，但要承认具体代价看影响范围和浏览器优化。

### 事件循环和渲染谁先谁后？

一轮宏任务执行完、微任务清空后，浏览器可能在下一宏任务前进行渲染（若需要）。所以微任务里的 DOM 修改可能在本帧 paint 前生效——这也是 `Promise.then` 里改 DOM 常能立刻看到的原因之一。
