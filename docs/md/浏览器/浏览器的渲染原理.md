# 浏览器的渲染原理

从下载 HTML 到屏幕上出现像素，浏览器大致经历：**解析 → 样式计算 → 布局 → 分层 → 绘制 → 合成**。理解这条链路有助于做性能优化与排障。

## 1. 解析阶段：DOM 与 CSSOM

**HTML → DOM**  
浏览器自上而下解析 HTML，遇到标签就构建 DOM 树；遇到 `<script>` 且未声明 `defer`/`async` 时，会**阻塞**后续 HTML 解析（除非 `async` 或放到底部等策略）。因此首屏常会强调脚本加载顺序与拆分。

**CSS → CSSOM**  
样式表解析成 CSSOM。CSS 默认也会阻塞渲染（外部 CSS 未完成时，浏览器往往避免绘制「无样式内容」造成严重闪烁）。`media` 不匹配的外链 CSS 可能以较低优先级加载。

DOM 与 CSSOM 共同决定最终样式规则的应用结果。

## 2. 渲染树（Render Tree）

在 DOM 上附着计算后的样式，生成**渲染树**（一般忽略 `display: none` 等不可见节点，`visibility: hidden` 仍会占位）。

这一步可以理解为：哪些节点要画、各自长什么样。

## 3. 布局（Layout / Reflow）

根据视口宽度、盒模型、浮动/定位/弹性/网格等，计算每个节点在视口中的**几何信息**（位置、大小）。  
依赖几何信息的属性变化（如宽度、高度、位置、字体大小）可能触发**重新布局**，成本较高。

## 4. 绘制（Paint）

把元素的视觉表现（颜色、边框、阴影、文字等）拆成**绘制指令**。  
通常只影响外观、不改变几何的修改（例如 `color`、`background-color`，在部分情况下）可能只触发展示层的**重绘**而跳过完整布局（具体仍受浏览器实现与属性影响，面试答「可能重绘、严重变更仍会触发布局」更稳妥）。

## 5. 合成（Composite）

现代浏览器会将页面拆成多个**合成层**（层叠上下文、`transform`/`opacity`、固定/粘性定位等常促使提升为独立层），由 GPU 参与**合成**成最终帧。  
合理使用**合成层**（例如用 `transform` 做动画代替不断改 `top/left`）有时能避开大面积重布局与重绘。

## 6. 关键渲染路径（CRP）小结

一条常见的首帧路径可概括为：

1. 解析 HTML 构建 DOM（期间受脚本、CSS 阻塞策略影响）。  
2. 解析 CSS 构建 CSSOM。  
3. 生成渲染树 → Layout → Paint → Composite。  

**优化直觉**：减少阻塞资源体积与次数、尽早可交互、避免大段同步脚本、控制布局抖动、动画尽量走合成友好属性。

## 7. 重排与重绘（面试高频）

- **重排（Reflow）**：布局信息变化，需要重新计算几何，可能影响后续绘制与合成。  
- **重绘（Repaint）**：外观变化，但不一定需要全量重排（视浏览器与属性而定）。  
- **减轻策略**：批量读写样式（减少强制同步布局）、使用 `requestAnimationFrame`、虚拟列表、避免在循环里读 `offsetHeight` 再连续写样式等。
