# 浏览器的事件循环

## 面试定位

事件循环是浏览器高频题，常和 Promise、`setTimeout`、页面卡顿、Vue `nextTick`、React 状态更新一起考。回答重点是：JavaScript 单线程、任务队列、宏任务、微任务、渲染时机。

## 核心原理

浏览器主线程负责执行 JavaScript、处理用户交互、计算样式、布局和绘制。JavaScript 是单线程执行的，如果同步任务一直占用主线程，页面就无法响应点击、滚动和渲染。

事件循环的作用是协调同步代码、异步回调和渲染流程。

一次典型循环可以概括为：

1. 执行一个宏任务。
2. 执行过程中遇到同步代码，直接进入调用栈。
3. 遇到异步任务，把回调放入对应任务队列。
4. 当前宏任务结束后，清空所有微任务。
5. 浏览器根据需要执行渲染。
6. 取下一个宏任务继续执行。

## 宏任务和微任务

常见宏任务：

- `script` 整体代码
- `setTimeout`
- `setInterval`
- 用户交互事件
- 网络 I/O 回调
- `MessageChannel`

常见微任务：

- `Promise.then`
- `Promise.catch`
- `Promise.finally`
- `queueMicrotask`
- `MutationObserver`

微任务的优先级高于下一个宏任务。当前宏任务执行完后，会先清空微任务队列，再进入下一个宏任务。

```javascript
console.log(1);

setTimeout(() => {
    console.log(2);
}, 0);

Promise.resolve().then(() => {
    console.log(3);
});

console.log(4);

// 1 4 3 2
```

## Promise 的执行顺序

Promise 构造函数里的 `executor` 是同步执行的，`then` 回调才是微任务。

```javascript
console.log("start");

new Promise((resolve) => {
    console.log("executor");
    resolve();
}).then(() => {
    console.log("then");
});

console.log("end");

// start
// executor
// end
// then
```

## 微任务嵌套

清空微任务队列时，如果微任务里又产生新的微任务，也会继续执行，直到队列为空。

```javascript
Promise.resolve().then(() => {
    console.log("A");
    Promise.resolve().then(() => {
        console.log("B");
    });
});

setTimeout(() => {
    console.log("C");
}, 0);

// A B C
```

这也是为什么大量微任务可能饿死宏任务，导致页面迟迟没有机会处理用户事件或渲染。

## 渲染时机

浏览器通常会在一次宏任务结束、微任务清空后，判断是否需要渲染。不是每一轮事件循环都一定渲染，浏览器会结合刷新率、页面状态和优化策略决定。

`requestAnimationFrame` 的回调会在浏览器下一次绘制前执行，适合做动画相关的读写。

```javascript
requestAnimationFrame(() => {
    // 下一帧绘制前执行
});
```

如果一个长同步任务执行很久，渲染会被阻塞，用户看到的就是卡顿。

```javascript
const start = Date.now();
while (Date.now() - start < 3000) {}
// 3 秒内主线程被占满，页面无法响应
```

## 和 Vue nextTick 的关系

Vue 会把多次状态更新合并，异步刷新 DOM。`nextTick` 用来等本轮 DOM 更新完成后再执行回调，底层通常会优先使用微任务。

```javascript
state.count++;

await nextTick();
// 此时再读取 DOM，能拿到更新后的结果
```

面试里可以说：`nextTick` 不是“下一秒”，而是等当前同步状态修改触发的 DOM 更新批处理完成。

## 面试回答

可以这样答：

> 浏览器的事件循环用来协调 JavaScript 执行、异步回调和页面渲染。JavaScript 在主线程上单线程执行，同步代码会先进入调用栈。异步任务完成后，回调会进入任务队列。任务分为宏任务和微任务，当前宏任务执行完后，会先清空所有微任务，然后浏览器再根据需要进行渲染，最后进入下一个宏任务。`Promise.then` 属于微任务，`setTimeout` 属于宏任务，所以 Promise 回调通常会先于定时器执行。理解事件循环也能解释页面卡顿：如果长任务一直占用主线程，浏览器就没有机会响应用户事件和绘制页面。

## 高频追问

### 为什么 `setTimeout(fn, 0)` 不是立刻执行？

`setTimeout` 只是把回调放入宏任务队列，必须等当前宏任务和所有微任务执行完，事件循环取到它时才会执行。

### 微任务太多会有什么问题？

微任务会在进入下一个宏任务前被清空，如果持续产生微任务，可能导致用户事件和渲染迟迟无法执行，造成卡顿。

### `requestAnimationFrame` 和 `setTimeout` 做动画有什么区别？

`requestAnimationFrame` 会在浏览器绘制前执行，节奏和屏幕刷新更匹配；`setTimeout` 只按时间入队，不保证和渲染帧对齐。
