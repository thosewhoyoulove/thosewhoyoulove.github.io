# 浏览器的缓存机制

## 面试定位

缓存是三年岗 **网络 + 性能** 交叉高频题。面试官想确认：线上静态资源为什么能秒开、发版后用户如何拿到新版本、304 是什么意思、强缓存和协商缓存怎么配。

不要只背响应头名词，要能讲清 **浏览器决策顺序** 和 **HTML / 带 hash 静态资源 / API** 三类资源的策略差异。

---

## 面试回答

### 总答：浏览器缓存怎么工作？

> 我会分成强缓存和协商缓存两层。
>
> **强缓存**命中时，浏览器直接用本地副本，**不发请求**，主要看 `Cache-Control`（优先）和 `Expires`。
>
> **强缓存过期后**走协商缓存：浏览器带 `If-None-Match`（ETag）或 `If-Modified-Since`（Last-Modified）问服务器；资源没变返回 **304**，正文不传输，继续用本地；变了返回 **200** 和新资源。
>
> 项目里我的习惯是：**HTML 用 `no-cache`**，保证入口能更新；**带 contenthash 的 JS/CSS/图片长缓存**；**用户数据和敏感接口用 `no-store`**。

一句话总结：

> 强缓存省请求，协商缓存省带宽；策略按资源类型分，HTML 入口与静态资源分离治理。

---

### 题 1：强缓存和协商缓存的区别？

> **强缓存**：在有效期内，浏览器认为资源仍新鲜，不发起 HTTP 请求，Network 面板常显示 `(disk cache)` 或 `(memory cache)`。
>
> **协商缓存**：必须发请求，由服务器判断资源是否变化；不变则 304，变则 200 带新 body。
>
> 顺序永远是：**先查强缓存 → 过期再协商**。两者可以配合，不是二选一。

---

### 题 2：`Cache-Control` 里 `no-cache` 和 `no-store` 有什么区别？

> **`no-cache`**：可以存，但**用之前必须向服务器验证**（常走协商缓存）。适合 HTML 入口——本地可以存，但要确认是不是最新。
>
> **`no-store`**：**完全不缓存**，每次都要完整请求。适合敏感数据、个性化接口。
>
> 很多人误以为 `no-cache` 是不缓存，这是经典坑。

---

### 题 3：`ETag` 和 `Last-Modified` 怎么选？

> **`Last-Modified`** 看文件最后修改时间，精度到秒，且「内容未变但 touch 了文件」也会失效。
>
> **`ETag`** 是资源标识，通常基于内容 hash，更精确。两者同时存在时，**ETag 优先级更高**。
>
> 代价是 ETag 要服务器计算和比对，集群环境下生成规则不一致可能影响命中。三年岗答到这里够用。

---

### 题 4：为什么静态资源文件名要带 hash？

> 带 hash 后，**内容变 = 文件名变**，旧文件可以 `max-age=31536000` 长期强缓存，不会误用旧 JS。
>
> 新版本通过 HTML 引用新文件名即可。HTML 本身 `no-cache`，用户每次会拿到最新引用关系。这是前端工程化里 cache busting 的标准做法。

```text
app.8f3a1c.js   → 内容变 → app.newhash.js
index.html      → no-cache，引用新 hash 文件
```

---

### 题 5：普通刷新、强刷、回车，缓存行为有什么不同？

> **地址栏回车**：一般走正常缓存策略，强缓存未过期可能直接用。
>
> **普通刷新（F5）**：多数浏览器会**跳过强缓存**，仍可能带验证头走协商（304）。
>
> **强制刷新（Ctrl+F5）**：通常**跳过本地缓存**，全量重新拉资源。
>
> 排查线上「用户看不到新版本」时，这三者和 CDN、Service Worker 要一起考虑。

---

## 核心原理

### 决策流程

```text
发起请求
  → 有本地缓存？
      否 → 正常请求 200
      是 → 强缓存未过期？
              是 → 直接用（不发请求）
              否 → 带 If-None-Match / If-Modified-Since
                      → 304 用本地 / 200 更新
```

### 强缓存关键头

| 头 | 说明 |
| --- | --- |
| `Cache-Control: max-age=N` | N 秒内有效（HTTP/1.1，优先） |
| `Expires` | 绝对过期时间（HTTP/1.0，受客户端时钟影响） |
| `public` / `private` | 是否允许共享代理缓存 |
| `immutable` | 提示资源期内不会变，减少不必要的重新验证 |

```http
Cache-Control: public, max-age=31536000, immutable
```

### 协商缓存关键头

| 响应头 | 请求头 | 比较依据 |
| --- | --- | --- |
| `ETag: "abc123"` | `If-None-Match: "abc123"` | 资源标识 |
| `Last-Modified: ...` | `If-Modified-Since: ...` | 最后修改时间 |

304 响应体为空，节省带宽；浏览器合并响应头后继续使用本地副本。

### 三类资源的推荐策略

| 资源 | 策略 | 原因 |
| --- | --- | --- |
| HTML | `Cache-Control: no-cache` | 入口要拿到最新资源引用 |
| JS/CSS/图片（带 hash） | 长强缓存 + `immutable` | 文件名即版本 |
| API / 用户数据 | `no-store` 或短缓存 + 私有 | 正确性优先 |

### 与 CDN、Service Worker 的关系

- **CDN** 也遵守缓存头；发版时要关注 CDN 刷新或 hash 策略。
- **Service Worker** 可以在浏览器缓存之上再拦截请求；排查缓存问题要看 Application 面板 SW 是否命中。

## 高频追问

### 为什么看到 304 但页面还是旧的？

可能 HTML 被强缓存了（应配 `no-cache`），或 SW/CDN 层仍返回旧资源；要用禁用缓存 + 看响应头排查全链路。

### `max-age=0` 和 `no-cache` 一样吗？

不完全一样，但实践中都可能触发重新验证；具体以实现为准，面试可说「都会倾向于向服务器确认 freshness」。

### 接口能不能强缓存？

公开、变化少的数据可以短 `max-age` + ETag；用户态、权限态接口默认 `no-store`，避免串数据。
