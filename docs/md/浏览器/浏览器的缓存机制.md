# 浏览器的缓存机制

## 面试定位

浏览器缓存是网络和性能面试高频题。常见追问包括：强缓存和协商缓存区别、`Cache-Control` 常用字段、`ETag` 和 `Last-Modified` 区别、为什么刷新后状态码是 304、前端静态资源如何配置缓存。

## 核心原理

浏览器缓存的目标是减少重复请求、降低服务器压力、提升页面加载速度。

缓存大致分为两层：

1. 强缓存：缓存有效期内直接使用本地资源，不向服务器发请求。
2. 协商缓存：向服务器确认资源是否变化，未变化返回 304，浏览器继续使用本地缓存。

请求资源时，浏览器通常先判断强缓存；强缓存失效后，再走协商缓存。

## 强缓存

强缓存命中时，浏览器不会向服务器发起请求。

主要响应头：

- `Cache-Control`
- `Expires`

### Cache-Control

`Cache-Control` 是 HTTP/1.1 的缓存控制方式，优先级高于 `Expires`。

常见字段：

```http
Cache-Control: max-age=31536000
```

表示资源在 31536000 秒内有效。

```http
Cache-Control: no-cache
```

表示可以缓存，但每次使用前必须向服务器确认是否变化。它不是“不缓存”。

```http
Cache-Control: no-store
```

表示不允许缓存，适合敏感数据。

```http
Cache-Control: public
```

表示资源可以被浏览器和代理服务器缓存。

```http
Cache-Control: private
```

表示只能被浏览器私有缓存，不能被共享代理缓存。

### Expires

`Expires` 是 HTTP/1.0 的方式，值是一个绝对时间。

```http
Expires: Wed, 21 Oct 2026 07:28:00 GMT
```

缺点是依赖客户端本地时间，如果用户时间不准，可能影响缓存判断。

## 协商缓存

强缓存失效后，浏览器会带上缓存标识向服务器询问资源是否变化。

如果资源没变，服务器返回：

```http
304 Not Modified
```

浏览器继续使用本地缓存。

协商缓存主要有两组头：

- `Last-Modified` / `If-Modified-Since`
- `ETag` / `If-None-Match`

### Last-Modified

第一次响应：

```http
Last-Modified: Wed, 21 Oct 2026 07:28:00 GMT
```

下次请求：

```http
If-Modified-Since: Wed, 21 Oct 2026 07:28:00 GMT
```

服务器比较资源最后修改时间。如果没有变化，返回 304。

缺点：

- 时间精度通常到秒，1 秒内多次修改可能无法识别。
- 文件内容没变但修改时间变了，也可能导致缓存失效。

### ETag

`ETag` 是服务器为资源生成的唯一标识，通常基于内容、版本或 hash。

第一次响应：

```http
ETag: "abc123"
```

下次请求：

```http
If-None-Match: "abc123"
```

服务器比较当前资源的 ETag。如果一致，返回 304。

`ETag` 通常比 `Last-Modified` 更精确。如果两者同时存在，通常优先使用 `ETag`。

## 缓存流程

一次资源请求可以这样理解：

1. 浏览器检查是否有本地缓存。
2. 如果有缓存且强缓存未过期，直接使用缓存。
3. 如果强缓存过期，带上 `If-None-Match` 或 `If-Modified-Since` 发起请求。
4. 服务器判断资源没变，返回 304。
5. 服务器判断资源已变，返回 200 和新资源。

## 静态资源缓存策略

实际项目里常见策略：

### HTML 不强缓存

HTML 是入口文件，应该保证用户能拿到最新资源引用。

```http
Cache-Control: no-cache
```

### 带 hash 的静态资源长期缓存

JS、CSS、图片等构建后带 hash，内容变化文件名就变化，可以设置长期缓存。

```http
Cache-Control: public, max-age=31536000, immutable
```

例如：

```text
app.8f3a1c.js
style.a91cd.css
```

### 接口缓存谨慎使用

用户相关、权限相关、频繁变化的数据要谨慎缓存。敏感接口通常用：

```http
Cache-Control: no-store
```

## 用户操作对缓存的影响

不同浏览器细节可能不同，但一般可以这样理解：

- 地址栏回车：优先走正常缓存策略。
- 普通刷新：通常会跳过强缓存，走协商缓存。
- 强制刷新：通常会跳过本地缓存，重新请求资源。

## 面试回答

可以这样答：

> 浏览器缓存主要分强缓存和协商缓存。强缓存命中时不会发请求，主要由 `Cache-Control` 和 `Expires` 控制，其中 `Cache-Control` 优先级更高。强缓存失效后会走协商缓存，浏览器带上 `If-None-Match` 或 `If-Modified-Since`，服务器判断资源没变就返回 304，浏览器继续用本地缓存。协商缓存里 `ETag` 通常比 `Last-Modified` 更精确，因为它可以基于内容标识资源，而 `Last-Modified` 受时间精度影响。实际项目里，HTML 通常设置 `no-cache`，保证入口更新；带 hash 的 JS、CSS、图片可以设置长期强缓存；敏感接口用 `no-store`。

## 高频追问

### `no-cache` 和 `no-store` 的区别？

`no-cache` 表示可以缓存，但每次使用前必须向服务器确认。`no-store` 表示完全不缓存。

### 为什么静态资源要带 hash？

带 hash 后，资源内容变化会生成新文件名。旧文件可以长期缓存，新版本通过 HTML 引用新文件，既能提升缓存命中率，又能保证更新。

### 为什么有些请求返回 304？

说明浏览器走了协商缓存，服务器判断资源没有变化，所以不返回资源正文，让浏览器使用本地缓存。

### ETag 有什么问题？

ETag 需要服务器生成和比较，可能有计算成本。在多服务器部署时，如果 ETag 生成规则不一致，可能导致缓存命中不稳定。
