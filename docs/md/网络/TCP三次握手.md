# TCP 三次握手

## 面试定位

TCP 三次握手是网络基础高频题。面试重点不是背 SYN、ACK，而是解释为什么需要三次、两次为什么不够、握手解决了什么问题。

## 核心原理

TCP 是面向连接的可靠传输协议。在真正传输数据前，客户端和服务端需要确认双方的发送和接收能力，并同步初始序列号。

三次握手流程：

```text
Client                      Server
  | ---- SYN, seq=x -------> |
  | <--- SYN+ACK, seq=y, ack=x+1
  | ---- ACK, ack=y+1 -----> |
  |        连接建立          |
```

## 三次过程

### 第一次：客户端发送 SYN

客户端发送 `SYN` 报文，表示希望建立连接，并带上客户端初始序列号 `seq = x`。

客户端进入 `SYN_SENT` 状态。

### 第二次：服务端返回 SYN + ACK

服务端收到客户端 SYN 后，返回 `SYN + ACK`。

- `ack = x + 1`：确认收到客户端 SYN。
- `seq = y`：服务端自己的初始序列号。

服务端进入 `SYN_RCVD` 状态。

### 第三次：客户端返回 ACK

客户端收到服务端的 SYN + ACK 后，返回 ACK。

- `ack = y + 1`：确认收到服务端 SYN。

双方进入 `ESTABLISHED` 状态，连接建立。

## 为什么需要三次

三次握手的核心目标是确认双方的发送和接收能力都正常。

第一次后：

- 服务端知道客户端能发送。
- 服务端知道自己能接收。

第二次后：

- 客户端知道自己能发送、能接收。
- 客户端知道服务端能发送、能接收。

第三次后：

- 服务端知道客户端能接收。
- 双方确认彼此能力正常。

如果只有两次，服务端无法确认客户端是否能收到自己的响应。

## 为什么不是两次

两次握手可能导致历史连接请求造成错误连接。

例如客户端之前发送的 SYN 因网络延迟很久才到达服务端。如果两次握手就建立连接，服务端回复 SYN+ACK 后可能直接进入连接状态，但客户端其实并不想建立这个旧连接，服务端资源就会被浪费。

第三次 ACK 可以让客户端确认“这是当前有效连接”，服务端收到后再正式建立连接。

## 为什么不是四次

三次已经能确认双方收发能力和初始序列号。再多一次没有必要，会增加连接建立延迟。

## SYN Flood

SYN Flood 是一种攻击方式。攻击者大量发送 SYN，但不完成第三次 ACK，导致服务端维护大量半连接。

防御方式：

- SYN Cookie。
- 调整半连接队列。
- 限流。
- 防火墙或负载均衡防护。

## 面试回答

可以这样答：

> TCP 三次握手是建立可靠连接的过程。第一次客户端发 SYN，告诉服务端我要建立连接并同步客户端初始序列号；第二次服务端回 SYN+ACK，确认客户端 SYN，同时同步服务端初始序列号；第三次客户端回 ACK，确认收到服务端 SYN，连接建立。三次握手的目的不是形式上发三包，而是确认双方的发送和接收能力都正常，并同步序列号。两次不够，因为服务端无法确认客户端能否收到自己的响应，也可能因为历史 SYN 报文导致服务端误建立连接。

## 高频追问

### 三次握手发生在什么时候？

HTTP 请求前。如果是 HTTPS，通常先 TCP 三次握手，再进行 TLS 握手，最后发送 HTTP 请求。

### 初始序列号有什么用？

TCP 用序列号保证数据按序、去重和重传。握手阶段双方同步初始序列号，后续数据传输才能基于序列号确认。

### 什么是半连接队列？

服务端收到 SYN 并返回 SYN+ACK 后，连接处于半连接状态，会进入半连接队列，等待客户端第三次 ACK。
