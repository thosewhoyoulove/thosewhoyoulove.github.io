# TCP 四次挥手

## 面试定位

TCP 四次挥手常和三次握手一起考。面试重点是：为什么关闭连接要四次、为什么有 `TIME_WAIT`、为什么要等 2MSL、服务端大量 `CLOSE_WAIT` 说明什么。

## 核心原理

TCP 是全双工通信，客户端和服务端都有独立的发送方向。关闭连接时，一方说“我没数据要发了”，不代表另一方也没数据要发。

因此 TCP 关闭通常需要双方分别关闭自己的发送通道。

```text
Client                         Server
  | ------ FIN, seq=x --------> |
  | <----- ACK, ack=x+1 ------- |
  | <----- FIN, seq=y --------- |
  | ------ ACK, ack=y+1 ------> |
  |        TIME_WAIT            |
```

## 四次过程

### 第一次：主动关闭方发送 FIN

主动关闭方发送 FIN，表示自己没有数据要发送了，但仍然可以接收数据。

主动关闭方进入 `FIN_WAIT_1`。

### 第二次：被动关闭方返回 ACK

被动关闭方收到 FIN 后，返回 ACK，表示已经知道对方要关闭发送方向。

被动关闭方进入 `CLOSE_WAIT`。

此时连接处于半关闭状态：主动关闭方不能再发数据，但还能接收数据；被动关闭方如果还有数据要发，可以继续发送。

### 第三次：被动关闭方发送 FIN

被动关闭方数据发送完后，也发送 FIN，表示自己也没有数据要发送了。

被动关闭方进入 `LAST_ACK`。

### 第四次：主动关闭方返回 ACK

主动关闭方收到 FIN 后，返回 ACK，并进入 `TIME_WAIT`。

被动关闭方收到 ACK 后进入 `CLOSED`。主动关闭方等待 2MSL 后也进入 `CLOSED`。

## 为什么是四次

因为 TCP 是全双工，两个方向要分别关闭。

服务端收到客户端 FIN 后，可能还有数据没发完，所以不能立刻把 ACK 和自己的 FIN 合并发送。它先 ACK，等数据处理完后再 FIN，因此通常是四次。

如果被动关闭方刚好没有数据要发送，也可能把 ACK 和 FIN 合并，看起来像三次，但逻辑上仍然是双方分别关闭。

## TIME_WAIT 的作用

主动关闭方最后会进入 `TIME_WAIT`，等待 2MSL。

主要原因：

1. 确保最后一个 ACK 能被对方收到。如果 ACK 丢了，对方会重发 FIN，主动关闭方还能再回 ACK。
2. 让旧连接中的延迟报文在网络中自然消失，避免影响后续相同四元组的新连接。

MSL 是报文在网络中的最大生存时间，2MSL 可以覆盖一个报文和它响应报文的最大存活周期。

## CLOSE_WAIT 问题

如果服务端出现大量 `CLOSE_WAIT`，通常说明对端已经关闭连接，但本端应用程序没有及时关闭 socket。

常见原因：

- 应用代码没有正确释放连接。
- 阻塞在业务逻辑，迟迟不 close。
- 连接池或异常处理有问题。

## 面试回答

可以这样答：

> TCP 四次挥手是关闭连接的过程。因为 TCP 是全双工通信，两个方向需要分别关闭。主动关闭方先发 FIN，表示自己不再发送数据；被动关闭方回 ACK，表示收到关闭请求，但它可能还有数据要发，所以不会马上关闭；等被动关闭方数据发完，再发 FIN；主动关闭方最后回 ACK，并进入 TIME_WAIT。TIME_WAIT 等 2MSL 是为了确保最后的 ACK 能被对方收到，也让旧连接中的延迟报文在网络中消失，避免影响新连接。

## 高频追问

### 为什么挥手不是三次？

因为收到 FIN 后，被动关闭方可能还有数据要发送，所以 ACK 和 FIN 通常不能立即合并。它需要先确认，再等数据发完后发送 FIN。

### TIME_WAIT 在哪一方？

通常在主动关闭连接的一方。

### 大量 TIME_WAIT 是不是一定有问题？

不一定。主动关闭大量短连接的一方会出现 TIME_WAIT。是否有问题要结合数量、端口耗尽、连接复用和系统参数判断。
