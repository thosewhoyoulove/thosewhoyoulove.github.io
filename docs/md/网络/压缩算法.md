# 前端资源压缩：Brotli 与 Gzip

## 面试定位

压缩通常出现在首屏优化、构建优化和网络传输优化里。面试重点是：为什么压缩能提升性能、Brotli 和 Gzip 怎么选、动态压缩和静态预压缩有什么区别。

## 核心原理

前端资源压缩主要减少 HTML、CSS、JavaScript、SVG、JSON 等文本资源的传输体积，从而降低下载时间。

浏览器通过请求头告诉服务器支持哪些压缩算法：

```http
Accept-Encoding: br, gzip, deflate
```

服务器选择一种压缩方式，并通过响应头告诉浏览器：

```http
Content-Encoding: br
```

浏览器收到后自动解压，业务代码无感知。

## Gzip 和 Brotli 对比

| 维度 | Gzip | Brotli |
| --- | --- | --- |
| 兼容性 | 非常好 | 主流现代浏览器支持 |
| 压缩率 | 较好 | 通常更高 |
| 压缩速度 | 快 | 高等级压缩更慢 |
| 解压速度 | 快 | 通常也很快 |
| 适合场景 | 动态内容、通用兜底 | 静态资源预压缩 |

Brotli 通常对 JS、CSS、HTML 有更好的压缩率，适合构建阶段生成 `.br` 文件。Gzip 兼容性强，常作为兜底。

## 动态压缩和静态预压缩

动态压缩：请求到来时服务器实时压缩。

优点：适合动态内容。

缺点：消耗 CPU，压缩等级不能太高。

静态预压缩：构建时提前生成 `.gz` 或 `.br` 文件。

优点：可以用更高压缩等级，不占用线上请求时 CPU。

缺点：只适合静态资源，需要服务器正确配置。

## Vite 示例

```javascript
import compress from "vite-plugin-compression";

export default {
    plugins: [
        compress({
            ext: ".br",
            algorithm: "brotliCompress",
            deleteOriginFile: false,
        }),
    ],
};
```

## 注意事项

- 图片、视频这类已压缩资源通常不适合再 Gzip/Brotli。
- 小文件压缩收益有限，可能不值得额外处理。
- 压缩只解决传输体积，不解决 JS 执行时间。
- 需要配合缓存和 CDN 才能更好发挥效果。

## 面试回答

可以这样答：

> 前端资源压缩通过减少传输体积提升加载速度。浏览器会通过 `Accept-Encoding` 声明支持 `br`、`gzip` 等算法，服务器返回 `Content-Encoding` 告诉浏览器使用了哪种压缩。Gzip 兼容性强，适合通用兜底；Brotli 对 HTML、CSS、JS 这类文本资源通常压缩率更高，更适合构建阶段做静态预压缩。实际项目里我会对构建产物生成 `.br` 和 `.gz`，CDN 或 Nginx 根据浏览器能力返回对应文件，同时注意图片视频这类已压缩资源不重复压缩。

## 高频追问

### Brotli 一定比 Gzip 好吗？

不一定。Brotli 通常压缩率更高，但高等级压缩更慢，动态压缩场景要考虑 CPU 成本。静态资源预压缩更适合 Brotli。

### 压缩能解决首屏所有问题吗？

不能。压缩降低下载体积，但如果瓶颈是 JS 执行慢、接口慢、渲染阻塞，仍需要代码分包、缓存、渲染优化等手段。
