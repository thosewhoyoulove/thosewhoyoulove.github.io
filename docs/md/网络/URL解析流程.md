# 从输入 URL 到页面展示

## 面试定位

这是前端 **综合压轴题**：浏览器、DNS、TCP/TLS、HTTP、缓存、渲染会在一题里串起来。三年岗不必背每个报文字段，但要能 **先讲主链路 60 秒，再按追问下钻**。

面试官常在这个题里追问：**HTTPS 握手、对称/非对称加密、为什么不是全程非对称、304 会不会发请求、script 为什么阻塞**。

---

## 面试回答

### 总答（60～90 秒版）

> 用户在地址栏输入 URL 后，浏览器主进程先做 **URL 解析和规范化**——补全协议、检查 HSTS、处理非法字符，然后交给 **网络进程**。
>
> 网络进程 **先查缓存**（Service Worker → HTTP 强缓存 → 协商缓存），命中可能直接返回，不必完整走网络。未命中则 **DNS** 把域名解析成 IP，再 **TCP 三次握手** 建连；若是 HTTPS，还要 **TLS 握手** 校验证书并协商 **对称会话密钥**。
>
> 之后发 **HTTP 请求**，服务器返回响应；响应体进入 **渲染进程**，解析 HTML/CSS 建 DOM/CSSOM，合成渲染树，layout、paint、composite 上屏。过程中遇到 script、css、img 等会触发子资源请求。`DOMContentLoaded` 表示 DOM 就绪，`load` 表示依赖资源基本加载完。

一句话总结：

> URL → 解析/缓存 → DNS → TCP → TLS（HTTPS）→ HTTP → 渲染；HTTPS 用非对称换密钥、对称传数据。

---

### 题 1：输入 URL 后，逐步发生了什么？（详细版）

下面按 **时间顺序** 展开，适合面试官说「再细一点」时用。

#### 第 1 步：URL 解析（浏览器主进程）

用户输入的不一定是完整 URL，浏览器要先 **规范化**：

| 输入 | 浏览器处理 |
| --- | --- |
| `example.com` | 补全为 `https://example.com/`（默认协议因浏览器/HSTS 而定） |
| `http://example.com` | 若域名在 HSTS 预加载列表，可能 **强制升级 HTTPS** |
| `  example.com/path ` | 去空格、非法字符 **百分号编码**（如空格 → `%20`） |
| `mailto:a@b.com` | 非 HTTP 协议，转交系统邮件客户端，不走网页加载 |
| 搜索关键词 | 走默认搜索引擎，生成搜索 URL |

URL 结构也会被解析：`protocol`（https）、`host`（域名+端口）、`path`、`query`、`hash`。  
**注意**：`hash`（`#` 后）默认 **不会** 发给服务器，只在前端路由/锚点使用。

#### 第 2 步：缓存检查（网络进程）

在真正联网前，按优先级查缓存：

```text
Service Worker 拦截？
  → HTTP 强缓存（Cache-Control / Expires）未过期？
    → 协商缓存（ETag / Last-Modified）→ 304？
      → 都未命中 → 继续 DNS
```

- **强缓存命中**：Network 面板可能显示 `(disk cache)`，**不发 HTTP 请求**。
- **304**：**会发请求**，但响应体为空，用本地副本。
- **Service Worker**：可在浏览器缓存之上自定义策略（离线包、预缓存）。

详见 [浏览器缓存机制](/md/浏览器/浏览器的缓存机制.md)。

#### 第 3 步：DNS 解析（网络进程）

把 `www.example.com` 解析为 IP（如 `93.184.216.34`）：

```text
浏览器 DNS 缓存
  → 操作系统缓存（hosts）
  → 路由器/ISP 本地 DNS
  → 递归：根 DNS → 顶级域（.com）→ 权威 DNS → IP
```

优化：`dns-prefetch`、`preconnect` 可提前完成解析或连接。

#### 第 4 步：TCP 三次握手（网络进程）

拿到 IP 后，与服务器 **80/443 端口** 建立 TCP 连接：

```text
客户端 ── SYN, seq=x ──────────→ 服务端
客户端 ←─ SYN+ACK, seq=y, ack=x+1 ── 服务端
客户端 ── ACK, ack=y+1 ─────────→ 服务端
        （连接建立，可以传应用层数据）
```

**为什么是三次**：双方都要确认「自己能发、对方能收」，并同步初始序列号；两次无法可靠确认客户端的接收能力。

详见 [TCP 三次握手](/md/网络/TCP三次握手.md)。

#### 第 5 步：TLS 握手（HTTPS，网络进程）

HTTP 明文，HTTPS = HTTP + **TLS**（TLS 1.3 为主，兼容 TLS 1.2）。TCP 建连后，在传 HTTP 之前先建 **加密通道**。

**核心结论（必背）**：

| 阶段 | 加密方式 | 干什么 |
| --- | --- | --- |
| TLS 握手 | **非对称** + 密钥交换算法 | 验证服务器身份、协商 **对称会话密钥** |
| 传 HTTP 数据 | **对称**（如 AES-GCM） | 加密业务数据，速度快 |

**为什么不能全程用非对称？**  
非对称（RSA/ECC）加解密 **慢一个数量级以上**，适合传几百字节；HTML/JS/接口 JSON 是大量数据，必须用对称加密。

**TLS 1.3 简化握手（1-RTT，概念版）**：

```text
1. ClientHello
   客户端：支持的 TLS 版本、加密套件、随机数 client_random

2. ServerHello + Certificate + 密钥交换参数
   服务端：选定套件、证书（含公钥）、随机数 server_random
   客户端：校验证书链、域名、有效期

3. 双方用 ECDHE 等算法算出相同的 premaster secret
   再导出 session key（对称密钥）

4. 后续 HTTP 请求/响应用 session key 对称加密
```

**证书校验在防什么？**  
防 **中间人** 冒充服务器。没有 CA 签名的假证书，浏览器会报「您的连接不是私密连接」。

对称/非对称 **直觉类比** 见下文 [TLS 专题：对称 vs 非对称](#tls-专题对称加密-vs-非对称加密)。

详见 [HTTPS](/md/网络/HTTPS.md)。

#### 第 6 步：发送 HTTP 请求（网络进程）

TLS 通道就绪后，发送明文 HTTP（由 TLS 层加密成密文 on the wire）：

```http
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 ...
Accept: text/html,application/xhtml+xml
Accept-Encoding: gzip, br
Cookie: session=abc123
Connection: keep-alive
```

- **GET**：参数在 query；**POST**：body 带表单/JSON。
- **Keep-Alive**：同一 TCP 连接上复用，减少握手开销（HTTP/2/3 多路复用更进一步）。

#### 第 7 步：服务器处理并返回响应

Nginx / Node / Java 等：路由、鉴权、查库、渲染模板或读静态文件。

```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Encoding: gzip
Cache-Control: no-cache
Set-Cookie: session=xyz; HttpOnly; Secure; SameSite=Lax

<!DOCTYPE html>...
```

常见状态码：200 成功、301/302 重定向、304 协商缓存、404 不存在、502 网关错误。  
重定向时浏览器会 **发起新 URL 请求**（可能再次走 DNS/TCP/TLS，或复用连接）。

#### 第 8 步：浏览器解析与渲染（渲染进程）

网络进程通过 IPC 把 HTML 交给 **渲染进程**（每 tab 独立沙箱）：

```text
HTML → DOM
CSS  → CSSOM
     ↘
      Render Tree → Layout → Paint → Composite → 屏幕
```

- **CSS**：阻塞渲染（需要 CSSOM 才能算样式）。
- **同步 script**：阻塞 HTML 解析；`defer` / `async` / `type=module` 行为不同。
- 子资源（JS/CSS/图片/字体）触发 **新的网络请求**（可能命中缓存）。

详见 [浏览器渲染原理](/md/浏览器/浏览器的渲染原理.md)。

#### 第 9 步：事件与「页面完成」

| 事件 | 含义 |
| --- | --- |
| `DOMContentLoaded` | DOM 解析完，**不等**图片等 |
| `load` | 窗口依赖资源基本加载完 |
| `beforeunload` / `pagehide` | 离开页面 |

### 题 2：HTTPS 里对称加密和非对称加密分别干什么？（口述版）

> **非对称加密**有一对密钥：**公钥**公开（在证书里），**私钥**只在服务器。公钥加密的数据只有私钥能解；适合 **证明身份** 和 **安全地协商密钥**，但慢，不适合传整个网页。
>
> **对称加密**加解密用 **同一把会话密钥**，速度快，适合加密 HTTP 正文。问题是密钥怎么传到对方而不被窃听——TLS 握手用非对称或 **ECDHE 密钥交换** 解决这个问题，双方独立算出同一把对称密钥，链路上不传明文密钥。
>
> 所以记忆：**握手阶段非对称/密钥交换，数据阶段对称。** 不是「HTTPS 等于非对称」，而是 **混合加密**。

## 核心原理

### 全链路一图

```text
┌──────────── 浏览器主进程 ────────────┐
│ 输入 URL → 解析/规范化/HSTS          │
└─────────────────┬───────────────────┘
                  ▼
┌──────────── 网络进程 ───────────────────────────────────┐
│ SW 缓存 → HTTP 缓存 → DNS → TCP 握手 → TLS 握手(HTTPS)   │
│ → HTTP 请求/响应 → 解码/解压 → 交给渲染进程               │
└─────────────────┬───────────────────────────────────────┘
                  ▼
┌──────────── 渲染进程 ───────────────────────────────────┐
│ DOM + CSSOM → Render Tree → Layout → Paint → Composite  │
│ → 子资源并行/串行加载 → DOMContentLoaded / load          │
└─────────────────────────────────────────────────────────┘
```

### TLS 专题：对称加密 vs 非对称加密

#### 对称加密（Symmetric）

- **同一把密钥**加密和解密。
- 算法示例：**AES-128-GCM**、**ChaCha20-Poly1305**（TLS 1.3 常用）。
- **优点**：极快，适合 MB 级网页和 API。
- **缺点**：密钥必须交给对方——如果在不安全的信道上直接发送密钥，会被窃听。

**生活类比**：你和朋友有一把 **相同的保险柜钥匙**（会话密钥）。传文件进柜子锁上再发，很快。但 **第一次把钥匙交给朋友** 很难——邮递可能被拆。

#### 非对称加密（Asymmetric）

- **密钥对**：公钥（Public）、私钥（Private）。
- **公钥加密 → 仅私钥能解密**（保密）。
- **私钥签名 → 公钥能验证**（证明身份，数字签名）。
- 算法示例：**RSA**、**ECDHE**（椭圆曲线，常用于密钥交换）。
- **优点**：公钥可以公开贴在证书上，解决「如何把密钥交给对方」。
- **缺点**：计算慢，通常 **比 AES 慢 100～1000 倍**，只用于握手短数据。

**生活类比**：**带投递口的信箱**（公钥）——谁都能投信进去；只有主人有 **信箱钥匙**（私钥）能取。投信（加密）方便，但不可能用信箱机制传送整本 encyclopedia。

#### TLS 如何「混合」使用

```text
                    TLS 握手（少量字节，慢没关系）
┌─────────────────────────────────────────────────────────┐
│ 1. 服务器证书 + 公钥  →  客户端验证 CA、域名            │
│ 2. ECDHE/RSA 密钥交换  →  双方得到 premaster secret     │
│ 3. 派生 session key   →  对称密钥（如 AES 密钥）        │
└─────────────────────────────────────────────────────────┘
                              │
                              ▼
                    数据传输（大量字节，必须快）
┌─────────────────────────────────────────────────────────┐
│  HTTP 明文  →  TLS 用 session key（AES-GCM）加密  →  密文 │
└─────────────────────────────────────────────────────────┘
```

**TLS 1.2 vs 1.3（面试常问一句）**：

| 版本 | 特点 |
| --- | --- |
| TLS 1.2 | 握手约 2-RTT；套件多，配置不当可能有弱加密 |
| TLS 1.3 | 握手约 1-RTT；禁用很多不安全算法；**默认前向安全**（ECDHE 临时密钥，历史流量即使私钥泄露也难解密） |

**前向安全（Forward Secrecy）**：每次会话用 **临时** ECDHE 密钥，不依赖长期 RSA 私钥直接解密历史密文。三年岗提到这句是加分项。

#### 证书在其中的角色

证书 ≈ **身份证 + 公钥容器**，由 **CA** 签名：

```text
证书内容：域名、公钥、有效期、颁发者
客户端：用系统内置根证书验证签名链 → 确认「这个公钥真的属于 example.com」
```

没有可信证书，中间人可以 **冒充服务器** 给你另一把公钥，实现窃听（中间人攻击）。

### 各阶段耗时感知（项目/面试）

| 阶段 | 典型耗时量级 | 优化方向 |
| --- | --- | --- |
| DNS | 几十 ms～数百 ms | 预解析、HTTPDNS、减少跨域 |
| TCP + TLS | 1～3 RTT | TLS 1.3、会话复用、preconnect |
| TTFB | 取决于后端 | CDN、缓存、接口性能 |
| 解析渲染 | 取决于 DOM/JS 体积 | 分包、defer、关键 CSS |

## 高频追问

### 输入 URL 后一定会发 HTTP 请求吗？

不一定。强缓存、Service Worker、301 本地缓存、`file:` 协议等都可能跳过完整网络请求。

### 为什么 HTTPS 不全程用 RSA 加密 HTTP body？

RSA 慢且不适合大块数据；业界标准就是握手协商 **AES 等对称密钥** 再传数据。

### 对称密钥会被窃听吗？

若握手正确（证书校验通过、使用 ECDHE），攻击者在链路上只能看到密文，拿不到 session key。若用户忽略证书警告或遭恶意根证书，仍可能 MITM。

### `hash` 会变吗？会不会请求服务器？

默认 **不会** 把 `#/path` 发给服务器；SPA 前端路由改 hash 不触发完整页面导航（除非刷新）。

### DOMContentLoaded 和 load 区别？

见上文表格；性能监控里 LCP 等与两者都不同，别混为一谈。

### HTTP/2、HTTP/3 在这条链路的哪？

仍在 TCP/TLS 之上，改的是 **多路复用、头部压缩、队头阻塞** 等，不改变「DNS → 连接 → 请求 → 渲染」主链。HTTP/3 用 QUIC 替代 TCP。
