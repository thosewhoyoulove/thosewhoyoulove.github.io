# HTTPS

## 面试定位

HTTPS 是网络与安全 **必考题**，常嵌在「输入 URL 后发生了什么」里追问。三年岗要答清：**TLS 解决什么、证书验证什么、对称/非对称各用在哪、为什么不能全程非对称**。

---

## 面试回答

### 总答：HTTPS 和 HTTP 有什么区别？怎么保证安全？

> HTTPS = HTTP + **TLS**，在 TCP 之上加安全层，提供 **机密性**（加密）、**完整性**（防篡改）、**身份认证**（证书证明服务器是谁）。
>
> 建立 TCP 后先做 **TLS 握手**：服务器发 **证书**（含公钥），客户端校验 CA 签名、域名、有效期；双方通过 **ECDHE 等密钥交换** 协商出 **对称会话密钥**；之后 HTTP 正文用 **AES-GCM** 等对称算法加密传输。
>
> **非对称**用在握手——慢但适合交换密钥和验身份；**对称**用在传数据——快。不能全程非对称，否则性能无法接受。
>
> HTTPS 防链路窃听和大部分中间人，但不防 DNS 劫持（需 HSTS/DoH）、服务端漏洞、XSS 偷 Cookie。

一句话总结：

> HTTPS = 证书验身份 + 握手协商对称密钥 + 对称加密传 HTTP；混合加密，不是全程 RSA。

---

### 题 1：对称加密和非对称加密有什么区别？

> **对称加密**：加解密 **同一把密钥**，如 AES。快，适合大量 HTTP 数据；难点是密钥如何安全交给对方。
>
> **非对称加密**： **公钥 + 私钥** 成对。公钥加密只有私钥能解；私钥签名公钥能验。适合握手和身份证明； **慢**，不适合加密整页 HTML。
>
> TLS 的策略：**用非对称/ECDHE 解决「密钥从哪来」**，用 **对称加密传业务**。证书里的公钥主要用于验证身份和参与密钥交换，不是拿来加密每个 API 响应的。

---

### 题 2：TLS 握手大致做了什么？（TLS 1.3 概念版）

> 1. **ClientHello**：客户端支持的 TLS 版本、加密套件、随机数。  
> 2. **ServerHello + 证书**：服务端选定参数，下发证书链。  
> 3. **客户端验证书**：是否可信 CA、域名是否匹配、是否过期/吊销。  
> 4. **密钥交换**：ECDHE 等，双方算出相同的 **premaster secret**，再导出 **session key**。  
> 5. **Finished**：握手完成，后续 HTTP 用 session key 对称加密。

TLS 1.3 约 **1-RTT** 完成；TLS 1.2 通常 **2-RTT**。

---

### 题 3：证书是什么？浏览器怎么验证？

> 证书是 CA 签发的「域名 ↔ 公钥」绑定，含域名、公钥、有效期、颁发者、数字签名。
>
> 浏览器用内置 **根证书** 验证签名链，确认证书未被篡改且域名与访问地址一致。失败则中断连接（除非用户强行继续，有 MITM 风险）。

## 核心原理

### HTTPS 解决的三类问题

| 问题 | 机制 | 没有 HTTPS 的风险 |
| --- | --- | --- |
| 机密性 | 对称加密 HTTP | 明文被 Wi-Fi/运营商窃听 |
| 完整性 | MAC/AEAD（如 GCM） | 内容被篡改 |
| 身份 | 证书 + CA | 中间人冒充服务器 |

### 对称 vs 非对称（详解）

#### 对称加密

```text
明文 + 密钥(K) → 密文
密文 + 密钥(K) → 明文
```

- 常见算法：**AES-128-GCM**、**ChaCha20-Poly1305**
- TLS 里这叫 **bulk encryption**，加密 Application Data（HTTP）
- 密钥来源：TLS 握手从 premaster secret **派生**（HKDF），不在网络上传明文密钥

#### 非对称加密

```text
公钥加密：明文 + PubKey → 密文，仅 PrivKey 可解
私钥签名：摘要 + PrivKey → 签名，PubKey 可验证
```

- 常见：**RSA**（逐渐少用）、**ECDSA**（证书签名）、**ECDHE**（密钥交换）
- 证书中的公钥 **公开**；私钥 **仅服务器持有**

#### 为什么必须混合？

| 方式 | 加密 1MB 网页 | 密钥分发 |
| --- | --- | --- |
| 仅对称 | 快 | 不安全（密钥怎么传？） |
| 仅非对称 | 极慢 | 公钥可公开 |
| **TLS 混合** | 快（对称传数据） | 安全（握手协商密钥） |

**类比**：

- 对称 = 两人共用一把保险柜钥匙，传大文件快，但 **第一次交钥匙** 很难。
- 非对称 = 公开信箱口 + 私人取信钥匙，适合 **建立信任**，不适合传送整本书。

### TLS 1.3 握手示意

```text
Client                                    Server
  | -------- ClientHello ----------------> |
  |   (版本、套件、client_random)           |
  | <------- ServerHello + Certificate ---- |
  |   (server_random、证书链、ECDHE 参数)    |
  |   [客户端验证证书]                       |
  |   [双方 ECDHE 计算共享秘密]              |
  |   [导出 session key = f(pre_master)]    |
  | -------- Finished (加密) ------------> |
  | <------- Finished (加密) -------------- |
  | ======== 对称加密 HTTP 数据 =========== |
```

**前向安全**：ECDHE 使用 **临时** 密钥对，每次会话不同；即使日后 RSA 私钥泄露，也无法批量解密历史抓包。

### 证书校验清单

1. 证书是否在有效期内  
2. 访问域名与 SAN/CN 是否匹配（`*.example.com` 规则）  
3. 签名链是否追溯到受信任根  
4. 是否吊销（CRL / OCSP / OCSP Stapling）

### HTTPS 能防 / 不能防

| 能防 | 不能单独防 |
| --- | --- |
| 传输窃听、篡改 | DNS 污染（证书不匹配时可发现，但不能阻止错误解析） |
| 大部分 MITM | 用户忽略证书警告 |
| | 服务端被攻破、XSS 盗 Session |

### 性能优化（了解）

- TLS 1.3、会话复用（Session Ticket）  
- OCSP Stapling  
- HTTP/2、HTTP/3 连接复用  
- HSTS 避免 http→https 跳转

## 高频追问

### HTTPS 为什么不全程用非对称加密？

计算成本：RSA 2048 加密仅适合小数据；AES 硬件加速可处理 GB 级流量。标准是握手非对称/ECDHE + 数据对称。

### 公钥加密了 HTTP，服务器怎么知道客户端是谁？

**HTTPS 服务器认证**默认只证明服务器身份。客户端身份靠 **Cookie、Token、mTLS（双向证书）** 等应用层机制，不是 TLS 默认必有的。

### `http` 会自动变 `https` 吗？

不一定。HSTS 响应头或预加载列表可强制；否则可能先 301 再跳转，多一次往返。

### 抓包能看到 HTTPS 内容吗？

没有私钥/会话密钥时只能看到密文和 SNI 等元数据；企业代理安装自定义根证书时例外（需用户信任）。
