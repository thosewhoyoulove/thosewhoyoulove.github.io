# HTTPS

## 面试定位

HTTPS 是网络安全高频题。面试官通常会问：HTTPS 和 HTTP 区别、TLS 握手过程、对称加密和非对称加密为什么结合使用、证书如何校验、能防什么不能防什么。

## 核心原理

HTTPS = HTTP + TLS。它不是新的应用层语义，而是在 HTTP 和 TCP 之间加入 TLS 安全层。

HTTPS 主要解决三类问题：

- 机密性：内容加密，防止被窃听。
- 完整性：防止传输内容被篡改。
- 身份认证：通过证书确认服务端身份，降低中间人攻击风险。

HTTP 明文传输，容易被监听和篡改；HTTPS 传输内容加密，并通过证书校验服务器身份。

## 为什么同时用对称和非对称加密

非对称加密安全但计算成本高，适合握手阶段交换密钥。

对称加密速度快，适合大量数据传输。

所以 TLS 的思路是：握手阶段通过非对称加密或密钥交换算法协商出会话密钥，后续数据用对称加密传输。

## TLS 握手简化流程

以现代 TLS 为简化模型：

1. 客户端发送 `ClientHello`，包含支持的 TLS 版本、加密套件、随机数等。
2. 服务端返回 `ServerHello`，选择协议版本和加密套件，并返回证书。
3. 客户端校验证书：是否过期、域名是否匹配、证书链是否可信、是否被吊销。
4. 双方通过密钥交换算法协商会话密钥。
5. 后续 HTTP 数据使用会话密钥对称加密传输。

```text
ClientHello -> ServerHello + Certificate -> Key Exchange -> Encrypted HTTP
```

TLS 1.3 简化了握手流程，减少往返次数，并默认使用更安全的加密套件。

## 证书校验

证书由 CA 签发，包含域名、公钥、有效期、签名等信息。

浏览器校验证书时会检查：

- 证书是否在有效期内。
- 访问域名是否和证书匹配。
- 证书链是否能追溯到受信任根证书。
- 证书是否被吊销。

如果攻击者伪造证书，但没有受信任 CA 的签名，浏览器会提示不安全。

## HTTPS 能防什么

能防：

- 网络链路上的窃听。
- 传输过程中内容被篡改。
- 大部分中间人伪造服务端。

不能完全防：

- DNS 劫持。需要配合 HSTS、DoH、DNSSEC 等。
- 客户端已中毒或浏览器被篡改。
- 服务端自身漏洞。
- 用户主动信任错误证书。

## 性能优化

HTTPS 相比 HTTP 有握手成本，但现代优化已经让成本可控。

常见优化：

- TLS 1.3 减少握手往返。
- 会话复用降低重复握手成本。
- OCSP Stapling 减少证书状态查询延迟。
- HTTP/2 或 HTTP/3 复用连接。
- 开启 HSTS，减少 HTTP 到 HTTPS 跳转。

## 面试回答

可以这样答：

> HTTPS 本质是 HTTP 加 TLS 安全层，主要提供加密传输、完整性校验和身份认证。HTTP 是明文传输，容易被窃听和篡改；HTTPS 会在 TCP 连接后进行 TLS 握手，服务端返回证书，客户端校验证书链、域名和有效期，确认服务端身份。握手阶段会协商会话密钥，后续数据用对称加密传输，因为对称加密性能更好。HTTPS 可以防止链路窃听、篡改和大部分中间人攻击，但不能解决所有安全问题，比如 DNS 劫持、客户端中毒、服务端漏洞仍需其他手段配合。

## 高频追问

### HTTPS 为什么不全程用非对称加密？

非对称加密计算成本高，不适合大量数据传输。实际是用非对称能力完成身份认证和密钥协商，再用对称加密传业务数据。

### 证书过期会发生什么？

浏览器会提示连接不安全，通常不会正常建立可信 HTTPS 连接。用户强行继续访问会有安全风险。

### HTTPS 能防 DNS 劫持吗？

不能完全防。DNS 劫持发生在域名解析层。HTTPS 可以在连接到错误服务器时通过证书校验发现域名不匹配，但不能阻止 DNS 响应本身被篡改。
