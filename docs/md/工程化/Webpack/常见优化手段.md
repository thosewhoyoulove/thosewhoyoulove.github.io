# Webpack 优化清单

## 面试定位

Webpack 优化题不要背成配置大全。更好的回答方式是先区分优化目标：构建速度、产物体积、加载性能、调试体验，然后说明如何用指标验证。

## 面试回答

> Webpack 优化我会先定位瓶颈，再按目标处理。构建速度方面，缩小 loader 和 resolve 范围、开启 Webpack 5 filesystem cache、减少 Babel 工作量、对重任务使用并行或更快的编译器。产物体积方面，开启 Tree Shaking、检查 `sideEffects`、按需引入、拆分 vendor 和异步 chunk、压缩 JS/CSS/图片。加载性能方面，用 `contenthash` 做长期缓存，合理拆包，必要时 external 到 CDN。调试体验方面，开发环境用较快的 source map，生产环境用 hidden source map 配合错误平台。最后要用构建耗时、bundle 体积、请求瀑布图和线上指标验证收益。

一句话总结：

> Webpack 优化 = 先定位，再分别优化构建速度、产物体积、加载性能和调试定位。

## 优化框架

```text
分析阶段
  -> 构建耗时
  -> bundle 体积
  -> 请求瀑布图
  -> sourcemap 与错误定位

优化阶段
  -> 缩小范围
  -> 缓存与并行
  -> Tree Shaking 与压缩
  -> 分包与缓存
  -> Source Map 策略

验证阶段
  -> build time
  -> main chunk size
  -> LCP / TTI
  -> cache hit
  -> error stack 可读性
```

## 1. 构建速度优化

| 手段 | 作用 | 注意点 |
| --- | --- | --- |
| 缩小 loader 范围 | 减少要编译的文件 | `include` / `exclude` 不要误伤源码 |
| 优化 resolve | 减少模块查找成本 | 合理配置 alias、extensions |
| filesystem cache | 复用上次构建结果 | 配置和 lockfile 变化要能失效 |
| swc / esbuild | 替代部分 Babel 转译 | Babel 插件能力不一定完全等价 |
| thread-loader | 多进程处理重任务 | 小项目可能因通信成本变慢 |

示例：

```js
module.exports = {
  cache: {
    type: "filesystem",
  },
  module: {
    rules: [
      {
        test: /\.[jt]sx?$/,
        include: path.resolve(__dirname, "src"),
        use: "babel-loader",
      },
    ],
  },
};
```

## 2. 产物体积优化

| 手段 | 作用 | 深挖 |
| --- | --- | --- |
| Tree Shaking | 删除未使用导出 | [Tree Shaking 原理](/md/工程化/构建优化/TreeShaking原理.md) |
| splitChunks | 拆分公共依赖和异步包 | [分包策略](/md/工程化/构建优化/分包策略.md) |
| 动态 import | 延迟加载非首屏模块 | [分包策略](/md/工程化/构建优化/分包策略.md) |
| Terser / CssMinimizer | 压缩 JS 和 CSS | 注意生产构建耗时 |
| 图片压缩 | 减少静态资源体积 | 关注视觉质量和构建成本 |

## 3. 加载性能优化

| 手段 | 适用场景 | 风险 |
| --- | --- | --- |
| `contenthash` | 静态资源长期缓存 | chunk 拆分不稳会影响缓存 |
| runtimeChunk | 降低 runtime 变化影响 | 多入口项目更明显 |
| CDN external | 超大稳定依赖 | CDN 可用性和版本一致性 |
| preload / prefetch | 控制资源加载时机 | 滥用会抢占关键资源 |
| Gzip / Brotli | 降低传输体积 | 需要服务端或 CDN 配合 |

## 4. Source Map 策略

开发环境优先反馈速度，生产环境优先安全和错误定位。

| 环境 | 建议 |
| --- | --- |
| 开发 | `eval-cheap-module-source-map` |
| 测试 | `source-map` |
| 生产 | `hidden-source-map` 并上传监控平台 |

详见：[Source Map 原理](/md/工程化/构建优化/SourceMap.md)。

## 高频追问

### 为什么不能一上来就加 `thread-loader`？

多进程有启动和通信成本。只有 Babel、TS 编译这类重任务才可能收益明显，小项目或轻任务可能更慢。

### `splitChunks` 拆得越细越好吗？

不是。拆得太细会增加请求数量、运行时调度成本和缓存碎片。要结合首屏体积、缓存命中和路由切换耗时判断。

### 如何证明优化有效？

至少给出前后指标：构建耗时、主包体积、vendor 体积、首屏资源数量、LCP、缓存命中率。没有指标的优化很难说明价值。
