# Webpack 构建流程

## 面试定位

Webpack 构建流程题考察你是否理解打包器的主链路。回答时要覆盖 `Compiler`、`Compilation`、loader、依赖图、chunk、assets 和 plugin 钩子。

## 面试回答

> Webpack 构建可以理解为从 entry 出发递归构建依赖图，最后生成可部署资源。启动时 Webpack 会读取配置、合并 CLI 参数、创建 `Compiler` 并注册插件。每次构建会创建一个 `Compilation`，从入口模块开始 resolve 文件，经过 loader 转换源码，再由 parser 分析 AST 里的 `import`、`require` 等依赖。Webpack 会递归构建 ModuleGraph，再根据入口、动态 import 和 splitChunks 生成 chunk。优化阶段会做 Tree Shaking、模块合并、压缩等处理，最后把 chunk 转成 assets，在 emit 阶段输出到磁盘。Plugin 则通过 Tapable 钩子介入这些生命周期。

一句话总结：

> Webpack = 初始化 Compiler -> 创建 Compilation -> 构建模块依赖图 -> 生成 chunk -> 优化 assets -> emit 输出。

## 核心原理

```text
初始化
  -> 读取配置
  -> 创建 Compiler
  -> 注册 plugin

编译
  -> 创建 Compilation
  -> 从 entry 创建入口模块
  -> resolve 查找文件
  -> loader 转换模块
  -> parser 分析依赖

图构建
  -> 递归构建 ModuleGraph
  -> 生成 ChunkGraph

优化输出
  -> Tree Shaking / splitChunks / runtime
  -> 生成 assets
  -> emit 到 output.path
```

## 一个例子

```text
src/main.js
  -> import App from "./App"
  -> import "./style.css"

style.css
  -> css-loader 解析 CSS 依赖
  -> style-loader 或 MiniCssExtractPlugin 处理注入/抽离
```

Webpack 会把 JS、CSS、图片等不同资源都纳入模块图。区别在于不同类型文件会先经过不同 loader 转成 Webpack 能理解的模块。

## 高频追问

### Webpack 为什么需要 loader？

Webpack 核心主要理解 JavaScript 模块。CSS、图片、字体、TypeScript、Vue SFC 等都需要 loader 转换，才能进入统一的模块依赖图。

### 动态 import 在构建时发生什么？

动态 import 会形成异步 chunk。Webpack 构建阶段生成单独的 chunk 文件，运行时再通过 Webpack runtime 按需加载。

### Watch 模式下每次都重新创建 Compiler 吗？

通常不会。`Compiler` 代表运行环境会复用，文件变化会触发新的 `Compilation`，重新生成本次构建的模块、chunk 和 assets。
