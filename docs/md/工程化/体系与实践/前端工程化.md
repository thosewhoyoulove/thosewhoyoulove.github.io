# 前端工程化

## 面试定位

这道题适合中高级前端面试。回答要从模块化、构建、质量、测试、CI/CD、监控和团队协作展开，并能结合自己项目讲落地。

## 核心原理

前端工程化是用**工具链 + 规范 + 流程**把前端开发从「手工作坊」变成「工业化流水线」，核心目标是提效、降错、可协作。体系概览见 [工程化体系](/md/工程化/体系与实践/工程化体系.md)。

---

## 1. 模块化

把复杂系统拆成独立、可复用的单元，解决命名冲突、依赖混乱、代码复用问题。

| 维度 | 方案 |
| --- | --- |
| JS 模块化 | ES Module（主流）、CommonJS（Node）、AMD/UMD（历史） |
| CSS 模块化 | CSS Modules、CSS-in-JS、BEM 命名约定 |
| 资源模块化 | Webpack/Vite 把图片、字体、JSON 都当模块导入 |

**效果**：代码按职责分文件；依赖关系显式声明（`import`）；打包工具可做 Tree Shaking。

---

## 2. 构建工具

把源代码编译、打包、优化成可上线的产物。

| 工具 | 定位 |
| --- | --- |
| Webpack | 大而全，生态成熟，配置复杂，适合大型项目 |
| Vite | 开发模式用 ESM 原生加载，生产用 Rollup，启动快 |
| esbuild / Rollup | 极速打包 / 库打包专用 |
| Turborepo / Nx | Monorepo 场景的构建加速和任务编排 |

构建过程做的事：

- **转译**：TypeScript、JSX、新语法降级。
- **打包**：模块合并、Chunk 拆分。
- **优化**：压缩、Tree Shaking、代码分割、资源哈希。

详见 [Webpack 构建流程](/md/工程化/Webpack/构建流程.md)、[为什么 Vite 快](/md/工程化/Vite/为什么Vite快.md)。

---

## 3. 代码质量

用静态分析 + 自动化检查，在代码合并前拦截问题。

| 工具 | 作用 |
| --- | --- |
| ESLint | JS/TS 语法、最佳实践检查 |
| Prettier | 代码格式统一 |
| Stylelint | CSS/Less/Sass 规范检查 |
| TypeScript | 类型检查，编译期发现错误 |
| Husky + lint-staged | Git commit 前自动跑检查 |

**效果**：团队代码风格统一；低级错误（未定义变量、类型不匹配）在本地被拦截，不流入代码库。

---

## 4. 版本控制与协作

Git 是基础设施，工程化在协作层面的扩展：

- **分支策略**：Git Flow、GitHub Flow、主干开发。
- **Commit 规范**：Conventional Commits（feat/fix/docs 等前缀）。
- **Code Review**：PR/MR 机制 + 自动化检查卡点。
- **Monorepo**：多包在一个仓库统一管理，Lerna / pnpm Workspaces / Turborepo。

详见 [Monorepo](/md/工程化/体系与实践/Monorepo.md)。

---

## 5. 自动化测试

| 类型 | 作用 | 工具 |
| --- | --- | --- |
| 单元测试 | 函数/模块级别的逻辑验证 | Jest、Vitest |
| 集成测试 | 多模块协作场景验证 | Testing Library |
| E2E 测试 | 模拟真实用户操作 | Cypress、Playwright |
| 视觉回归测试 | UI 变化对比 | Percy、Chromatic |

**工程化价值**：

- 改代码有信心：跑一遍测试就知道有没有改坏。
- CI 里自动跑，合并前强制通过。

---

## 6. CI/CD 持续集成与部署

把构建、测试、部署串成自动化流水线。

```
代码推送 → 触发 CI → 安装依赖 → Lint + 测试 → 构建 → 部署
```

| 平台 | 典型场景 |
| --- | --- |
| GitHub Actions | 开源项目、GitHub 托管 |
| GitLab CI | 企业私有 GitLab |
| Jenkins | 传统企业、高度自定义 |
| Vercel / Netlify | 前端项目一键部署 |

**效果**：

- 每次提交自动跑测试 + 构建，问题早发现。
- 主分支合并后自动部署，缩短交付周期。

详见 [CI&CD](/md/工程化/体系与实践/CI&CD.md)。

## 7. 工程化的实际落地

一个成熟的前端项目通常包含：

```plaintext
├── .husky/              # Git hooks
├── .eslintrc.js         # ESLint 配置
├── .prettierrc          # Prettier 配置
├── tsconfig.json        # TypeScript 配置
├── vite.config.ts       # 构建配置
├── package.json
│   ├── scripts          # npm scripts（dev/build/test/lint）
│   └── lint-staged      # 提交前检查范围
├── .github/workflows/   # CI 配置
└── src/
```

开发流程：

1. `npm run dev` 启动开发服务器（Vite/Webpack）。
2. 写代码，ESLint + Prettier 实时检查和格式化。
3. `git commit` 触发 Husky → lint-staged → ESLint + 测试。
4. 推送代码，CI 自动跑完整测试 + 构建。
5. 合并到主分支，自动部署。

## 小结

| 维度 | 核心收益 |
| --- | --- |
| 模块化 | 代码可维护、可复用 |
| 构建工具 | 开发提效、产物优化 |
| 代码质量 | 统一规范、减少低级错误 |
| 版本控制 | 可追溯、可协作 |
| 自动化测试 | 回归有保障、重构有信心 |
| CI/CD | 交付自动化、缩短反馈周期 |

前端工程化不是「选一个工具」，而是**一整套流程 + 规范 + 工具链的组合拳**，目标是让团队专注于业务逻辑，而非重复的手工劳动。

## 面试回答

可以这样答：

> 前端工程化是用工具、规范和流程保障开发效率和交付质量。模块化解决代码组织和复用；构建工具负责 TS/JSX 转译、打包、代码分割、压缩和资源 hash；代码质量通过 ESLint、Prettier、Stylelint、TypeScript、Husky、lint-staged 在本地和提交前拦截问题；测试体系用单测、集成测试、E2E 保证回归；CI/CD 把安装依赖、检查、测试、构建、部署串成自动流程；上线后还要有错误监控、性能监控和回滚机制。实际落地时我会根据项目规模取舍，不会为了工具而工具。

## 高频追问

### 工程化和架构有什么区别？

架构更多关注系统模块边界、数据流、扩展性；工程化更多关注开发、构建、测试、发布、质量和协作流程。两者会互相影响。

### 如何推动老项目工程化改造？

先从低风险高收益开始，例如统一格式化、lint、构建脚本、CI 检查；再逐步补 TypeScript、测试、构建优化、目录规范和发布流程。

### 如何衡量工程化收益？

可以看构建耗时、发布耗时、线上错误率、回滚耗时、PR 检查通过率、新人上手时间等指标。
