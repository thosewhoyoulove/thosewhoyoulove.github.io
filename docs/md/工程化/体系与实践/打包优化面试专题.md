# 打包优化面试专题

## 面试定位

这篇是工程化打包题的总览入口，不再重复展开所有细节。面试时先用它串起回答框架，再按追问跳到 Webpack、Vite、Tree Shaking、HMR、Source Map、分包等单题章节。

## 面试回答

> 打包优化我会分成开发效率和线上加载性能两条线。Webpack 是从入口出发构建依赖图，loader 负责模块转换，plugin 通过 Tapable 钩子介入生命周期，最后生成 chunk 和 assets；Vite 开发态基于浏览器原生 ESM 按需编译，并用 esbuild 预构建依赖，所以冷启动和 HMR 更快。优化时我不会直接堆配置，而是先用构建耗时、包体积、请求瀑布图和线上性能指标定位瓶颈，再分别处理缓存、缩小编译范围、分包、Tree Shaking、Source Map、压缩和长期缓存。

一句话总结：

> 打包优化不是某个配置项，而是围绕构建速度、产物体积、缓存命中、调试定位和线上加载体验做取舍。

## 核心链路

```text
源码入口
  -> 模块解析
  -> loader / transform
  -> 依赖图
  -> chunk 拆分
  -> Tree Shaking / 压缩 / hash
  -> 输出 assets
  -> 浏览器加载与缓存
```

## 项目表达模板

```text
背景：项目启动慢 / 首屏包大 / 线上错误难定位
定位：用 profile、bundle analyzer、Network、Lighthouse 找瓶颈
方案：缓存、缩小编译范围、分包、Tree Shaking、Source Map 策略
验证：构建耗时、主包体积、LCP、缓存命中、错误堆栈可读性
取舍：拆包过细、source map 泄露、CDN 稳定性、构建配置复杂度
```

## 高频追问

### 打包优化先从哪里下手？

先分析再优化。构建慢看 loader/plugin 耗时、缓存命中和编译范围；产物大看 bundle 分析、重复依赖和分包；线上慢看首屏资源、请求瀑布图和缓存策略。

### Vite 快是不是代表生产构建也不用打包？

不是。Vite 快主要在开发态，生产环境仍需要 Rollup 打包来控制请求数量、做 Tree Shaking、压缩、CSS 提取和 chunk 拆分。

### 分包和 Tree Shaking 谁更重要？

解决的问题不同。Tree Shaking 删除没用代码，分包改变代码加载时机和缓存结构。实际项目通常两者都需要，再结合压缩和缓存策略验证效果。
