# 前端工程化体系

## 面试定位

工程化体系题常用于判断候选人是否只会写页面，还是能从团队效率、质量保障、构建部署和长期维护角度思考项目。回答时不要只罗列工具，要讲清每一层解决的问题。

## 核心原理

前端工程化的本质是用工具链、规范和流程，把开发、测试、构建、发布、监控这些重复且容易出错的环节自动化、标准化。

```plaintext
前端工程化体系
├── 代码管理
│   ├── Git / 分支策略 / Code Review
│   └── Monorepo / MultiRepo
├── 依赖管理
│   ├── npm / yarn / pnpm
│   └── lockfile / workspace / 版本治理
├── 构建工具
│   ├── Webpack / Vite / Rollup / esbuild
│   └── 代码分割 / Tree Shaking / 资源 hash
├── 代码质量
│   ├── ESLint / Prettier / Stylelint
│   ├── TypeScript
│   └── Husky / lint-staged / Commitlint
├── 测试体系
│   ├── 单元测试 / 集成测试 / E2E
│   └── 视觉回归 / 覆盖率
├── CI/CD
│   ├── 安装依赖 / Lint / Test / Build
│   └── 部署 / 灰度 / 回滚 / 通知
└── 线上保障
    ├── 性能监控 / 错误监控 / 日志
    └── 告警 / 回滚 / 复盘
```

## 面试回答

可以这样答：

> 前端工程化不是某个工具，而是一整套提升交付效率和质量的体系。代码管理层面要有 Git 分支策略、Code Review、提交规范；依赖管理层面要用 lockfile 保证安装一致性，大项目可以用 pnpm workspace 或 Monorepo；构建层面用 Vite、Webpack、Rollup 做转译、打包、压缩、Tree Shaking 和代码分割；质量层面用 ESLint、Prettier、TypeScript、Husky、lint-staged 在提交前拦截问题；CI/CD 层面在合并前自动跑 lint、测试、构建，主分支自动部署，并保留回滚能力；线上还要有性能监控和错误监控。工程化的目标是让问题尽早暴露，让发布可重复、可追踪、可回滚。

## 高频追问

### 工程化最核心的收益是什么？

提效、降错、可协作、可追溯。不是为了堆工具，而是让团队稳定交付。

### 小项目需要工程化吗？

需要，但不用过度。小项目至少要有格式化、lint、构建脚本和基础发布流程；大型项目再引入测试、CI/CD、Monorepo、监控等更完整体系。

### 工程化如何体现业务价值？

减少线上事故、缩短交付周期、降低新人上手成本、提升代码可维护性和发布稳定性。
