# Monorepo

## 面试定位

Monorepo 常用于多应用、多包、组件库、SDK 和微前端协作场景。面试重点不是“它是什么”，而是它解决了什么协作问题、带来什么工程成本，以及 pnpm workspace、Turborepo、Nx 分别负责哪一层。

## 面试回答

> Monorepo 是把多个相关项目或包放在同一个 Git 仓库里统一管理。它适合多个应用共享组件库、工具库、类型定义、SDK，或者跨包改动很频繁的场景。优点是依赖、规范、脚本和 CI 可以统一，跨包修改能在一个 PR 里完成，版本联动更清晰。代价是仓库变大后安装、构建、测试和权限管理都会更复杂，所以通常用 pnpm workspace 管包和本地链接，用 Turborepo 或 Nx 做任务编排、增量构建、缓存和 affected 分析。是否选择 Monorepo，关键看项目之间是否强关联，而不是看它听起来更高级。

一句话总结：

> Monorepo 用统一仓库降低强关联项目的协作成本，但必须配套包边界、任务缓存和 CI 影响范围分析。

## 核心原理

```text
apps/
  web/
  admin/
packages/
  ui/
  utils/
  api-client/

统一依赖 -> 本地包链接 -> 任务拓扑 -> 增量构建 -> 统一 CI
```

## Monorepo vs MultiRepo

| 维度 | Monorepo | MultiRepo |
| --- | --- | --- |
| 代码共享 | 本地包直接引用，跨包改动方便 | 需要发布版本或同步多个仓库 |
| 规范治理 | 统一 lint、test、build、commit 规范 | 每个仓库单独维护 |
| CI/CD | 可做 affected 分析和增量构建 | 流程相互独立 |
| 权限边界 | 需要额外设计目录权限和发布权限 | 天然按仓库隔离 |
| 适用场景 | 强关联、多包协作、多应用共享 | 项目独立、团队隔离强 |

## 工具分层

| 工具 | 解决什么 |
| --- | --- |
| pnpm workspace | workspace 包管理、本地包链接、依赖安装 |
| Turborepo | 任务编排、缓存、增量构建 |
| Nx | 项目图、affected 分析、任务编排、生成器 |
| Changesets | 多包版本管理和发布 |

## 常见风险

- 包边界不清，业务代码互相穿透。
- CI 全量执行，仓库越大越慢。
- 权限和发布流程复杂。
- 共享包变更影响面大，需要版本和变更说明。
- 工具链学习成本高。

## 项目表达

```text
背景：多个应用共享组件、工具和接口类型，跨仓库同步成本高。
方案：使用 pnpm workspace 管理 packages，用统一 lint/test/build 规范约束边界。
优化：接入 Turborepo/Nx 缓存和 affected 分析，CI 只跑受影响任务。
结果：跨包改动一次 PR 完成，重复配置减少，构建时间通过缓存下降。
取舍：需要维护包边界、发布流程和权限治理。
```

## 高频追问

### pnpm workspace 解决什么？

它负责 workspace 包发现、本地包链接和依赖安装。pnpm 的内容寻址存储和硬链接机制也能减少重复依赖占用。

### Turborepo 和 Nx 解决什么？

它们解决任务编排、依赖拓扑、增量构建、缓存和受影响项目分析，避免每次都全量构建。

### Monorepo 一定比 MultiRepo 好吗？

不是。项目完全独立、权限隔离强、发布节奏完全不同的时候，MultiRepo 更简单。Monorepo 适合共享代码多、协同变更多的项目。
