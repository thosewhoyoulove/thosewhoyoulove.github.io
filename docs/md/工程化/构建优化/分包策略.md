# 分包策略

## 面试定位

分包题常和首屏优化、缓存策略、Webpack `splitChunks`、Vite/Rollup `manualChunks` 一起问。回答重点是：为什么分、按什么维度分、分包过细有什么代价。

## 面试回答

> 分包的目标不是把文件拆得越多越好，而是在首屏体积、缓存命中和请求数量之间取平衡。常见策略包括路由级动态导入，把非首屏页面拆出去；把 React、Vue、ECharts 这类稳定第三方依赖拆成 vendor chunk，提高长期缓存命中；把多页面共享的业务模块拆成 common chunk；对大弹窗、图表、编辑器做组件级懒加载。分包后要看首屏 JS 体积、请求瀑布图、缓存命中率和路由切换耗时，避免拆得太细导致请求调度成本上升。

一句话总结：

> 分包是用更合理的 chunk 结构换首屏更小、缓存更稳，但不能牺牲过多请求调度成本。

## 核心原理

```text
入口 main
  -> 首屏必须代码：进入主包
  -> 路由页面代码：动态 import 拆包
  -> 稳定第三方依赖：vendor 包
  -> 多处复用业务模块：common 包
  -> 大型非首屏能力：交互时懒加载
```

Webpack 示例：

```js
module.exports = {
  optimization: {
    splitChunks: {
      chunks: "all",
      cacheGroups: {
        react: {
          test: /[\\/]node_modules[\\/](react|react-dom)[\\/]/,
          name: "react-vendor",
          priority: 20,
        },
        vendors: {
          test: /[\\/]node_modules[\\/]/,
          name: "vendors",
          priority: 10,
        },
      },
    },
  },
};
```

动态导入：

```js
const ChartPanel = () => import("./ChartPanel");
```

## 常见策略

| 策略 | 适用场景 | 风险 |
| --- | --- | --- |
| 路由级分包 | SPA 页面多，首屏只用少量页面 | 路由切换可能等待 |
| vendor 分包 | React/Vue/ECharts 等稳定依赖 | vendor 过大仍阻塞首屏 |
| common 分包 | 多入口或多页面共享模块 | 过度共享会增加请求 |
| 组件懒加载 | 大弹窗、编辑器、地图、图表 | 首次交互有加载延迟 |
| CDN external | 超大且稳定的库 | 依赖 CDN 可用性和版本治理 |

## 高频追问

### 分包越细越好吗？

不是。分包过细会增加请求数量、运行时调度成本和缓存碎片。HTTP/2 能缓解请求并发问题，但不能消除浏览器调度、解析和执行成本。

### vendor 包应该怎么拆？

不要把所有 `node_modules` 都粗暴塞进一个巨大 vendor。可以按稳定性和体积分层，比如框架包、图表包、编辑器包分开，让变化频率不同的依赖互不影响缓存。

### 如何验证分包是否有效？

看首屏 JS 体积、LCP、请求瀑布图、主包变化频率、缓存命中率和异步路由加载耗时。只看 chunk 数量没有意义。
