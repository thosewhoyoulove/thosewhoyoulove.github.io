# 一份实用的 vite.config.ts 配置（适配 Vue + TypeScript）

## 面试定位

Vite 配置题通常考察你是否真正做过项目工程化。回答时要能说明每个配置解决的问题，而不是只会贴配置。

## 核心原理

`vite.config.ts` 是 Vite 项目的工程入口，负责插件接入、路径解析、环境变量、CSS 预处理、开发代理、构建输出和 Rollup 分包等配置。配置的目标是让开发体验、构建产物和部署环境保持一致。

```typescript

// vite.config.ts
import { defineConfig, loadEnv } from 'vite'
import vue from '@vitejs/plugin-vue' // Vue 插件
import path from 'path'
import AutoImport from 'unplugin-auto-import/vite' // 自动导入 API
import Components from 'unplugin-vue-components/vite' // 自动导入组件
import { ElementPlusResolver } from 'unplugin-vue-components/resolvers' // Element Plus 组件解析器
import vueSetupExtend from 'vite-plugin-vue-setup-extend' // 支持 setup 语法糖扩展

// https://vitejs.dev/config/
export default defineConfig(({ mode }) => {
  // 根据当前工作目录中的 `mode` 加载 .env 文件
  const env = loadEnv(mode, process.cwd())

  return {
    base: '/', // 开发或生产环境服务的公共基础路径
    plugins: [
      vue(), // 提供 Vue 3 单文件组件支持
      vueSetupExtend(), // 启用 setup 语法糖扩展
      AutoImport({
        imports: ['vue', 'vue-router', 'pinia'], // 自动导入 Vue、Vue Router、Pinia 相关 API
        dts: 'src/auto-imports.d.ts', // 生成类型声明文件
        resolvers: [ElementPlusResolver()] // 自动导入 Element Plus 组件
      }),
      Components({
        dts: 'src/components.d.ts', // 生成组件类型声明文件
        resolvers: [ElementPlusResolver()] // 自动注册 Element Plus 组件
      })
    ],
    resolve: {
      alias: {
        '@': path.resolve(__dirname, './src') // 设置 @ 指向 src 目录
      }
    },
    css: {
      preprocessorOptions: {
        scss: {
          additionalData: `@use "@/styles/variables.scss" as *;` // 全局引入 scss 变量
        }
      }
    },
    server: {
      host: '0.0.0.0', // 监听所有地址
      port: 5173, // 开发服务器端口号
      open: true, // 启动时自动打开浏览器
      proxy: {
        '/api': { // 代理配置
          target: env.VITE_API_URL, // 目标地址
          changeOrigin: true, // 允许跨域
          rewrite: (path) => path.replace(/^\/api/, '') // 路径重写
        }
      }
    },
    build: {
      outDir: 'dist', // 输出目录
      assetsDir: 'static', // 静态资源目录
      sourcemap: false, // 是否生成 source map
      rollupOptions: {
        output: {
          manualChunks(id) { // 手动分块配置
            if (id.includes('node_modules')) {
              // 将 node_modules 中的每个包分割成单独的块
              return id
                .toString()
                .split('node_modules/')[1]
                .split('/')[0]
                .toString()
            }
          }
        }
      }
    }
  }
})


```

## 面试回答

可以这样答：

> Vite 常见配置主要分几类。插件层面用 `@vitejs/plugin-vue` 支持 Vue SFC，用自动导入插件减少重复 import；`resolve.alias` 用来配置 `@` 指向 `src`，避免复杂相对路径；`css.preprocessorOptions` 可以注入全局 SCSS 变量；`server.proxy` 用于开发环境代理接口，解决本地跨域并保持请求路径一致；`build` 里配置输出目录、sourcemap、资源目录和 Rollup 分包策略。环境变量一般通过 `loadEnv(mode, process.cwd())` 读取，注意只有 `VITE_` 前缀变量会暴露给客户端。

## 高频追问

### Vite 环境变量为什么要 VITE_ 前缀？

为了避免把服务端密钥误暴露给客户端。只有 `VITE_` 前缀变量会被注入到客户端代码中。

### manualChunks 能不能按每个 node_modules 包都拆？

不建议无脑拆太细。chunk 太多会增加请求和调度成本，应该按业务、体积和缓存收益设计分包。

### 开发代理能解决线上跨域吗？

不能。Vite proxy 只在本地 dev server 生效。线上跨域要通过服务端 CORS、网关代理或同域部署解决。
