# 框架开放性面试题

## 面试定位

开放性题目没有标准答案，面试官看的是：**是否理解框架设计取舍、能否结合复杂业务表达、有没有工程化权衡**。

回答结构建议：**先给观点（1～2 句）→ 展开机制差异 → 说心智负担/代价 → 项目里怎么权衡 → 边界（不踩「哪个框架更好」）**。

本文按题号收录，每题 **先面试回答、后原理**。

---

## 第 1 题：从 Vue 响应式到 React 不可变，复杂状态的最大心智负担是什么？如何权衡「暴力重刷」与「精准更新」？

### 题目（原题）

> 从 Vue 的响应式思维切换到 React 的「不可变数据（Immutability）」思维，你认为在维护复杂应用状态时，**最大的开发心智负担**是什么？你会如何权衡 React 的「暴力重刷」与 Vue 的「精准更新」？

---

### 面试回答

> 我觉得最大的心智负担，不是语法上要多写 `{ ...state }`，而是 **从「改数据就会自动更新对的地方」变成「我要自己设计：状态放哪、谁订阅、何时算变了、哪些 render 可以跳过」**。
>
> Vue 里我可以 `state.list.push(item)`，框架通过依赖追踪知道哪些组件用了 `list`，做 **组件级精准更新**。React 里同一件事我要 **不可变地** 产生新引用，并且默认从 setState 的组件往下 **整棵子树 reconcile**——看起来像「暴力重刷」，但 reconcile 是在 Virtual DOM 层做 diff，不等于无脑刷 DOM。
>
> 权衡上我不会简单选边，而是 **分层**：
> - **局部 UI 状态**：组件内 `useState`，接受 React 默认更新模型，靠组件拆分控范围。
> - **跨模块共享、要可追溯**：Redux / Zustand + 不可变更新（Immer），换可预测性和调试，接受一点样板。
> - **性能敏感路径**：`memo` + 稳定 props、`useMemo` 缓存派生值、列表虚拟化；必要时把高频变更状态 **下放到叶子**，避免从根重渲染。
> - **从 Vue 迁过来时**：最该改的习惯是 **不要原地 mutate props/state**，以及 **不要把所有状态堆在顶层 Context**——那才会真的「暴力重刷」整个应用。
>
> 所以我的结论是：React 的心智负担在 **显式建模更新边界**；Vue 的心智负担在 **搞清响应式丢失场景**（解构、Map、异步）。复杂应用里两者都要做架构，只是 React 把优化权更多交给开发者。

一句话总结：

> Vue 自动回答「更新谁」；React 要求你回答「变了什么、从哪 render、如何跳过」——复杂状态的心智负担在 **更新边界的显式设计**。

---

### 核心原理

#### 两种更新哲学

| 维度 | Vue（响应式） | React（不可变 + reconcile） |
| --- | --- | --- |
| 数据写法 | 可变对象，`reactive`/`ref` | 不可变更新，新引用触发 render |
| 谁需要更新 | 依赖收集 → **组件级精准** | 从 setState 节点向下 **re-render 子树** |
| DOM 更新 | 仍走 patch，但范围通常更小 | diff 后 **最小 DOM 变更** |
| 默认性能 | 下限高，少优化也能用 | 大子树/频繁父级更新需手动优化 |
| 调试模型 | 追踪依赖链较隐式 | 状态快照清晰，时间旅行友好 |

```text
Vue:  改数据 → 命中 dep → 调度相关组件 update → patch
React: setState → render 子树 → reconcile(diff) → commit DOM
```

#### 「暴力重刷」到底刷什么？

面试里要纠正一个误解：

- React **重刷的是 render 阶段**（重新执行函数组件、生成新 Element 树），不是每次都重绘整个 DOM。
- **commit 阶段**仍按 diff 结果做最小 DOM 操作。
- 真正昂贵的是：**无意义的 render 计算**（大组件、重计算、子组件未 memo）和 **级联重渲染**（Context、顶层 state）。

Vue 的「精准更新」指 **调度范围** 更细（组件级）；React 通过 **拆分组件 + memo + 状态下沉** 达到类似效果，但需要开发者主动做。

#### 复杂状态下，React 侧四类心智负担

1. **不可变更新的表达成本**  
   深层嵌套要 spread 或 Immer；团队要统一「禁止 mutate」规范。

2. **更新范围的自觉**  
   父组件 state 变 → 默认所有未 memo 的子组件 re-render；要会画 **状态归属图**。

3. **引用稳定性**  
   `useCallback` / `useMemo` 不是炫技，而是给 `React.memo` 子组件 **稳定的 props 引用**，否则 memo 失效。

4. **异步与批处理**  
   `setState` 批处理、闭包旧 state、`useEffect` 依赖——和「可变对象改完即生效」的直觉不同。

#### 如何权衡：不是选框架，是选策略

| 场景 | 倾向做法 | 原因 |
| --- | --- | --- |
| 表单、局部交互 | 组件内 state，不 premature optimize | 更新范围天然小 |
| 全局用户/权限 | Context 只放低频字段，或 Zustand  selector | 避免根节点频繁 render |
| 大列表、表格 | 虚拟列表 + 行级 memo + 稳定 rowKey | 减少 reconcile 规模 |
| 可预测业务流 | Redux Toolkit + Immer | 不可变成本由工具吸收 |
| 从 Vue 迁移 | 先拆组件边界，再谈 memo | 先缩小「重刷」范围 |

**Vue 侧也要知道的边界**：解构 reactive 丢响应式、大对象深度代理成本、`shallowRef` 等——说明「精准」不是免费的。

---

### 项目举证（口述模板）

> 我们有个复杂 B 端筛选页，之前在 Vue 里一个大 `reactive` 对象驱动整页，改一个字段只更新相关区块，很顺。迁到 React 后如果把整页 state 放在一个 reducer 里，任何字段变化都会导致整页 re-render，Profiling 里 Long Task 明显。
>
> 我们的改法是：筛选条件、列表数据、详情抽屉 **拆成三个状态域**；列表行用 `memo`，回调用 `useCallback` 稳定引用；派生筛选结果用 `useMemo`。这样保留了 React 不可变和可追溯，又把 render 范围压到接近 Vue 的「按需更新」体验。

### 高频追问

#### React 既然会 diff，为什么还要不可变？

不可变让 **「是否变化」** 在 state 层就可判定（引用比较）；mutable 原地改，引用不变，React 可能跳过更新，且破坏快照和调试。diff 解决的是 **render 之后** 的 DOM 最小变更，不能替代 state 语义。

#### `useMemo` / `memo` 是不是 React 在补「精准更新」？

可以这么理解：框架默认 reconcile 子树，memo 是开发者声明 **「props 没变就跳过 render」**，类似手动依赖边界。Vue 3 的 `computed`/`watchEffect` 是自动依赖；React 更显式。

#### Vue 就没有「暴力重刷」吗？

有。例如父组件 forceUpdate、大列表未做虚拟化、响应式对象过大导致依赖过宽。只是 **默认路径** 更偏精准。

#### 你会推荐复杂应用选 Vue 还是 React？

开放性题避免站队。答：**团队栈、生态、招聘、现有代码** 优先；复杂状态两者都要 **分域、限范围、可观测（Profiler）**。
