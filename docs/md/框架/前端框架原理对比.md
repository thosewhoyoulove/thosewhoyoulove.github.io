# 前端框架原理对比

面试中偶尔会问"你了解其他框架吗"、"Svelte/Solid 和 Vue/React 有什么区别"。不需要深入源码，但要能说清楚核心思路的差异。

---

## 问题：前端框架本质上在解决什么问题？

所有前端框架都在解决同一个核心问题：**数据变了，怎么高效地更新 DOM？**

不同框架的区别在于"怎么知道数据变了"和"怎么更新 DOM"这两步的策略不同：

| 框架 | 怎么知道数据变了 | 怎么更新 DOM |
| --- | --- | --- |
| **React** | 开发者调 setState 显式通知 | Virtual DOM diff 整棵子树 |
| **Vue** | Proxy 自动追踪依赖 | Virtual DOM diff（但只 diff 依赖变了的组件） |
| **Svelte** | 编译期分析赋值语句 | 编译生成直接操作 DOM 的代码，无 Virtual DOM |
| **Solid** | 细粒度信号（Signal）运行时追踪 | 直接操作 DOM，无 Virtual DOM |
| **Angular** | Zone.js 拦截异步操作 + 脏检查 | 增量 DOM（Incremental DOM） |

---

## 问题：Svelte 的原理是什么？和 Vue/React 有什么区别？

### 核心思路：编译时框架

Svelte 的核心理念是**把框架的工作从运行时转移到编译时**。

Vue 和 React 都需要在浏览器里跑一个运行时（Virtual DOM diff、响应式系统等）。Svelte 在构建阶段就把组件编译成**直接操作 DOM 的命令式代码**，运行时几乎没有框架开销。

### 响应式：编译期赋值分析

```svelte
<script>
  let count = 0
  
  function increment() {
    count += 1  // 编译器看到赋值，自动生成更新 DOM 的代码
  }
</script>

<button on:click={increment}>{count}</button>
```

编译后大致变成：

```js
function increment() {
  count += 1
  $$invalidate('count', count)  // 编译器插入的更新调用
  // 直接更新对应的 DOM 文本节点，不走 diff
}
```

**和 Vue 的区别**：
- Vue 用 Proxy 在运行时追踪依赖，Svelte 在编译时就确定了"哪个变量影响哪个 DOM 节点"
- Vue 有 Virtual DOM 做 diff，Svelte 直接生成 `element.textContent = count` 这样的代码

**和 React 的区别**：
- React 需要 setState 显式触发，Svelte 直接赋值就行
- React 整棵子树重新 render + diff，Svelte 精确到具体的 DOM 操作

### 优势

- **包体积小**：没有运行时框架代码，打包后只有业务逻辑
- **性能好**：没有 Virtual DOM 的创建和 diff 开销
- **写法简洁**：不需要 useState、ref 这些 API，直接用变量

### 劣势

- **生态小**：组件库、工具链远不如 Vue/React 成熟
- **大型项目验证少**：企业级复杂应用的案例不多
- **编译产物膨胀**：组件多了之后，每个组件都带一份更新逻辑，总体积可能反超

## 问题：Solid.js 的原理是什么？

### 核心思路：细粒度响应式 + 无 Virtual DOM

Solid 结合了 React 的 JSX 写法和 Vue 的响应式思路，但比两者都更极端：

```jsx
import { createSignal } from 'solid-js'

function Counter() {
  const [count, setCount] = createSignal(0)
  
  // 这个函数只执行一次！不像 React 每次 render 都重新执行
  console.log('组件函数只跑一次')
  
  return <button onClick={() => setCount(c => c + 1)}>{count()}</button>
}
```

### 关键区别：组件函数只执行一次

- **React**：状态变了 → 组件函数重新执行 → 生成新 Virtual DOM → diff → 更新 DOM
- **Solid**：组件函数只在挂载时执行一次，建立好响应式绑定后，数据变化直接更新对应的 DOM 节点

### Signal（信号）

Solid 的响应式基于 Signal：

```jsx
const [count, setCount] = createSignal(0)
// count() 是 getter，调用时收集依赖
// setCount() 是 setter，调用时通知订阅者
```

和 Vue 的 ref 很像，但 Solid 的追踪粒度更细——它追踪的不是"哪个组件依赖了这个数据"，而是"哪个 DOM 表达式依赖了这个数据"。

### 和 Vue 的区别

| | Vue | Solid |
| --- | --- | --- |
| 响应式粒度 | 组件级（数据变了重新 render 组件） | DOM 表达式级（数据变了直接改 DOM） |
| Virtual DOM | 有 | 无 |
| 组件函数执行 | 每次更新都重新执行 render | 只执行一次 |
| 模板/JSX | 模板（可选 JSX） | JSX |

### 优势

- 性能极好（benchmark 常年第一梯队）
- 没有 Virtual DOM 开销
- 不需要 memo/useCallback 这些优化手段
- JSX 写法对 React 开发者友好

### 劣势

- 生态很小，组件库少
- "组件函数只跑一次"的心智模型和 React 不同，容易踩坑
- 解构 Signal 会丢失响应式（和 Vue 的 reactive 解构问题类似）

## 问题：Angular 的原理是什么？和 Vue/React 有什么区别？

### 核心思路：Zone.js + 脏检查 + 依赖注入

Angular 是一个**全功能框架**（不只是视图层），内置了路由、HTTP、表单、DI 等。

### 变更检测：Zone.js

Angular 用 Zone.js 猴子补丁了所有异步 API（setTimeout、Promise、addEventListener 等），任何异步操作完成后自动触发变更检测。

```typescript
@Component({
  template: `<button (click)="increment()">{{ count }}</button>`
})
class CounterComponent {
  count = 0
  increment() {
    this.count++  // 不需要 setState 或 ref
    // Zone.js 检测到事件处理完成，自动触发变更检测
  }
}
```

### 脏检查

变更检测时，Angular 从根组件开始，自上而下检查每个组件的模板绑定是否变了（比较新旧值）。默认策略是**每次都检查整棵树**。

**优化**：`ChangeDetectionStrategy.OnPush`——只有 Input 引用变了或组件内部触发了事件才检查该子树。

### Angular 16+ 的 Signals

Angular 也在引入 Signal（类似 Vue 的 ref / Solid 的 createSignal），逐步替代 Zone.js 的脏检查，走向细粒度响应式。

### 和 Vue/React 的区别

| | Angular | Vue | React |
| --- | --- | --- | --- |
| 定位 | 全功能框架 | 渐进式框架 | UI 库 |
| 变更检测 | Zone.js + 脏检查 | Proxy 响应式 | setState 触发 |
| 模板 | HTML + 指令 | Template | JSX |
| 语言 | TypeScript（强制） | JS/TS（可选） | JS/TS（可选） |
| DI | 内置依赖注入 | provide/inject（轻量） | Context（轻量） |
| 学习曲线 | 高 | 低-中 | 中 |
| 适合场景 | 大型企业应用 | 中后台/通用 | 灵活/跨平台 |

## 问题：Virtual DOM 到底是好是坏？为什么 Svelte/Solid 不用它？

### Virtual DOM 的价值

1. **跨平台**：Virtual DOM 是一层抽象，可以渲染到 DOM、Native（React Native）、Canvas 等
2. **心智模型简单**：开发者只需要描述"UI 应该长什么样"，框架负责算出怎么从旧状态变到新状态
3. **批量更新**：多次状态变更可以合并成一次 DOM 操作

### Virtual DOM 的代价

1. **创建 VNode 的开销**：每次 render 都要创建一棵新的虚拟树
2. **Diff 的开销**：即使只有一个文本变了，也要遍历比较
3. **内存占用**：同时存在新旧两棵树

### 为什么 Svelte/Solid 不用

它们的核心观点是：**如果编译器/运行时已经精确知道"哪个数据影响哪个 DOM 节点"，就不需要 diff 来"找出"变化了**。

- Svelte：编译时就确定了绑定关系，直接生成 `node.textContent = value` 的代码
- Solid：运行时通过 Signal 追踪到 DOM 表达式级别，数据变了直接更新对应节点

### 面试怎么说

> "Virtual DOM 不是银弹，它的价值在于提供了一层抽象（跨平台、声明式）和批量更新能力。但它有创建和 diff 的开销。Svelte 和 Solid 证明了如果能在编译时或运行时精确追踪数据到 DOM 的映射关系，可以跳过 diff 直接更新，性能更好。不过 Virtual DOM 的生态优势（React Native、SSR）目前还是不可替代的。"

## 问题：这些框架的性能排名大概是什么样的？

基于 js-framework-benchmark 的大致排名（原始 DOM 操作性能）：

```
Solid ≈ Svelte > Vue 3 > React（未优化）> Angular（默认策略）
```

**但实际项目中**：
- 性能差异在大部分业务场景下感知不到
- 开发效率、生态成熟度、团队熟悉度比框架性能重要得多
- React/Vue 通过优化手段（memo、虚拟列表、编译优化）可以达到接近的性能

**面试怎么说**：
> "纯 benchmark 上 Solid 和 Svelte 最快，因为没有 Virtual DOM 开销。但实际项目中框架性能很少是瓶颈，更多是业务代码的问题——大列表没做虚拟滚动、不必要的重渲染、接口瀑布流这些。选框架主要看生态和团队，不是看 benchmark。"

---

## 总结：各框架一句话定位

| 框架 | 一句话 |
| --- | --- |
| **React** | 函数式 UI + Fiber 调度，生态最大，灵活度最高 |
| **Vue** | 响应式 + 模板编译，开发效率高，上手快 |
| **Svelte** | 编译时框架，无运行时开销，适合轻量应用 |
| **Solid** | 细粒度 Signal + JSX，性能极致，React 开发者友好 |
| **Angular** | 全功能企业框架，强约束，适合大团队 |
