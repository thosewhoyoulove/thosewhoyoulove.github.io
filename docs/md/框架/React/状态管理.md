# React 状态管理

## 面试定位

状态管理题考察的是状态作用域设计，而不是背库名。面试官常问 Redux、Zustand、Recoil 的区别，以及什么时候不该引入状态库。

## 面试回答

> React 状态管理要先按作用域拆分。组件内部交互状态用 `useState`；复杂局部状态用 `useReducer`；兄弟组件共享可以状态提升；主题、用户信息这类低频跨层级数据可以用 Context；只有当状态跨页面共享、更新路径复杂、需要持久化、调试追踪或多人协作维护时，才考虑 Redux、Zustand、Recoil 这类状态库。Redux 强调 action、reducer 和可预测数据流，适合复杂业务和强规范团队；Zustand 更轻量，用 store + selector 订阅，适合中小型全局状态；Recoil 以 atom / selector 建模，适合状态依赖图较复杂的场景，但生态和长期维护要谨慎评估。

一句话总结：

> 状态管理先看作用域，局部状态留在组件内，复杂跨模块共享状态才引入外部状态库。

## 核心原理

```text
组件内部 -> useState
复杂局部逻辑 -> useReducer
兄弟共享 -> 状态提升
低频全局读取 -> Context
复杂跨页面共享 -> Redux / Zustand / Recoil
```

## Redux / Zustand / Recoil 对比

| 方案 | 模型 | 优点 | 适用场景 |
| --- | --- | --- | --- |
| Redux | store + action + reducer | 可预测、生态成熟、调试追踪强 | 复杂业务、多人协作、强规范 |
| Zustand | store + hook + selector | 轻量、样板少、接入快 | 中小型全局状态、局部 store |
| Recoil | atom + selector | 状态依赖图表达自然 | 细粒度状态依赖、实验性较强场景 |
| Context | Provider + Consumer / Hook | 内置、简单 | 主题、语言、用户信息等低频状态 |

## 高频追问

### Context 能替代 Redux 吗？

简单低频共享状态可以。高频变化的大状态如果直接放 Context，可能导致大范围重渲染，也缺少 Redux 的中间件、调试、规范和生态能力。

### Redux 为什么强调不可变更新？

不可变更新让状态变化可追踪，也方便通过引用比较判断是否需要更新。Redux Toolkit 用 Immer 降低了不可变写法成本，但数据流仍然是 action -> reducer -> new state。

### Zustand 为什么轻量？

Zustand 不要求 action/reducer 模板，store 可以直接暴露更新方法，组件通过 selector 订阅自己关心的状态片段，接入成本低。
