# React 组件设计能力

## 面试定位

组件设计题考察的是工程经验。面试官不只看你会不会写组件，而是看你能否设计清晰边界、可复用 API、状态归属和扩展能力。

## 面试回答

> 我设计 React 组件会先区分业务组件和通用组件。通用组件关注 API 稳定、受控/非受控、可组合、样式扩展、可访问性和边界状态；业务组件关注业务流程清晰、状态放在合适层级、副作用隔离和可测试性。状态设计上，组件内部状态只放本地交互，跨组件共享再提升或放到 store。复用上优先用组合而不是过度继承，比如 children、render props、compound components 或自定义 Hooks。性能上不会一开始就加 memo，而是先确认是否有无效渲染，再用 React.memo、useMemo、useCallback 或状态下沉优化。

一句话总结：

> 好组件的核心是职责单一、状态归属清晰、API 可组合、边界状态完整、扩展成本低。

## 设计维度

| 维度 | 要点 |
| --- | --- |
| 职责边界 | 一个组件只解决一类问题 |
| 状态归属 | 谁拥有状态，谁负责更新 |
| API 设计 | props 命名稳定，避免暴露内部实现 |
| 可组合性 | 用 children、slot-like props、自定义 Hook 拆逻辑 |
| 边界状态 | loading、empty、error、disabled、readonly |
| 可访问性 | 键盘操作、语义标签、aria |
| 性能 | 状态下沉、避免无效渲染、必要时 memo |

## 高频追问

### 受控组件和非受控组件怎么选？

表单值需要被外部控制、校验或联动时用受控组件；只需要组件内部保存临时状态、外部最终读取时可以用非受控。通用组件常同时支持 `value` / `defaultValue`。

### 什么时候抽自定义 Hook？

当一段逻辑包含状态、副作用或事件处理，并且和 UI 展示可以分离、多个组件会复用时，可以抽成自定义 Hook。不要为了复用几行代码就过度抽象。

### 怎么避免组件过度设计？

先满足当前明确场景，把扩展点留在稳定边界上。不要预设大量未来不会用的 props；当重复出现第二、第三个真实场景时再抽象。
