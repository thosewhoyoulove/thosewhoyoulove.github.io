# CSS 样式兼容性

## 面试定位

CSS 兼容性题常用于考察实际项目经验。回答重点不是背浏览器支持表，而是讲清如何发现兼容问题、如何选择方案、如何做降级和自动化处理。

## 核心原理

CSS 兼容性问题来自不同浏览器、不同版本、不同渲染引擎对 CSS 标准的支持差异。

常见差异来源：

- 新特性支持不一致，例如 Grid、容器查询、新颜色函数。
- 老浏览器缺少 CSS 变量、Flex 完整支持。
- 移动端浏览器内核差异。
- 默认样式不同。
- 私有前缀差异，例如 `-webkit-`。

## 常见处理方式

### 查兼容性

使用 Can I Use、MDN、浏览器 DevTools 判断目标浏览器支持情况。

### 渐进增强

先提供基础可用方案，再对支持新特性的浏览器增强体验。

```css
.layout {
    display: block;
}

@supports (display: grid) {
    .layout {
        display: grid;
    }
}
```

### Autoprefixer

通过 PostCSS 自动添加必要前缀。

```javascript
// postcss.config.js
module.exports = {
    plugins: {
        autoprefixer: {},
    },
};
```

配合 browserslist 控制目标浏览器：

```json
{
  "browserslist": ["> 1%", "last 2 versions", "not dead"]
}
```

### Normalize / Reset

统一浏览器默认样式，减少不同浏览器初始表现差异。

### Polyfill 或降级方案

CSS 变量、Grid 等特性在旧浏览器可能需要降级，不一定都值得 polyfill。

## 具体场景

### Flexbox

现代浏览器支持良好，旧 IE 支持不完整。如果需要兼容 IE，要谨慎使用复杂 flex 行为。

### Grid

现代浏览器支持良好，IE 的旧版 Grid 语法差异大，通常不建议为 IE 强行写完整 Grid 兼容。

### CSS Variables

IE 不支持 CSS 变量。需要兼容时可使用预处理器变量或构建时替换。

### 移动端 1px 问题

高 DPR 屏幕上 CSS 1px 可能看起来偏粗。常见方案有 transform 缩放、伪元素边框、使用框架封装方案。

## 面试回答

可以这样答：

> CSS 兼容性问题主要来自不同浏览器对标准支持不一致。实际项目里我会先根据产品目标确定 browserslist，再用 Can I Use 或 MDN 判断特性支持。工程上通过 PostCSS + Autoprefixer 自动补前缀，用 normalize/reset 统一默认样式；对于新特性采用渐进增强，比如用 `@supports` 判断是否支持 Grid，不支持就降级成普通布局。不是所有兼容问题都要硬兼容，像 IE 对 Grid 和 CSS 变量支持差，就要根据用户占比和维护成本决定是否降级。

## 高频追问

### Autoprefixer 根据什么加前缀？

根据 browserslist 配置和 Can I Use 数据决定是否需要添加浏览器前缀。

### reset 和 normalize 区别？

reset 倾向于清空浏览器默认样式；normalize 倾向于保留合理默认值，同时修复跨浏览器差异。

### 如何处理旧浏览器不支持新 CSS 特性？

先判断目标用户是否需要支持；需要时提供降级样式、使用 `@supports`、构建时转换或换用更兼容的实现。
