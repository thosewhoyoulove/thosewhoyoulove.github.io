# JavaScript 中的编程范式

## 面试定位

编程范式不是最高频八股题，但常出现在项目深挖里：为什么 React 更偏声明式？为什么 Redux 强调不可变数据？Vue 的响应式和事件驱动有什么关系？面试回答要能把范式和实际代码组织方式联系起来。

## 核心原理

编程范式是组织代码和思考问题的方式。JavaScript 是多范式语言，既可以写命令式代码，也可以写函数式、面向对象、事件驱动和响应式代码。

前端开发里最常见的结论是：

- 写业务 UI 时，框架通常鼓励声明式。
- 写复杂数据转换时，常用函数式思想。
- 写组件、领域模型、SDK 时，常用封装和面向对象思想。
- 写浏览器交互、Node 服务、异步流程时，离不开事件驱动。
- 写 Vue、MobX、RxJS 这类数据变化自动更新的逻辑时，会用到响应式思想。

## 命令式编程

命令式编程关注“怎么做”，代码明确描述每一步操作。

```javascript
const nums = [1, 2, 3];
let sum = 0;

for (let i = 0; i < nums.length; i++) {
    sum += nums[i];
}

console.log(sum); // 6
```

优点是执行步骤清晰，适合底层逻辑、性能敏感代码。缺点是状态变化分散时，可读性和维护性会下降。

## 声明式编程

声明式编程关注“要什么结果”，把过程交给语言、框架或库处理。

```javascript
const nums = [1, 2, 3];
const sum = nums.reduce((total, num) => total + num, 0);

console.log(sum); // 6
```

React 和 Vue 模板都属于声明式 UI：开发者描述状态对应的 UI，框架负责把状态变化更新到 DOM。

```jsx
function User({ name }) {
    return <div>{name}</div>;
}
```

面试里可以说：声明式能降低 UI 和状态同步的心智负担，但底层仍然需要框架做命令式 DOM 更新。

## 函数式编程

函数式编程强调纯函数、不可变数据、高阶函数和函数组合。

### 纯函数

相同输入永远得到相同输出，并且不修改外部状态。

```javascript
function add(a, b) {
    return a + b;
}
```

非纯函数例子：

```javascript
let count = 0;

function addCount() {
    count++;
    return count;
}
```

### 不可变数据

不直接修改原数据，而是返回新数据。

```javascript
const user = { name: "omen", age: 18 };
const nextUser = { ...user, age: 19 };
```

这在 React 状态更新里非常重要，因为 React 依赖引用变化判断是否需要更新。

### 高阶函数

函数可以作为参数，也可以作为返回值。

```javascript
const nums = [1, 2, 3];
const doubled = nums.map((num) => num * 2);
```

## 面向对象编程

面向对象把数据和行为封装到对象中，通过对象之间的协作完成逻辑。

核心概念：

- 封装：隐藏内部状态，暴露稳定接口。
- 继承：复用父类能力。
- 多态：同一接口在不同对象上表现不同。
- 抽象：提取稳定模型，屏蔽实现细节。

```javascript
class RequestClient {
    constructor(baseURL) {
        this.baseURL = baseURL;
    }

    get(path) {
        return fetch(`${this.baseURL}${path}`);
    }
}

const client = new RequestClient("/api");
client.get("/user");
```

前端项目里，SDK、工具类、编辑器、图形系统、业务领域模型常会使用面向对象。

## 事件驱动编程

事件驱动强调“发生事件后触发对应逻辑”。浏览器和 Node.js 都大量使用事件驱动。

```javascript
button.addEventListener("click", () => {
    console.log("clicked");
});
```

异步请求、用户点击、键盘输入、WebSocket 消息，本质上都可以看成事件。

事件驱动的好处是解耦事件生产者和消费者。风险是事件链路复杂后，调用关系不如同步代码直观，需要做好命名、清理和边界管理。

## 响应式编程

响应式编程关注数据流和变化传播：当数据变化时，依赖它的逻辑自动更新。

Vue 的响应式系统就是典型例子。数据被读取时收集依赖，数据被修改时触发依赖更新。

```javascript
const state = reactive({ count: 0 });

effect(() => {
    console.log(state.count);
});

state.count++;
```

这类思想适合 UI 自动更新、表单联动、数据流处理。缺点是隐式依赖较多时，排查更新来源会变复杂。

## 面试回答

可以这样答：

> JavaScript 是多范式语言，可以写命令式、声明式、函数式、面向对象、事件驱动和响应式代码。命令式关注怎么一步步执行，声明式关注最终结果，React 和 Vue 的 UI 写法就更偏声明式；函数式强调纯函数、不可变数据和高阶函数，常见于数组处理、Redux、React 状态更新；面向对象强调封装、继承、多态，适合 SDK、复杂领域模型或图形系统；事件驱动是浏览器和 Node 的基础，用户交互、网络请求、消息通知都靠事件触发；响应式则强调数据变化自动传播，Vue 响应式系统就是典型例子。实际项目里不是只用一种范式，而是根据场景组合使用。

## 高频追问

### React 为什么说是声明式？

因为开发者描述的是“状态是什么时 UI 应该长什么样”，而不是手动写每一步 DOM 操作。状态变化后，React 负责计算差异并更新视图。

### 为什么 React 状态更新强调不可变？

不可变数据能让引用变化变得可追踪，方便 React 做浅比较和性能优化。如果直接修改原对象，组件可能无法正确感知变化。

### 函数式编程是不是不能有副作用？

不是完全不能有副作用，而是要控制副作用，把副作用集中到明确边界里。例如请求、日志、DOM 操作都是副作用，业务中要避免它们散落在纯数据计算逻辑里。

### 面向对象在前端是不是过时了？

没有。UI 层现在更偏声明式和函数式，但 SDK、图形编辑器、播放器、复杂表单引擎、领域模型仍然经常使用面向对象封装。
