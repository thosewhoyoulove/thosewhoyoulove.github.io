# Go 全栈面试速记

## 面试定位

JD 要求 **熟悉 react/nodejs/go**。前端候选人 **不需要假装精通 Go**，但要能说明：**Go 在团队里常做什么、和 Node 如何分工、基础并发模型、你如何协作联调**。

## 面试回答

可以这样答：

> 我主力栈是 React 和 Node，Go 更多作为团队协作的高性能服务端语言。Node 适合 BFF、SSR、脚本和 I/O 密集的接口聚合；Go 适合对延迟和并发要求更高的网关、核心 API、工具链和一些云原生组件，比如 Vite 预构建用的 esbuild 就是 Go 写的。我了解 Go 的 goroutine 和 channel 并发模型、静态编译部署简单、错误处理显式等特点，联调时关注接口契约、protobuf/JSON 一致性和 trace 链路。如果岗位需要我写 Go，我会从现有模块和测试入手，先保证小范围交付质量，而不是一上来改核心框架。

一句话总结：

> 前端全栈视角：React 做体验，Node 做聚合与迭代，Go 做高性能与稳定的核心服务；我精通前两者，Go 能协作、能读、能按规范扩展。

## 核心原理

### Go vs Node（面试用对比表）

| 维度 | Go | Node.js |
| --- | --- | --- |
| 运行时 | 编译型，单二进制 | V8 + 事件循环 |
| 并发 | goroutine + channel（CSP） | 单线程 JS + 异步 I/O |
| 典型场景 | 网关、RPC、高并发 API、CLI/工具 | BFF、SSR、实时 I/O、脚本 |
| 类型 | 静态强类型 | TS 可选 |
| CPU 密集 | 多 goroutine 可利用多核 | 需 Worker/子进程/外包 |
| 生态 | 云原生、微服务 | 前端同构、npm 生态 |

### Go 基础（够应付概念题）

**goroutine**：轻量协程，`go func()` 启动，由运行时调度。

**channel**：协程间通信，`ch <- v` / `<-ch`，常用带 buffer 或 `select` 多路复用。

**原则**：不要通过共享内存通信，而通过通信共享内存（面试可提一句）。

**错误处理**：`if err != nil`，无 try/catch。

**模块**：`go mod`，与 npm 类似依赖管理。

### 和前端协作

- 接口：REST / gRPC；统一错误码与 requestId。
- 鉴权：JWT 校验常在网关或 Go 中间件，前端只存 token 策略。
- 部署：Go 容器镜像小；Node 注意内存与单进程。
- 调试：用 traceId 串联浏览器 → BFF(Node) → Go 服务。

### 诚实边界（建议主动说）

- 手写复杂 Go 并发、内存优化、底层 GC 调优 → 非前端强项。
- 表达 **学习路径**：读现有 handler → 补单测 → 小需求上线。

## 高频追问

### 为什么 Vite 快和 Go 有关？

开发态依赖**预构建**用 esbuild（Go 实现）做 TS/JS 转换，比纯 JS 打包器快；生产打包仍是 Rollup。见 [Vite 为什么快](/md/工程化/Vite/为什么Vite快.md)。

### goroutine 和线程区别？

goroutine 更轻，调度在用户态；数量可远大于 OS 线程。适合大量并发 I/O 与任务。

### Node 和 Go 同时存在是否重复？

不重复：Node 贴近前端团队和迭代速度；Go 贴近性能与核心服务稳定性。BFF(Node) + 核心服务(Go) 是常见组合。
