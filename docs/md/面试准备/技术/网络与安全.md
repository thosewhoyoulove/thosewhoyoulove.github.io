# 网络与安全

## 面试定位

网络与安全是前端面试里的基础高频模块。它既能考察浏览器和 HTTP 基础，也能考察你在项目里是否有安全意识。

回答时不要只背定义，要把“风险、原理、防御、项目场景”连起来。

## 核心原理

前端常见网络问题主要围绕：

- HTTP 请求和响应。
- 缓存。
- 跨域。
- HTTPS。
- WebSocket。
- Cookie 和鉴权。

安全问题主要围绕：

- XSS：恶意脚本在页面执行。
- CSRF：利用用户登录态伪造请求。
- 敏感信息泄露。
- 权限和越权。
- 第三方依赖和脚本风险。

网络题关注通信链路是否理解，安全题关注风险边界是否清楚。

## HTTP 和 HTTPS

HTTP 是明文传输，容易被窃听和篡改。HTTPS 在 HTTP 和 TCP 之间加入 TLS，提供机密性、完整性和身份认证。

TLS 大致流程：

1. 客户端和服务端协商协议版本和加密套件。
2. 服务端返回证书。
3. 客户端校验证书是否可信。
4. 双方协商会话密钥。
5. 后续使用对称加密传输数据。

面试可以这样答：

```text
HTTPS 不是换了一个端口这么简单，它主要通过证书解决身份认证，通过密钥协商生成会话密钥，通过对称加密和完整性校验保证传输安全。
```

## HTTP 缓存

HTTP 缓存分为强缓存和协商缓存。

强缓存命中时，浏览器直接使用本地缓存，不向服务器发请求。常见字段是 `Cache-Control` 和 `Expires`。

协商缓存会向服务器确认资源是否变化。常见字段是 `ETag`、`If-None-Match`、`Last-Modified`、`If-Modified-Since`。

常见策略：

- 带 hash 的静态资源设置长缓存。
- HTML 不做长期强缓存。
- 接口缓存根据业务实时性决定。

## 跨域和 CORS

跨域是浏览器同源策略限制导致的。协议、域名、端口任意一个不同，都属于不同源。

CORS 的核心是服务端通过响应头告诉浏览器是否允许跨域访问。

如果请求是复杂请求，比如带自定义头、非简单方法、特殊 Content-Type，浏览器会先发送 OPTIONS 预检请求。

开发环境常用代理解决跨域，生产环境更推荐服务端正确配置 CORS、网关同域转发或 BFF。

## XSS

XSS 是攻击者把恶意脚本注入页面，并在用户浏览器中执行。

常见危害：

- 窃取 Cookie 或 Token。
- 篡改页面内容。
- 伪造用户操作。
- 劫持跳转。

防御方式：

- 对不可信输入做过滤。
- 输出到 HTML 时做编码。
- 避免直接使用 `innerHTML` 拼接不可信内容。
- 富文本使用白名单过滤。
- Cookie 设置 `HttpOnly`。
- 配置 CSP 限制脚本来源。

## CSRF

CSRF 是攻击者诱导已登录用户向目标网站发起非本意请求。

它利用的是浏览器会自动携带 Cookie。

防御方式：

- CSRF Token。
- Cookie 设置 `SameSite`。
- 校验 `Origin` 或 `Referer`。
- 关键操作二次确认。
- 不用 GET 做有副作用的操作。

## Cookie、localStorage 和 Token

Cookie 可以自动随请求发送，适合服务端会话，但要注意 `HttpOnly`、`Secure`、`SameSite`。

localStorage 使用方便，但容易被 XSS 读取，不适合存高敏感信息。

如果用 Token，要考虑过期时间、刷新机制、退出登录失效、泄露后的处理方式。

## 面试回答

网络和安全我一般会从链路和风险两个角度回答。

网络方面，HTTP 负责请求响应，HTTPS 通过 TLS 解决明文传输的窃听、篡改和身份认证问题。缓存分强缓存和协商缓存，静态资源一般用 hash 加长缓存，HTML 保持可更新。跨域是浏览器同源策略导致的，生产环境通常通过 CORS、网关同域或 BFF 解决。

安全方面，XSS 是恶意脚本在页面执行，重点防御是输入过滤、输出编码、CSP、HttpOnly Cookie 和避免直接渲染不可信 HTML。CSRF 是利用用户登录态伪造请求，重点防御是 SameSite、CSRF Token、Origin/Referer 校验，以及关键操作二次确认。

在项目里，我会尽量避免把敏感信息放到 localStorage，对富文本和第三方脚本保持谨慎，对接口权限也不会只依赖前端路由控制。

## 高频追问

### HTTPS 一定安全吗？

HTTPS 只能保证传输链路更安全，不能解决所有安全问题。

如果页面本身有 XSS，Token 仍然可能被窃取；如果服务端鉴权有漏洞，也可能发生越权。

### XSS 和 CSRF 最大区别是什么？

XSS 是脚本在用户页面里执行，攻击者能直接操作页面上下文。

CSRF 是诱导用户浏览器带着登录态发请求，攻击者不一定能读取响应。

### localStorage 能不能存 Token？

可以存，但有风险。

如果页面发生 XSS，localStorage 里的 Token 很容易被读取。更安全的方案通常是 HttpOnly Cookie 配合 SameSite、CSRF 防护和短有效期策略。

### OPTIONS 预检请求能不能避免？

可以通过使用简单请求减少预检，比如使用简单方法和简单请求头。

但不应该为了省预检牺牲接口语义和安全策略。预检本质是浏览器的安全确认机制。
