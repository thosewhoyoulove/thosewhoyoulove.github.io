# 网络与安全速记

## 面试定位

三年前端岗的网络与安全，**不是考你背 RFC**，而是考：**HTTP 链路是否清楚、线上问题能否定位、XSS/CSRF 能否结合项目讲防御**。

面试官通常从「输入 URL 发生了什么」「HTTP 和 HTTPS 区别」「跨域怎么解决」「XSS 和 CSRF 区别」切入，再追到缓存、预检、Cookie/Token、WebSocket。

**答题框架**：结论 → 机制 → 项目做法 → 边界（HTTPS 不能防 XSS 等）。

## 面试回答

### 总答：三年岗你怎么准备网络与安全？

> 我按 **链路 + 风险** 两条线准备。
>
> **链路**：URL → 缓存 → DNS → TCP → TLS → HTTP → 渲染；HTTPS 是握手用非对称/ECDHE 协商密钥、数据用对称加密。
>
> **风险**：XSS 是脚本在用户页面执行，防输入输出和 CSP；CSRF 是浏览器自动带 Cookie 发伪造请求，防 SameSite、CSRF Token、校验 Origin。
>
> 项目里静态资源带 hash 长缓存、HTML `no-cache`；跨域生产走网关同域或 BFF，开发用代理；Token 优先 HttpOnly Cookie，避免敏感信息进 localStorage。

一句话总结：

> 网络考链路与会话，安全考 XSS/CSRF 原理与防御；都要能落到项目配置。

---

### 题 1：跨域是什么？前端怎么解决？

> **跨域**是浏览器 **同源策略** 限制：协议、域名、端口任一不同即跨源，Ajax/fetch 默认读不到响应。
>
> **生产**常用：服务端配 **CORS**（`Access-Control-Allow-Origin` 等）；或 **网关/BFF 同域** 转发，浏览器只请求同域 API。
>
> **开发**常用 devServer **proxy**，避免临时改后端 CORS。
>
> 带 `Authorization`、`Content-Type: application/json` 的跨域请求会先 **OPTIONS 预检**，服务端要正确响应预检头。

---

### 题 2：XSS 和 CSRF 有什么区别？

> **XSS**：恶意 **脚本在 victim 页面里执行**，能读 DOM、偷 Cookie（非 HttpOnly）、发请求。防御：编码输出、CSP、HttpOnly、避免 `innerHTML` 拼不可信内容。
>
> **CSRF**：攻击者 **诱导已登录用户** 的浏览器向目标站 **自动带 Cookie** 发请求，攻击者通常 **读不到响应**。防御：SameSite、CSRF Token、校验 Origin/Referer、关键操作二次确认。
>
> 记忆：**XSS 偷的是页面上下文，CSRF 借的是登录态发请求。**

---

### 题 3：Token 放 Cookie 还是 localStorage？

> **localStorage**：实现简单，但 **XSS 可读**，适合低敏感、配合短 TTL 和 CSP 的场景。
>
> **HttpOnly Cookie**：JS 读不到，降低 XSS 偷 Token 风险；需配 **Secure、SameSite**，并防 CSRF（SameSite=Lax/Strict 或 CSRF Token）。
>
> 三年岗标准答法：**高敏感会话优先 HttpOnly Cookie + SameSite；若用 Bearer Token 放内存/Header，要严控 XSS 面。**

## 核心原理（提纲）

### HTTP 要点

- 无状态；会话靠 Cookie / Token / Session。
- 安全方法 vs 幂等：`GET` 幂等，`POST` 通常不幂等。
- 持久连接 `Keep-Alive`；HTTP/2 多路复用。

### 缓存要点

- 强缓存 → 协商缓存 → 200/304；详见 [浏览器缓存机制](/md/浏览器/浏览器的缓存机制.md)。

### CORS 要点

- 简单请求：直接发，服务端带允许头即可。
- 非简单请求：先 OPTIONS 预检 → 再真实请求。

### 安全要点

| 攻击 | 利用 | 核心防御 |
| --- | --- | --- |
| XSS | 不可信数据当代码/HTML 执行 | 编码、CSP、HttpOnly |
| CSRF | 自动携带 Cookie | SameSite、Token、Origin |

## 高频追问

### HTTPS 一定安全吗？

只保证 **传输链路**；页面 XSS、服务端漏洞、用户忽略证书警告仍不安全。

### 304 还会发请求吗？

会。强缓存命中才不发请求；304 是协商缓存，请求仍在，只是无 body。

### 预检能省掉吗？

简单请求可省；不要为了省预检把 JSON API 改成 form-urlencoded 牺牲语义。

### 前端能单独防 CSRF 吗？

要 **前后端配合**：SameSite、Token、Origin 校验主要在服务端生效，前端负责带 Token、不滥用 GET 写操作。
