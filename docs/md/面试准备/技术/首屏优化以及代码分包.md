# 首屏优化以及代码分包

## 面试定位

首屏优化和代码分包是性能优化里最容易结合项目深挖的问题。

面试官通常会追问：

1. 首屏慢怎么定位。
2. 代码分包解决了什么问题。
3. 分包过细有什么副作用。
4. Vite 或 Webpack 里具体怎么做。
5. 如何证明优化有效。

## 核心原理

首屏加载时间取决于“关键资源何时加载完成、何时解析执行完成、何时渲染出核心内容”。

代码分包的核心目标是减少首屏必须下载和执行的 JavaScript，把非首屏代码延后到用户需要时再加载。

但代码分包只解决一部分问题。首屏还可能受图片、字体、接口、CSS、服务端响应、第三方脚本影响。

## 首屏慢怎么定位

先看指标和链路：

- TTFB：服务端和网络是否慢。
- FCP：页面是否尽快出现内容。
- LCP：最大首屏内容是否过晚出现。
- JS 执行时间：主线程是否被阻塞。
- Network 瀑布图：资源是否过大或串行加载。

定位工具：

- Chrome DevTools Performance。
- Network 面板。
- Lighthouse。
- 构建产物分析工具。
- 线上性能监控。

## 代码分包策略

### 路由级分包

最常见的是按路由动态加载。

```js
const Home = () => import("@/views/Home.vue");
const About = () => import("@/views/About.vue");
```

这样用户访问首页时，不需要加载所有页面代码。

### 组件级懒加载

对首屏非关键组件，比如图表、弹窗、富文本编辑器、大型列表，可以使用异步组件。

```js
import { defineAsyncComponent } from "vue";

const Chart = defineAsyncComponent(() => import("@/components/Chart.vue"));
```

### 第三方库拆分

大型依赖可以单独拆 chunk，例如图表库、UI 库、工具库。

```js
export default {
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          vue: ["vue", "vue-router", "pinia"],
          echarts: ["echarts"],
          utils: ["lodash-es", "dayjs"],
        },
      },
    },
  },
};
```

这样做的好处是缓存更稳定，业务代码变更时不一定影响第三方依赖缓存。

### 按需引入

UI 库、图标库、图表库都要避免全量引入。

比如 Element Plus 可以配合自动导入插件，ECharts 只注册用到的图表类型和组件。

## 静态资源优化

首屏优化不能只看 JS。

图片优化：

- 使用 WebP/AVIF。
- 压缩图片。
- 响应式图片。
- 非首屏图片懒加载。
- 设置宽高避免 CLS。

压缩优化：

- gzip。
- Brotli。
- JS/CSS 压缩。

缓存优化：

- 静态资源文件名带 hash。
- 静态资源设置长缓存。
- HTML 不设置长期强缓存。
- CDN 加速公共资源。

## 分包的副作用

分包不是越细越好。

可能的问题：

- 请求数量增加。
- 产生瀑布式加载。
- 首次进入某些页面变慢。
- 公共依赖拆分不合理导致重复加载。
- 预加载策略不当浪费带宽。

所以分包后要看 Network 瀑布图和真实用户路径，而不是只看首页 bundle 变小。

## 面试回答

首屏优化我会先定位瓶颈，再决定是否做代码分包。

如果发现首屏 JS 包体积过大，我会先用构建分析工具看哪些模块占用最多，再按路由、非首屏组件和大型第三方依赖做分包。比如首页不需要的图表、编辑器、弹窗，都会延迟加载。对于 Vue 项目，可以使用动态 import 和 `defineAsyncComponent`；对于 Vite，可以用 Rollup 的 `manualChunks` 拆分稳定依赖。

但我不会只看包体积。分包后还要看请求数量、瀑布图和缓存命中情况。如果拆得太细，可能导致请求链路变长。最终我会用 Lighthouse、Performance、Network 和线上监控对比优化前后的 LCP、资源体积、JS 执行时间。

## 高频追问

### 首屏优化有哪些方向？

资源体积、请求链路、缓存、图片、JS 执行、接口并行、SSR/SSG、CDN、第三方脚本治理。

不要只回答懒加载，懒加载只是其中一类方案。

### manualChunks 怎么设计？

优先拆稳定的大型依赖，比如框架、UI 库、图表库、工具库。

业务代码则按路由或模块拆。拆完要检查是否重复打包和请求过多。

### CDN external 有什么风险？

优点是减少构建产物体积，利用 CDN 缓存。

风险是外部资源不可用会影响页面，需要考虑版本锁定、SRI、降级方案和网络环境。

### 如何证明分包有效？

看优化前后的首屏 JS 体积、请求瀑布、LCP、主线程执行时间和真实用户数据。

只看 bundle 变小不够，因为请求链路变长也可能让体验变差。
