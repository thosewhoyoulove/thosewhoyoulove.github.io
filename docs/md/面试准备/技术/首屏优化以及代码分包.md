# 项目实战：首屏优化与代码分包

## 面试定位

这篇不再重复完整性能原理，而是用于项目表达。首屏优化属于运行时性能，代码分包属于构建产物优化，面试时要把二者串成一个闭环：先定位首屏瓶颈，再判断分包是不是合适方案。

## 面试回答

> 首屏优化我会先定位瓶颈，而不是直接做分包。先用 Lighthouse、Performance、Network 和线上监控看 TTFB、LCP、JS 体积、图片体积、接口是否串行、主线程是否有长任务。如果瓶颈是首屏 JS 过大，我会用构建分析工具找出大模块，再做路由级动态 import、组件级懒加载、大型依赖按需引入和 vendor 拆分；如果瓶颈在图片、接口或服务端响应，就分别做图片压缩、缓存、接口并行、SSR/SSG 或 CDN。优化后要看首屏资源体积、请求瀑布图、LCP、JS 执行时间和真实用户数据，避免分包过细造成瀑布请求。

一句话总结：

> 首屏优化先定位，代码分包只解决首屏 JS 过大的问题，最终要用 LCP、资源体积和请求链路验证。

## 项目表达结构

```text
背景：某页面首屏慢，用户进入后白屏时间长。
定位：Network 发现主包大，Performance 发现 JS 执行阻塞 LCP。
方案：路由分包、图表组件懒加载、ECharts 按需引入、vendor 拆分、图片压缩和缓存。
验证：主包体积下降，LCP 改善，请求瀑布没有明显变长。
取舍：分包过细会增加请求数量，非首屏模块首次打开会有加载延迟。
```

## 高频追问

### 分包一定能提升首屏吗？

不一定。只有当首屏瓶颈是 JS 下载、解析或执行过重时，分包才明显有效。如果瓶颈是 TTFB、大图、字体或接口串行，分包收益有限。

### manualChunks 怎么设计？

优先拆稳定的大型依赖，比如框架、图表库、编辑器、工具库。业务代码按路由或模块拆，拆完要检查重复打包和请求数量。

### CDN external 有什么风险？

它能减少构建产物体积并利用 CDN 缓存，但会引入外部资源可用性、版本锁定、SRI、降级方案和网络环境差异等问题。

### 如何证明分包有效？

看优化前后的首屏 JS 体积、请求瀑布、LCP、主线程执行时间和真实用户数据。只看 bundle 变小不够。
