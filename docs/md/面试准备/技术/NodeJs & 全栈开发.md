# NodeJs & 全栈开发

## 面试定位

Node.js 和全栈开发问题主要考察你是否理解前后端协作边界，而不是要求你像后端一样精通所有数据库和架构细节。

前端候选人回答这类问题时，要重点说明：

1. Node.js 为什么适合 I/O 密集场景。
2. BFF 或服务端接口层解决了什么问题。
3. API、鉴权、日志、异常、数据库访问如何设计。
4. 哪些场景不适合直接用 Node.js 主线程处理。

## 核心原理

Node.js 的 JavaScript 执行线程是单线程，但 I/O 不是简单地在主线程里等待。

它通过事件循环和异步 I/O 模型处理大量并发连接。文件、网络、数据库等耗时操作交给系统能力或 libuv 线程池处理，完成后再把回调放回事件循环。

所以 Node.js 适合 I/O 密集型服务，比如接口聚合、网关、BFF、SSR、脚本工具、实时通信服务。

但 CPU 密集型任务会阻塞主线程，比如大规模计算、图片处理、复杂加密等。这类任务应该放到 Worker Thread、子进程、队列或专门服务里。

## Node.js 如何处理高并发

面试可以这样答：

```text
Node.js 的高并发不是靠给每个请求创建一个线程，而是靠事件驱动和异步非阻塞 I/O。
主线程负责执行 JavaScript 和调度回调，耗时 I/O 交给底层处理。
I/O 完成后再把回调放入事件循环，因此可以用较少线程处理大量连接。
```

要补充边界：如果请求里有大量同步计算，仍然会阻塞事件循环，导致所有请求响应变慢。

## Nest.js 常见设计

Nest.js 是一个偏工程化的 Node.js 框架，借鉴了 Angular 的模块化和装饰器风格。

常见装饰器：

- `@Module`：组织模块。
- `@Controller`：声明控制器。
- `@Get`、`@Post`：声明路由方法。
- `@Injectable`：声明可注入服务。
- `@UseGuards`：鉴权。
- `@UseInterceptors`：拦截请求响应。
- `@Body`、`@Query`、`@Param`：获取请求参数。

Nest.js 适合中大型服务，因为它能把路由、业务服务、鉴权、参数校验、异常过滤、日志这些能力分层管理。

## RESTful API 设计

好的 API 设计要做到语义清晰、稳定可演进。

常见规范：

- URL 用资源名词，比如 `/users`、`/orders`。
- HTTP 方法表达动作：GET 查询、POST 创建、PUT/PATCH 更新、DELETE 删除。
- 使用合适状态码。
- 统一错误结构。
- 支持分页、筛选、排序。
- 关键操作考虑幂等。
- API 文档用 OpenAPI 或类似工具维护。

前端视角还要关注字段命名一致、错误码可处理、接口是否支持取消和重试、是否有超时策略。

## JWT 鉴权

JWT 由 header、payload、signature 组成。服务端通过签名校验 token 是否可信。

优点是无状态，适合分布式服务扩展。

风险包括：

- 泄露后在有效期内可被使用。
- payload 只是编码，不是加密，不能放敏感信息。
- 单个 token 不容易服务端主动失效，除非使用黑名单、版本号或短有效期。
- 存在 XSS 时，localStorage 中的 token 容易被读取。

更稳妥的实践是 HTTPS、短有效期 access token、refresh token、HttpOnly Cookie 或严格的存储策略、服务端校验 `exp`、`iss`、`aud`。

## 数据库和 ORM

前端讲数据库优化时不用展开太深，但要能说明基本原则：

- 给高频查询字段加索引。
- 避免 N+1 查询。
- 分页查询避免一次返回过多数据。
- 只查询必要字段。
- 对热点数据使用缓存。
- 慢查询要有日志和监控。
- 事务边界要清晰。

ORM 如 Prisma、TypeORM 的价值是提高开发效率、类型提示和迁移管理。但也要避免在循环里逐条查库，复杂查询也不能完全依赖 ORM 自动生成。

## BFF 的价值

BFF 是 Backend For Frontend，面向前端体验定制接口层。

它常用于：

- 聚合多个后端接口。
- 裁剪前端不需要的字段。
- 做接口适配和协议转换。
- 统一鉴权、错误码、日志。
- 降低前端页面请求数量。

BFF 的代价是多了一层服务，需要考虑稳定性、监控、缓存和发布风险。

## 面试回答

我理解 Node.js 在前端项目里的价值主要是补齐前后端之间的工程边界。

Node.js 本身是事件驱动和异步非阻塞 I/O，适合做接口聚合、BFF、SSR、脚本工具和实时通信。它的主线程负责执行 JavaScript，I/O 操作交给底层处理，完成后再通过事件循环回调，所以可以支撑大量 I/O 并发。但如果做 CPU 密集计算，就会阻塞主线程，需要用 Worker、子进程或独立服务处理。

如果用 Nest.js，我会按模块拆分 Controller、Service、DTO、Guard、Interceptor 和 Exception Filter。API 设计上会关注 RESTful 语义、统一错误结构、鉴权、参数校验、日志和接口文档。鉴权如果使用 JWT，会注意短有效期、刷新机制、HttpOnly Cookie 或安全存储策略，避免 token 泄露后长期有效。

所以我不会把 Node.js 当成简单转发接口的工具，而是把它作为前端工程和后端服务之间的协作层。

## 高频追问

### Node.js 单线程为什么还能高并发？

因为单线程指 JavaScript 执行线程，I/O 操作可以异步交给底层处理。

大量请求不需要为每个连接创建线程，I/O 完成后再通过事件循环处理回调。

### Node.js 不适合什么场景？

不适合在主线程里处理大量 CPU 密集任务。

比如大文件同步计算、复杂图片处理、长时间加密计算，这些会阻塞事件循环，影响所有请求。

### JWT 最大风险是什么？

泄露后在有效期内可以被使用，而且 payload 不能存敏感信息。

实践上要配合短有效期、刷新机制、HTTPS、HttpOnly Cookie 或安全存储，以及服务端失效策略。

### BFF 会不会增加复杂度？

会。BFF 多了一层服务，需要监控、部署、缓存和容灾。

但如果前端页面需要聚合多个接口、适配多个业务系统，BFF 可以明显降低前端复杂度和请求成本。
