# Vue 兼容包的作用

## 面试定位

`@vue/compat` 是 Vue 2 迁移到 Vue 3 时的过渡工具。这个问题通常不是单独考 API，而是考察你是否理解大型项目渐进迁移的策略。

面试回答要讲清楚：

1. 它解决什么迁移问题。
2. 为什么不能长期依赖它。
3. 如何用它降低升级风险。
4. 迁移过程中要关注哪些破坏性变化。

## 核心原理

`@vue/compat` 是 Vue 3 的兼容构建版本。它在 Vue 3 运行时里提供一层兼容逻辑，让部分 Vue 2 API 和行为还能继续运行，并在控制台给出迁移警告。

它的目标不是让 Vue 2 代码永久跑在 Vue 3 上，而是帮助项目分阶段迁移。

可以这样理解：

```text
@vue/compat = Vue 3 运行时 + Vue 2 兼容层 + 迁移警告
```

## 它解决什么问题

大项目从 Vue 2 升到 Vue 3，通常不能一次性改完所有组件、插件和 API。

`@vue/compat` 可以让项目先切到 Vue 3 生态，再逐步处理不兼容代码。

它能缓解这些问题：

- 老生命周期兼容，比如 `beforeDestroy`、`destroyed`。
- 部分 Vue 2 行为兼容。
- Options API 和 Composition API 共存。
- 第三方 Vue 2 插件短期继续运行。
- 通过 warning 暴露需要迁移的代码。

## 使用方式

典型方式是把 Vue 指向兼容构建，或者使用兼容创建方式。

```js
import { createCompatApp } from "@vue/compat";
import App from "./App.vue";

const app = createCompatApp(App);
app.mount("#app");
```

也可以通过构建配置 alias 到 `@vue/compat`。

实际项目里更重要的是开启兼容警告，根据 warning 分模块处理。

## 迁移策略

比较稳妥的迁移路径：

1. 先升级构建工具和依赖，保证项目能跑起来。
2. 引入 `@vue/compat`，让旧代码先在兼容模式下运行。
3. 根据兼容警告处理破坏性变化。
4. 优先迁移公共组件、路由、状态管理和高频页面。
5. 替换不兼容第三方插件。
6. 新代码使用 Vue 3 写法。
7. 逐步关闭兼容配置，最终移除 `@vue/compat`。

不要在迁移过程中同时做大规模业务重构，否则风险很难控制。

## 常见迁移点

需要重点关注：

- 生命周期命名变化。
- 全局 API 变化。
- `v-model` 语法变化。
- `$listeners` 合并到 `$attrs`。
- 插槽语法变化。
- 过滤器 filters 移除。
- Vue Router 3 到 4 的变化。
- Vuex 到 Pinia 或 Vuex 4 的选择。
- 第三方组件库是否支持 Vue 3。

## 为什么不能长期依赖

`@vue/compat` 是迁移工具，不是长期运行方案。

长期依赖会带来问题：

- 兼容层增加运行时成本。
- 代码里长期混用 Vue 2 和 Vue 3 风格。
- 团队难以形成统一规范。
- 后续升级和依赖维护更复杂。
- 一些兼容行为可能和 Vue 3 最佳实践冲突。

所以迁移目标应该是逐步消除 warning，最终切换到标准 Vue 3。

## 面试回答

`@vue/compat` 是 Vue 2 升级 Vue 3 的兼容构建版本，它的作用是降低大项目一次性迁移的风险。

在真实项目里，Vue 2 项目通常有大量历史组件、插件和旧 API，不可能一次性全部改成 Vue 3。使用 `@vue/compat` 后，可以先让项目运行在 Vue 3 的兼容模式下，同时保留一部分 Vue 2 行为，并通过控制台 warning 告诉我们哪些地方需要迁移。

但我不会把它当成长期方案。它更像迁移脚手架，目标是先跑起来，再按模块消除兼容警告，优先处理公共组件、路由、状态管理和高频页面，最后移除兼容包，切到标准 Vue 3。

## 高频追问

### @vue/compat 是不是能让 Vue 2 代码无成本升级？

不是。

它只能降低迁移风险，不能消除迁移成本。破坏性 API、插件兼容、路由和状态管理仍然需要处理。

### 为什么要渐进迁移？

大项目一次性迁移风险太高，容易影响业务交付和线上稳定性。

渐进迁移可以先保证项目可运行，再逐步处理 warning 和不兼容模块。

### 迁移时最容易出问题的地方是什么？

第三方插件、全局 API、生命周期、`v-model`、`$listeners`、过滤器、路由和状态管理。

这些地方通常影响范围大，要优先排查。

### 什么时候可以移除兼容包？

当兼容警告基本清零，核心页面和公共组件都已按 Vue 3 标准迁移，第三方依赖也支持 Vue 3，并且测试和灰度验证通过后，就可以移除。
