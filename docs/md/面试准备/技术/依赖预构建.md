# 依赖预构建

## 面试定位

依赖预构建通常会出现在 Vite 面试题里。它考察你是否理解 Vite 开发环境为什么快，以及 `optimizeDeps` 解决了什么问题。

回答时要把它和生产构建区分开：依赖预构建主要服务于开发环境。

## 核心原理

Vite 开发环境基于浏览器原生 ESM。源码可以按需加载，但第三方依赖有两个问题：

1. 很多依赖发布的是 CommonJS 或 UMD，不是浏览器可直接运行的 ESM。
2. 有些依赖内部有大量细碎模块，如果浏览器逐个请求，会产生大量请求。

依赖预构建就是 Vite 启动时用 esbuild 把这些第三方依赖提前转换和打包成浏览器更容易加载的 ESM。

它主要解决：

- CommonJS 转 ESM。
- 合并细碎依赖，减少请求数量。
- 缓存稳定依赖，加快后续启动。

## optimizeDeps 的作用

`optimizeDeps` 用来控制依赖预构建。

常见配置：

```js
export default {
  optimizeDeps: {
    include: ["some-large-library"],
    exclude: ["some-ignored-library"],
  },
};
```

`include` 用于手动指定需要预构建的依赖。适合 Vite 没有自动扫描到，但运行时又会用到的依赖。

`exclude` 用于排除不希望预构建的依赖。适合某些依赖需要保留源码处理，或者预构建后反而出问题。

`force` 可以强制重新预构建，常用于依赖变化后缓存不一致。

## 为什么 Vite 需要预构建

可以这样理解：

```text
项目源码经常变化，所以 Vite 开发态按需编译源码。
第三方依赖相对稳定，所以 Vite 提前把依赖预构建并缓存起来。
```

这样既能保持源码热更新快，又能避免浏览器请求大量依赖文件。

## 什么时候需要手动配置

常见场景：

- 依赖是动态引入，Vite 静态扫描不到。
- 某些库启动时加载很慢。
- 某些 CommonJS 包转换有问题。
- monorepo 里本地包被当成源码或依赖处理不符合预期。
- 依赖更新后缓存异常，需要强制重新优化。

## 和生产构建的区别

开发环境：

- 使用 esbuild 预构建依赖。
- 重点是启动快、热更新快。
- 使用浏览器原生 ESM 加载模块。

生产环境：

- 通常由 Rollup 做完整打包。
- 重点是产物体积、Tree Shaking、压缩、分包和缓存。

所以依赖预构建不能等同于生产环境的代码分包。

## 面试回答

Vite 的依赖预构建主要是为了解决开发环境下第三方依赖加载慢和格式不兼容的问题。

因为 Vite 开发态基于浏览器原生 ESM，源码可以按需加载，但很多第三方包并不是标准 ESM，比如 CommonJS 包，浏览器不能直接执行。还有一些库内部模块非常碎，如果浏览器一个个请求，会导致请求数量很多。所以 Vite 会在启动时用 esbuild 把依赖提前转换成 ESM，并把细碎模块合并起来缓存。

`optimizeDeps.include` 可以手动指定需要预构建的依赖，`exclude` 可以排除某些依赖，`force` 可以强制重新预构建。实际项目里，如果遇到依赖加载慢、动态依赖扫描不到、monorepo 本地包处理异常，就可能需要调整这些配置。

## 高频追问

### 依赖预构建为什么用 esbuild？

因为 esbuild 使用 Go 编写，构建速度很快，适合开发环境快速转换依赖。

Vite 开发态追求启动和热更新速度，所以用 esbuild 做依赖预构建很合适。

### include 和 exclude 怎么选？

Vite 没有自动识别但运行时会用到的依赖，可以放到 `include`。

预构建后出问题，或者希望作为源码参与处理的依赖，可以放到 `exclude`。

### 依赖预构建会影响线上包体积吗？

通常不会直接影响生产产物。

它主要作用于开发环境。生产构建仍然要看 Rollup 的 Tree Shaking、分包、压缩和缓存策略。

### 为什么修改依赖后有时要重新预构建？

因为 Vite 会缓存预构建结果。

依赖版本或内容变化后，缓存可能不一致，可以删除缓存或使用 `optimizeDeps.force` 重新构建。
