# 前端性能优化速记

## 面试定位

性能优化题考察的是“定位 -> 方案 -> 验证 -> 取舍”的闭环。不要只背懒加载、压缩、CDN，要先说指标和瓶颈。

## 面试回答

> 我做性能优化会先看指标，不会直接堆优化手段。如果是首屏慢，我会看 TTFB、FCP、LCP、资源瀑布图、JS 包体积、图片体积、CSS 阻塞和接口是否串行；如果是交互卡顿，我会看主线程长任务、组件重复渲染、大列表、同步计算和第三方脚本。定位后再选择方案，比如首屏侧做资源压缩、缓存、关键资源 preload、图片优化、接口并行、SSR/SSG；构建侧做 Tree Shaking、代码分包、按需引入和长期缓存；交互侧做虚拟列表、减少无效渲染、Web Worker、防抖节流。最后用 Lighthouse、Performance、Network 和线上 RUM 对比 LCP、CLS、INP、资源体积和错误率。

一句话总结：

> 性能优化要从指标出发，找到瓶颈后再选方案，并用同一口径的数据证明有效。

## 指标速记

| 指标 | 关注点 | 常见瓶颈 |
| --- | --- | --- |
| TTFB | 首字节时间 | 服务端慢、网络慢、缓存未命中 |
| FCP | 首次内容出现 | HTML/CSS/字体/首批资源阻塞 |
| LCP | 首屏核心内容出现 | 大图、主包过大、接口阻塞 |
| CLS | 视觉稳定性 | 图片无宽高、广告/异步内容插入 |
| INP | 交互响应 | 长任务、重复渲染、同步计算 |

## 项目表达模板

```text
背景：页面首屏慢 / 交互卡顿 / 资源体积大
指标：LCP、INP、JS 体积、请求数量、主线程长任务
定位：Performance、Network、bundle analyzer、线上 RUM
方案：分包、图片优化、缓存、接口并行、虚拟列表、Worker
结果：优化前后同口径对比
取舍：分包过细、缓存更新、图片质量、SSR 成本
```

## 高频追问

### 首屏加载慢怎么排查？

先看 Network 和 Performance。Network 看 TTFB、资源大小、请求是否串行、缓存是否命中；Performance 看主线程任务、渲染时机、LCP 元素和长任务。

### 代码分包有什么副作用？

分包能减少首屏包体积，但会增加请求数量和调度成本。拆得太细可能形成瀑布请求，所以要结合真实访问路径、HTTP/2、preload 和缓存命中判断。

### 怎么证明性能优化有效？

用同一口径对比优化前后数据。本地看 Lighthouse、Performance、Network；线上看 RUM，包括 LCP、CLS、INP、资源加载耗时和错误率。
