# 项目表达模板

面试讲项目时，目标不是把功能流水账背出来，而是证明你能发现问题、设计方案、推进落地、评估结果。

## 2 分钟版本

可以按这段结构准备：

> 我在这个项目中主要负责【模块/方向】。项目的核心目标是【业务目标】，当时遇到的主要问题是【性能/复杂度/协作/稳定性问题】。我做的方案是【关键技术方案】，落地过程中重点解决了【难点 1】和【难点 2】。最终结果是【指标/收益】，比如【加载时间下降、错误率下降、交付效率提升、用户体验改善】。

## 5 分钟版本

### 1. 背景

- 业务是什么？
- 用户是谁？
- 为什么这个项目重要？
- 当时已有方案有什么问题？

### 2. 职责

- 你负责的是核心模块、公共能力、性能优化，还是工程化建设？
- 你是独立完成、主导设计，还是参与协作？
- 你和后端、产品、测试、设计如何配合？

### 3. 难点

优先讲能体现技术判断的问题：

- 状态复杂：多个页面、多个角色、多个异步来源。
- 性能压力：首屏慢、列表卡、图表多、包体积大。
- 稳定性要求：异常兜底、权限边界、回滚机制。
- 工程复杂：多人协作、重复代码、构建慢、发布风险。

### 4. 方案

讲方案时避免只罗列技术名词，要说明取舍：

- 为什么选这个方案？
- 对比过哪些方案？
- 最终方案的代价是什么？
- 如何保证可维护性？

### 5. 结果

尽量量化：

- 首屏时间从多少降到多少。
- 包体积减少多少。
- 错误率下降多少。
- 构建或发布耗时减少多少。
- 复用组件覆盖多少页面。

如果没有历史数据，也要说明验证方式，例如 Lighthouse、Performance 面板、埋点、日志、用户反馈、线上监控。

## 常见项目追问

### 你做的最有技术含量的点是什么？

回答重点：不要说“用了某个库”。要说你解决了什么复杂问题，以及为什么这个问题不好解。

### 这个方案有什么缺点？

回答重点：主动承认取舍。例如为了上线速度牺牲了一部分抽象完整性，后续通过模块边界、文档和测试降低维护成本。

### 如果让你重新设计，你会怎么做？

回答重点：体现复盘能力。可以从架构分层、状态管理、监控告警、组件复用、测试覆盖、性能预算里选 2 到 3 个点回答。

### 你怎么证明优化有效？

回答重点：用指标闭环。优化前采样，优化后对比，再持续监控。不要只说“感觉快了”。

## 可复用句式

- “这个问题本质上不是某个 API 的使用问题，而是数据流和状态边界没有收敛。”
- “我当时没有直接重写，而是先把风险最高的链路拆出来做灰度验证。”
- “这个方案的收益是交付速度快，代价是抽象还不够完整，所以我补了文档和边界约束。”
- “我用线上指标验证，不只看本地开发环境的表现。”
