# 项目表达模板

## 面试定位

项目表达是面试里最能拉开差距的部分。技术八股能证明你知道概念，项目表达要证明你能发现问题、做技术判断、推进落地，并且能用结果闭环。

面试官通常不是想听功能流水账，而是在判断三件事：

1. 你是不是核心参与者。
2. 你有没有解决过真实复杂问题。
3. 你能不能把方案、取舍和结果讲清楚。

## 核心原理

项目回答要围绕“问题 - 方案 - 结果 - 复盘”展开。

一个好的项目回答应该同时包含：

- 业务背景：为什么要做。
- 你的职责：你具体负责什么。
- 技术难点：难在哪里。
- 方案设计：为什么这样做。
- 落地过程：怎么推进、怎么规避风险。
- 结果指标：优化了什么。
- 复盘思考：如果重做会怎么改。

最忌讳的是只说“我做了页面、调了接口、用了某个库”。这只能说明你参与了开发，不能说明你有判断力。

## 2 分钟版本

适合面试官让你“简单介绍一下项目”。

```text
我在这个项目里主要负责【模块/方向】。

这个项目的核心目标是【业务目标】，当时主要问题是【性能/复杂度/稳定性/协作问题】。

我做的方案是【关键方案】，重点解决了【难点 1】和【难点 2】。

落地时我通过【灰度/监控/测试/文档/评审】控制风险。

最终结果是【量化收益】，比如首屏时间下降、错误率下降、复用率提升、交付效率提升。
```

## 5 分钟版本

### 背景

先讲业务背景，不要一上来讲技术栈。

可以这样说：

```text
这个项目面向的是【用户/业务场景】，目标是解决【业务问题】。
在我接手时，系统主要有三个问题：【问题 1】、【问题 2】、【问题 3】。
这些问题带来的影响是【用户体验/研发效率/稳定性/性能】。
```

### 职责

职责要具体，避免泛泛说“参与开发”。

更好的表达是：

- 我负责核心模块设计。
- 我主导了性能优化方案。
- 我抽象了公共组件和业务模型。
- 我推动了接口规范和联调流程。
- 我负责从方案评审到上线验证的闭环。

### 难点

优先讲能体现技术判断的问题：

- 状态复杂：多个页面、多个角色、多个异步来源。
- 性能压力：首屏慢、列表卡、图表多、包体积大。
- 稳定性要求：异常兜底、权限边界、回滚机制。
- 工程复杂：多人协作、重复代码、构建慢、发布风险。

### 方案

讲方案时要补充取舍：

```text
我当时对比了【方案 A】和【方案 B】。
方案 A 的优点是【收益】，缺点是【成本】。
方案 B 的优点是【收益】，但在当前阶段会带来【风险】。
最后我选择【最终方案】，原因是它更符合当时的【工期/稳定性/维护成本/团队能力】。
```

### 结果

结果尽量量化。

- 首屏时间从多少降到多少。
- 包体积减少多少。
- 错误率下降多少。
- 构建或发布耗时减少多少。
- 复用组件覆盖多少页面。
- 需求交付周期缩短多少。

如果没有历史数据，也要说明验证方式，例如 Lighthouse、Performance 面板、埋点、日志、用户反馈、线上监控。

## 面试回答

我讲项目时一般不会从技术栈开始，而是先讲项目解决了什么业务问题，再讲我负责的边界。

比如我会先说明这个项目的目标、当时遇到的主要问题，然后聚焦到我负责的模块。接着讲技术难点，比如状态复杂、性能瓶颈、多人协作成本高，或者上线风险比较大。然后我会讲我为什么选择某个方案，有没有对比过其他方案，最终怎么落地，怎么通过测试、灰度、监控或回滚方案控制风险。

最后一定会补结果。如果有数据，就讲优化前后对比；如果没有数据，也会讲验证方式，比如通过 Performance、错误监控、埋点或用户反馈证明方案有效。

我觉得项目表达的重点不是说“我用了什么技术”，而是说清楚“我解决了什么问题，为什么这么做，结果怎么样”。

## 高频追问

### 你做的最有技术含量的点是什么？

不要回答“用了某个库”。应该回答这个库或方案解决了什么复杂问题。

可以按这个结构：

```text
最有技术含量的点是【问题】。
它难在【复杂度来源】。
我做了【方案】，核心是【关键设计】。
最后通过【指标/结果】证明有效。
```

### 这个方案有什么缺点？

要主动承认取舍。

例如：

```text
这个方案的优点是能快速落地，风险比较可控。
缺点是抽象还不够完整，如果后续业务继续扩展，需要把配置、状态和渲染逻辑再拆得更清楚。
当时我通过文档、边界约束和关键路径测试降低维护成本。
```

### 如果让你重新设计，你会怎么做？

回答要体现复盘能力。

可以从架构分层、状态管理、监控告警、组件复用、测试覆盖、性能预算里选 2 到 3 个点。

### 你怎么证明优化有效？

用指标闭环，不要只说“感觉快了”。

标准回答：

```text
我会先记录优化前指标，再针对瓶颈做改动，优化后用同样口径复测。
上线后继续通过监控看真实用户数据，避免只看本地环境。
```

## 可复用句式

- “这个问题本质上不是某个 API 的使用问题，而是数据流和状态边界没有收敛。”
- “我当时没有直接重写，而是先把风险最高的链路拆出来做灰度验证。”
- “这个方案的收益是交付速度快，代价是抽象还不够完整，所以我补了文档和边界约束。”
- “我用线上指标验证，不只看本地开发环境的表现。”
