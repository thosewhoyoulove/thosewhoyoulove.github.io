## 问题：useMemo 和 memo 的区别是什么？

一句话：`useMemo` 缓存的是**计算结果**，`React.memo` 缓存的是**组件渲染结果**。

`useMemo` 是 Hook，用在组件内部：

```tsx
const filteredList = useMemo(() => {
  return list.filter(item => item.name.includes(keyword))
}, [list, keyword])
```

它适合缓存比较重的计算，依赖不变时直接复用上一次结果，避免每次渲染都重新计算。

`React.memo` 是高阶组件，用在组件外部：

```tsx
const UserCard = React.memo(function UserCard(props: UserCardProps) {
  return <div>{props.name}</div>
})
```

它适合优化子组件重渲染。父组件重新渲染时，如果传给子组件的 props 浅比较没有变化，React 可以跳过这个子组件的重新渲染。

面试里可以这样总结：

> `useMemo` 解决的是组件内部“某个值要不要重新算”的问题，`React.memo` 解决的是“子组件要不要重新渲染”的问题。它们都不是默认必加的优化手段，只有计算成本高、组件渲染重，或者引用变化导致明显无效渲染时才值得使用。

### 常见追问：为什么 memo 之后子组件还是会渲染？

因为 `React.memo` 默认是浅比较 props。如果父组件每次都传新的对象或函数引用，浅比较就会认为 props 变化了。

```tsx
<UserCard options={{ showPhone: true }} onClick={() => open(user.id)} />
```

这里的 `options` 和 `onClick` 每次渲染都是新引用，所以 `memo` 很难生效。可以配合 `useMemo` 和 `useCallback` 稳定引用，但也要看收益，不要为了优化把代码写复杂。

## 问题：为什么不在 useMemo 里面写副作用，而是在 useEffect 里面写？

结论：`useMemo` 应该保持纯计算，`useEffect` 才是 React 专门用来处理副作用的地方。

`useMemo` 的语义是“根据依赖缓存一个计算结果”。它运行在渲染过程中，React 可能因为并发渲染、严格模式、重新调度等原因重新执行这段计算逻辑。如果在里面写请求、订阅、埋点、修改外部变量这类副作用，可能出现重复执行、执行时机不可控、清理困难的问题。

错误示例：

```tsx
const data = useMemo(() => {
  fetchUser(userId) // 不应该写在这里
  return formatUser(user)
}, [userId, user])
```

正确做法：

```tsx
useEffect(() => {
  let canceled = false

  fetchUser(userId).then(res => {
    if (!canceled) {
      setUser(res)
    }
  })

  return () => {
    canceled = true
  }
}, [userId])

const userViewModel = useMemo(() => {
  return formatUser(user)
}, [user])
```

面试回答可以这样说：

> `useMemo` 适合做同步、可重复执行、没有外部影响的派生计算，比如列表过滤、格式化数据、生成配置。`useEffect` 适合处理请求、订阅、定时器、DOM 操作、日志上报这些副作用，因为它在 commit 之后执行，并且可以通过返回清理函数处理取消订阅、清除定时器等逻辑。

## 问题：react-redux 的作用是什么？

`react-redux` 的作用是把 Redux store 和 React 组件连接起来。

Redux 本身只负责状态容器：

- `store` 保存全局状态。
- `dispatch` 触发 action。
- `reducer` 根据 action 生成新状态。
- `subscribe` 监听状态变化。

但 React 组件需要知道：

1. 怎么拿到 store。
2. 怎么订阅状态变化。
3. store 变化后怎么触发组件更新。
4. 怎么避免无关组件跟着重渲染。

这些就是 `react-redux` 解决的问题。

现代写法通常是：

```tsx
<Provider store={store}>
  <App />
</Provider>
```

组件里使用：

```tsx
const user = useSelector(state => state.user)
const dispatch = useDispatch()
```

面试里可以这样总结：

> Redux 是状态管理核心，react-redux 是 React 绑定层。它通过 Provider 把 store 放进 Context，再通过 useSelector 订阅 store 的变化，并在 selector 结果变化时触发组件更新。

## 问题：怎么实现一些 react-redux 的基础功能？

可以从三个核心 API 讲：`Provider`、`useDispatch`、`useSelector`。

### 1. 实现 Provider

`Provider` 本质是用 React Context 把 store 传下去。

```tsx
import React, { createContext, useContext, useEffect, useReducer } from 'react'

type Action = { type: string; payload?: unknown }
type Store<S> = {
  getState: () => S
  dispatch: (action: Action) => void
  subscribe: (listener: () => void) => () => void
}

const ReduxContext = createContext<Store<any> | null>(null)

export function Provider<S>({
  store,
  children,
}: {
  store: Store<S>
  children: React.ReactNode
}) {
  return (
    <ReduxContext.Provider value={store}>
      {children}
    </ReduxContext.Provider>
  )
}
```

### 2. 实现 useDispatch

`useDispatch` 只是从 Context 里取出 store.dispatch。

```tsx
export function useDispatch() {
  const store = useContext(ReduxContext)

  if (!store) {
    throw new Error('useDispatch must be used inside Provider')
  }

  return store.dispatch
}
```

### 3. 实现 useSelector

`useSelector` 要做三件事：

1. 首次渲染时读取 selector 结果。
2. 订阅 store 变化。
3. store 更新后比较新旧 selector 结果，变化了才触发组件更新。

```tsx
export function useSelector<S, T>(
  selector: (state: S) => T,
  equalityFn: (a: T, b: T) => boolean = Object.is,
) {
  const store = useContext(ReduxContext) as Store<S> | null

  if (!store) {
    throw new Error('useSelector must be used inside Provider')
  }

  const [, forceUpdate] = useReducer(x => x + 1, 0)
  const selectedRef = React.useRef(selector(store.getState()))

  useEffect(() => {
    return store.subscribe(() => {
      const nextSelected = selector(store.getState())

      if (!equalityFn(selectedRef.current, nextSelected)) {
        selectedRef.current = nextSelected
        forceUpdate()
      }
    })
  }, [store, selector, equalityFn])

  return selectedRef.current
}
```

真实的 `react-redux` 会更复杂，比如要处理并发渲染一致性、嵌套订阅、批量更新、selector 稳定性等问题。面试里写到这个程度，重点是证明你理解它的基本机制。

## 问题：react-redux 和 useContext + useReducer 的区别是什么？

`useContext + useReducer` 可以做一个轻量状态管理方案，但它不等于完整的 Redux 生态。

### 相同点

它们都能解决跨组件状态共享：

```tsx
const StateContext = createContext(null)
const DispatchContext = createContext(null)

function reducer(state, action) {
  switch (action.type) {
    case 'setUser':
      return { ...state, user: action.payload }
    default:
      return state
  }
}
```

这种方式适合中小型页面、局部业务模块、弹窗流程、表单向导等场景。

### 不同点

`useContext + useReducer` 的问题是 Context value 一变，消费这个 Context 的组件都会有重新渲染风险。如果状态很大、消费组件很多，就容易产生性能问题。你可以拆多个 Context 或自己写 selector，但复杂度会逐渐接近状态管理库。

`react-redux` 的优势是：

- 有成熟的订阅模型。
- 组件可以通过 selector 精确订阅需要的状态片段。
- 有 Redux DevTools、middleware、异步流、规范化状态等生态。
- 更适合大型应用里的全局状态和跨模块协作。

面试回答：

> 小范围状态我会优先用 `useReducer + Context`，比如某个复杂组件内部的状态机。跨页面、跨业务模块、需要调试工具和中间件的全局状态，我会用 Redux 或 react-redux。核心区别不是能不能存状态，而是订阅粒度、调试能力、生态能力和团队协作规范。

## 问题：Cursor 的 Debug 模式对比普通 Agent 模式，有什么区别？

根据 Cursor 官方文档和 2.2 更新说明，Agent 模式更适合复杂功能开发、重构和多文件编辑；Debug Mode 更像一个围绕 bug 复现、运行时日志和人工验证组织起来的调试循环。

普通 Agent 模式通常是：

```text
理解需求 -> 搜索代码 -> 修改代码 -> 运行检查 -> 修复报错
```

Debug 模式更强调：

```text
描述 bug -> 提出多个假设 -> 添加运行时日志 -> 让用户复现 -> 根据日志定位根因 -> 做定点修复 -> 再验证
```

两者区别可以这样答：

| 对比项 | Agent 模式 | Debug 模式 |
| --- | --- | --- |
| 目标 | 完成开发任务、重构、改功能 | 定位并修复 bug |
| 信息来源 | 主要依赖代码上下文、静态分析、命令输出 | 更依赖运行时日志、复现路径、验证结果 |
| 工作方式 | 倾向直接提出方案并修改 | 先建立假设，再用日志验证 |
| 人机协作 | 用户描述目标，Agent 自主执行 | 用户需要配合复现和确认修复 |
| 适合场景 | 新功能、多文件改造、工程任务 | 偶发问题、状态错乱、运行时行为和代码不一致的问题 |

面试回答可以这样说：

> 普通 Agent 更像开发助手，Debug 模式更像调试助手。Debug 的关键不是让模型凭代码猜问题，而是让它主动补运行时观测点，通过复现和日志验证假设，再做更小范围的修复。

## 问题：如果让你写一个 skill 实现 Debug 功能，你会怎么实现？

我会把它设计成一个标准化调试工作流，而不是只写一句“帮我修 bug”的提示词。

### Skill 目标

让 AI 在处理 bug 时强制经过几个阶段：

1. 收集复现信息。
2. 建立多个原因假设。
3. 找到最小观测点。
4. 插入临时日志或断言。
5. 让用户复现或跑测试。
6. 根据证据定位根因。
7. 做小范围修复。
8. 清理临时日志并补充验证。

### Skill 结构

```text
debug-skill/
  ├── SKILL.md
  ├── scripts/
  │   ├── collect-errors.js
  │   └── strip-debug-logs.js
  └── templates/
      └── bug-report.md
```

### SKILL.md 核心规则

```md
# Debug Skill

## When to use

Use this skill when the task is to fix a reproducible bug, runtime error,
state inconsistency, UI behavior mismatch, or flaky test.

## Workflow

1. Restate the bug with expected behavior and actual behavior.
2. Identify the entry point, triggering action, and affected modules.
3. List at least three hypotheses ordered by likelihood.
4. Add the smallest possible runtime logs or assertions to verify hypotheses.
5. Ask the user to reproduce only if the issue cannot be reproduced by command.
6. Use runtime evidence to choose the root cause.
7. Make the smallest fix that addresses the root cause.
8. Remove temporary logs unless they are intentionally kept as diagnostics.
9. Run targeted tests, typecheck, or reproduction command.
10. Summarize root cause, fix, and verification.
```

### 我会特别加的约束

- 不允许一上来大改。
- 不允许只凭猜测修。
- 每个假设要有验证方法。
- 临时日志要有统一前缀，比如 `[debug-skill]`，方便清理。
- 修复后必须删除临时日志或解释为什么保留。
- 如果不能本地复现，要明确让用户提供操作路径、截图、控制台日志或网络请求。

面试回答：

> 我理解 Debug skill 的核心不是“更会写代码”，而是把调试流程制度化。先让模型从猜测切换到证据驱动，通过假设、日志、复现、验证形成闭环，这样才能减少越修越乱的问题。

## 问题：写一下这个函数的类型

题目：

```ts
const merge = (a, b) => ({ ...a, ...b })
```

基础版本可以写成：

```ts
function merge<A extends object, B extends object>(a: A, b: B): A & B {
  return { ...a, ...b }
}
```

如果想保留箭头函数：

```ts
const merge = <A extends object, B extends object>(a: A, b: B): A & B => ({
  ...a,
  ...b,
})
```

但这里有一个细节：运行时 `{ ...a, ...b }` 如果有同名属性，`b` 会覆盖 `a`。单纯写 `A & B` 在类型上不能准确表达“后者覆盖前者”，尤其是同名字段类型不一致时。

更准确的类型可以写：

```ts
type Merge<A extends object, B extends object> = Omit<A, keyof B> & B

const merge = <A extends object, B extends object>(a: A, b: B): Merge<A, B> => ({
  ...a,
  ...b,
})
```

示例：

```ts
const result = merge(
  { id: 1, name: 'Tom' },
  { name: 'Jerry', age: 18 },
)

// result 类型接近：
// {
//   id: number
//   name: string
//   age: number
// }
```

面试回答可以补一句：

> 如果只是考泛型和交叉类型，`A & B` 可以；如果考对象展开的真实语义，`Omit<A, keyof B> & B` 更准确，因为后面的对象会覆盖前面的同名属性。

## 问题：说说你平时怎么使用 AI 工具？

我会把 AI 当成研发辅助工具，而不是直接替代开发。

主要用在几个环节：

1. 需求分析：拆功能点、边界场景、异常状态、接口依赖。
2. 方案设计：让 AI 帮我列几种方案和风险，再结合项目现有架构取舍。
3. 编码提效：生成类型定义、工具函数、表单校验、mock 数据和单测初稿。
4. Debug：提供报错堆栈、复现路径、相关代码，让它帮我列排查假设。
5. Code Review：检查空值、异步竞态、闭包、性能和测试遗漏。
6. 文档沉淀：生成技术方案、接口说明、复盘和组件 README 初稿。

但我不会直接复制 AI 生成的代码上线。我的验证流程是：

```text
AI 输出候选方案
  -> 人工判断是否符合项目约束
  -> 修改成项目风格
  -> 类型检查 / lint / 单测
  -> 本地验证 / 联调
  -> Code Review
```

也会注意数据安全，不把用户隐私、公司密钥、内部接口、核心业务策略直接发给外部 AI。

一句话：

> 我使用 AI 的原则是：让 AI 提效，让人做判断。AI 负责提供候选方案，开发者负责验证、取舍和交付质量。

## 问题：你项目里说“结合 RxJS 构建核心状态观察体系”，为什么要用 RxJS？为什么不用 async/await？

这个问题要先澄清：我的项目里不是简单把所有异步都换成 RxJS，而是借鉴响应式编程思想来处理**持续变化的状态流**。

`async/await` 适合处理一次性的异步流程：

```ts
const user = await fetchUser()
const config = await fetchConfig(user.id)
```

它表达的是“等一个结果回来，然后继续往下走”。

但云呼 SDK 这类场景里，很多东西不是一次性结果，而是持续变化的事件和状态：

- SIP 注册状态变化。
- 通话状态变化。
- 麦克风权限变化。
- 多标签页消息同步。
- 批量外呼队列进度。
- WebRTC stats 质量数据。
- 用户挂断、来电、接听、失败等事件。

这些更像一条持续的数据流：

```text
事件输入 -> 状态转换 -> 广播同步 -> UI 更新 -> 监控上报
```

如果全部用 `async/await`，很容易变成很多分散的回调和状态变量，取消、合并、节流、重试、订阅清理都会比较麻烦。

RxJS 或响应式思想的价值在于：

- 把事件、异步结果和状态变化统一成流。
- 可以组合多个事件源。
- 可以对流做过滤、去重、节流、重试。
- 订阅和取消订阅有明确生命周期。
- UI 层只关心状态变化，不关心底层事件来自哪里。

面试回答：

> `async/await` 适合一次性请求，RxJS 更适合持续事件流和复杂状态编排。云呼 SDK 的核心问题不是“等一个接口返回”，而是多事件源下的状态同步，比如通话状态、Worker 消息、批量任务、WebRTC 质量数据都在不断变化，所以用响应式观察体系更容易把状态变化收敛到统一通道里。

也要补充取舍：

> RxJS 学习成本高，不适合所有业务。简单页面请求我不会引入它；只有事件源多、状态持续变化、订阅关系复杂时，才值得用响应式流。

## 问题：你的企业级高级筛选系统具体做了什么？

这个项目不是一个普通筛选表单，而是一个面向企业获客场景的复杂检索系统。

用户会组合多个企业维度条件，比如行业、地区、规模、经营状态、关键词命中等。搜索之后，还要支持表格视图和卡片视图两种结果展示，并处理分页、排序、条件持久化、口令分享、详情页返回恢复这些问题。

我主要做了几个核心点：

### 1. 抽 EnterpriseFilter 编排层

我把稳定的数据流沉到 `EnterpriseFilter`：

- 条件选择。
- 条件序列化。
- 条件持久化。
- 请求参数组装。
- 搜索触发。
- 分页和排序。
- 视图切换。
- 缓存恢复。

具体筛选维度通过配置或子组件接入，框架本身不强耦合某个业务维度。

### 2. 设计双视图数据隔离

表格视图和卡片视图接口不同、字段不同、分页不同，所以我把两套状态隔离：

| 状态 | 表格视图 | 卡片视图 |
| --- | --- | --- |
| 页码 | `currentPage` | `cardCurrentPage` |
| 每页条数 | `pageSize` | `cardPageSize` |
| loading | `loading` | `cardLoading` |
| 搜索标识 | `searchKey` | `cardSearchKey` |

对外再提供统一方法：

- `getActivePage()`
- `getActivePageSize()`

调用方不用关心当前是表格还是卡片。

### 3. 做 searchKey 去重和缓存恢复

我会把筛选条件、分页、排序、模式、视图类型序列化成一个 `searchKey`。如果用户重复点击搜索，或者切换视图时目标视图已经有同条件数据，就不重复请求。

同时做了 `RuntimeCache`，用户从详情页返回时，可以恢复之前的条件、列表、页码和滚动位置。

### 4. 支持条件持久化和口令分享

条件选择后会结构化存储，并支持生成口令。销售主管配置好一套筛选条件后，可以复制给团队成员，别人粘贴后能还原同一套条件。

1 分钟回答：

> 我这个高级筛选项目主要解决复杂企业检索的问题。核心不是表单控件，而是条件编排、双视图数据隔离、请求去重、状态恢复和扩展性。我抽了 EnterpriseFilter 作为编排层，把条件选择、序列化、持久化、搜索、分页、排序、视图切换统一管理；表格和卡片视图因为接口和字段不同，所以设计了独立数据通道；性能上通过 loading 拦截、searchKey 指纹和 RuntimeCache 减少重复请求，并支持条件口令分享和详情页返回恢复。

## 问题：如果让你重构高级筛选系统，你会怎么重构？

我不会推翻原来的设计，因为原来的核心方向是对的：公共上下文上提，视图私有状态隔离。重构会围绕“边界更清晰、类型更强、可测试性更好”去做。

### 1. 把视图抽成 adapter

现在表格和卡片虽然状态隔离，但父组件还是知道不少细节。后续可以抽成视图适配器：

```ts
interface ResultViewAdapter<Params, Response, ViewModel> {
  viewType: 'table' | 'card'
  buildParams: (context: SearchContext) => Params
  request: (params: Params) => Promise<Response>
  normalize: (response: Response) => ViewModel
  getCacheKey: (context: SearchContext) => string
}
```

这样新增地图视图、看板视图时，不改主流程，只新增 adapter。

### 2. 把筛选条件做成 Schema 驱动

筛选维度可以统一成配置：

```ts
interface FilterFieldSchema {
  key: string
  label: string
  type: 'text' | 'select' | 'dateRange' | 'numberRange' | 'remoteSelect'
  operators: string[]
  defaultValue?: unknown
  serialize: (value: unknown) => unknown
  deserialize: (value: unknown) => unknown
}
```

这样条件渲染、校验、序列化、回显都能统一。

### 3. 引入更明确的状态机

搜索流程其实有状态流转：

```text
idle -> editing -> searching -> success / error
```

双视图还有自己的状态。可以用 reducer 或状态机把状态转移收敛，避免散落在多个方法里。

### 4. 补齐测试和监控

重点补：

- 条件序列化和反序列化单测。
- searchKey 生成规则单测。
- 表格 / 卡片切换集成测试。
- 返回恢复和口令导入测试。
- 请求耗时、重复请求率、接口错误率监控。

面试回答：

> 我会优先做渐进式重构，不会大拆大改。第一步先补类型和测试，把 searchKey、序列化、缓存恢复这些核心逻辑保护起来；第二步抽 view adapter，让表格和卡片的请求、转换、缓存规则下沉；第三步把筛选条件 Schema 化，让新增维度更配置化；最后再考虑状态机，把搜索流程和视图切换的状态转移收敛起来。

## 问题：如果说你所有做过的项目，你会怎么重构？

我会按项目类型分，不会用一个答案套所有项目。

### 1. 高级筛选系统

重构重点是：

- 视图 adapter 化。
- 筛选条件 Schema 化。
- 状态机收敛搜索流程。
- 强化 TypeScript 类型。
- 补核心逻辑测试。

目标是降低新增筛选维度和新增结果视图的成本。

### 2. 云呼 SDK

重构重点是：

- 把 BLL 状态机显式化。
- 把通话协议层、Worker 通信层、UI 层边界再拆清楚。
- 增加通话链路 traceId，串起拨号、SIP、媒体、Worker、UI 日志。
- 对 WebRTC stats 做统一采样和异常诊断。
- 给批量外呼队列补充更完整的取消、暂停、恢复能力。

目标是提升稳定性、可观测性和问题定位效率。

### 3. 架构升级项目

重构重点是：

- 继续清理 Vue 2 compat 警告。
- 老组件按业务模块迁移到 Composition API。
- 抽 composable 替代 mixin。
- 清理历史全局变量和隐式依赖。
- 建立性能预算和依赖治理规则。

目标是降低历史包袱，而不是为了新技术重写。

### 4. 数据可视化大屏

重构重点是：

- 图表配置 Schema 化。
- 数据请求和图表渲染解耦。
- 图表组件统一生命周期管理。
- 大屏适配逻辑统一抽象。
- 增加错误兜底和空状态。

目标是让新增图表和切换数据源更低成本。

总回答：

> 我重构项目不会一上来推翻重写，而是先找最影响长期维护的瓶颈：边界不清、状态混乱、类型缺失、测试不足、性能不可观测。然后按风险拆阶段，先补测试和类型保护核心逻辑，再抽稳定边界，最后逐步替换内部实现。重构的目标不是让代码看起来更高级，而是降低后续需求、排障和协作成本。

## 最后 2 分钟串联回答

如果面试官连续问这些问题，可以这样收束：

> React 里我会把纯计算和副作用分清楚：`useMemo` 缓存计算结果，`React.memo` 减少子组件无效渲染，副作用放在 `useEffect`，因为它有明确的执行时机和清理机制。状态管理上，`react-redux` 本质是通过 Provider 和订阅机制把 Redux store 接到 React 组件里；小范围状态我会用 `useReducer + Context`，大型跨模块状态才会用 Redux 这类更完整的方案。
>
> 项目上，我比较关注状态编排和可维护性。比如高级筛选系统里，我把复杂检索拆成条件编排、双视图隔离、searchKey 去重、RuntimeCache 恢复和口令分享；云呼 SDK 里，我用响应式观察思想处理持续变化的通话状态和多标签页同步，这类场景比单次 `async/await` 更适合用状态流思维。
>
> 平时我也会用 AI 工具提效，但我会把它放进工程闭环里：需求拆解、代码初稿、Debug 假设、Review 和文档都可以让 AI 辅助，但最终必须经过人工判断、类型检查、测试和真实验证。如果让我设计 Debug skill，我会把它做成假设、埋点、复现、验证、修复、清理日志的闭环，避免模型只凭猜测改代码。
