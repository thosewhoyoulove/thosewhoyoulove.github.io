# 智慧大屏数据可视化

---

## 问题：这个项目是做什么的？业务背景是什么？

这是一个面向企业管理层的数据驾驶舱项目，部署在会议室或展厅的大屏上，实时展示业务核心指标——销售漏斗、客户转化趋势、区域业绩分布、团队排名这些。

技术栈是 Vue 3 + ECharts + Vite，整个页面是一个全屏的 Dashboard，包含十几个图表模块，数据通过接口定时轮询刷新。

---

## 问题：可复用的 ECharts 组件是怎么设计的？

### 痛点

项目里有十几个图表，如果每个都从头写 ECharts 的初始化、resize 监听、销毁清理，代码重复量很大，而且容易漏掉某个生命周期处理导致内存泄漏。

### 设计：两层封装

**第一层：BaseChart 基础组件**

处理所有图表共性的事情：

```vue
<script setup>
// BaseChart 内部职责：
// 1. 挂载时 init 实例
// 2. ResizeObserver 监听容器尺寸变化，自动 resize
// 3. 卸载时 dispose 实例，移除监听
// 4. 暴露 onInit、onClick 等钩子给外部
// 5. watch option 变化时 setOption 更新
</script>
```

**第二层：类型图表组件**

基于 BaseChart 封装 `<LineChart>`、`<BarChart>`、`<PieChart>` 等，每个组件内部预设了一套 option 模板（颜色、字体、间距这些大屏风格的默认值），外部只需要传数据和少量定制配置：

```vue
<LineChart
  :xData="['1月', '2月', '3月']"
  :series="[120, 140, 180]"
  title="访问趋势"
/>
```

新加一个图表只需要传数据，不用关心初始化、resize、销毁这些，开发效率提升很明显。

### 配置驱动

后来产品希望运营能自己配图表，我又抽了一层 JSON schema：

```json
{
  "type": "line",
  "title": "访问趋势",
  "xAxis": ["1月", "2月", "3月"],
  "series": [120, 140, 180]
}
```

写了一个 `transformToOption(jsonConfig)` 函数，把简化的 JSON 配置转成标准 ECharts option。这样非前端人员通过后台配置界面就能新增图表，不用改代码。

## 问题：构建体积从 12MB 优化到 3.5MB 是怎么做的？

### 为什么会有 12MB

大屏项目用了完整的 ECharts（包含地图、3D、所有图表类型），加上 Ant Design 全量引入、moment.js 带全部语言包，还有一些大型地图 GeoJSON 数据文件，打包出来就到了 12MB。

### 优化手段

**1. ECharts 按需引入**

从 `echarts/core` 按需引入，只注册项目实际用到的图表类型和组件：

```js
import { use } from 'echarts/core'
import { LineChart, BarChart, PieChart } from 'echarts/charts'
import { GridComponent, TooltipComponent, LegendComponent } from 'echarts/components'
import { CanvasRenderer } from 'echarts/renderers'

use([LineChart, BarChart, PieChart, GridComponent, TooltipComponent, LegendComponent, CanvasRenderer])
```

完整 ECharts 约 800KB，按需引入后只有用到的部分会被打包，配合 Tree Shaking 效果很好。

**2. 大型资源异步加载**

地图 GeoJSON 数据（几 MB）不打进主包，改成运行时异步 fetch，只在用户进入地图页面时才加载。

**3. Brotli 压缩**

用 `vite-plugin-compression` 开启 Brotli 压缩，比 gzip 压缩率高 15-20%：

```js
// vite.config.js
import viteCompression from 'vite-plugin-compression'

export default {
  plugins: [
    viteCompression({ algorithm: 'brotliCompress' })
  ]
}
```

**4. 其他**

- moment.js 换成 dayjs（200KB → 2KB）
- Ant Design 按需引入
- 图片资源压缩 + WebP 格式

最终从 12MB 降到 3.5MB（gzip 后传输体积更小），首屏加载时间减少约 70%。

## 追问：WebP 是什么格式？如果浏览器不兼容怎么办？

> 这是一个经典的"工程化思维"考题。面试官本质上不是在考 WebP 的参数，而是在考：当新技术（高收益）和旧环境（低兼容）冲突时，你如何优雅地降级？

### WebP 是什么

WebP 是 Google 推出的图片格式，核心优势就是**更小**：

- 同等画质下，比 JPEG 小 25-34%，比 PNG 小 26%
- 既支持有损压缩，也支持无损压缩
- 支持透明度（Alpha 通道）和动画

### 降级方案（由浅入深）

**方案 A：HTML 原生 `<picture>` 标签（最优雅）**

让浏览器自己选格式，支持 WebP 就加载 WebP，不支持就自动 fallback：

```html
<picture>
  <source srcset="image.webp" type="image/webp">
  <img src="image.jpg" alt="示例图片">
</picture>
```

**方案 B：服务端 / CDN 动态判断（对前端零侵入）**

浏览器请求图片时，HTTP Header 的 `Accept` 字段会带上 `image/webp`。CDN 检测到支持就返回 WebP，不支持就返回原图。

实际项目中我们用的就是这种——阿里云 CDN 配一条规则就行，前端只管传原图，完全不用改代码。

**方案 C：JS 检测 + CSS 类名切换**

用 JS 加载一小段 Base64 编码的 WebP 图片，看是否成功，然后给 `<html>` 加 class：

```js
const webpSupport = new Image()
webpSupport.onload = () => document.documentElement.classList.add('webp')
webpSupport.onerror = () => document.documentElement.classList.add('no-webp')
webpSupport.src = 'data:image/webp;base64,UklGR...'  // 最小的 WebP base64
```

CSS 里根据类名切换背景图：

```css
.webp .hero-bg { background-image: url(hero.webp); }
.no-webp .hero-bg { background-image: url(hero.jpg); }
```

## 延伸：面试官压力测试"老浏览器不能用怎么办"的通用回答框架

遇到任何"新技术 vs 旧兼容"的问题（WebP、CSS Grid、容器查询、WebAssembly），都可以按这个结构回答：

### 第一层：明确渐进增强原则

> "我会采用渐进增强策略。现代浏览器享受更好的性能体验，老浏览器保证'能用'但'不一定完美'。"

### 第二层：拿出工程化手段

> "实际项目中不需要手动处理。CDN 支持自动转码，根据请求头动态分发格式；构建工具（PostCSS、Webpack 插件）能自动补全兼容代码。开发成本几乎为零。"

### 第三层：数据驱动决策

> "我会通过埋点监控看老版本浏览器的实际占比。如果占比低于 1%，且维护兼容性的成本（包体积增大、代码臃肿）超过了业务收益，我会建议'有损放弃'或引导用户升级。为极少数用户拖累全量用户的体验，本身也是一种不公平。"

### 万能公式

1. **肯定价值**：先说新技术解决什么痛点
2. **优雅降级**：具体的 Fallback 方案（`<picture>`、Polyfill、PostCSS）
3. **工程闭环**：CDN、构建插件等工具化手段
4. **业务视角**：兼容性是成本和收益的博弈

## 问题：防抖加载图表具体解决什么问题？

### 场景

大屏有两个高频触发图表更新的场景：

1. **窗口 resize**：大屏经常需要适配不同分辨率，拖动浏览器窗口时 resize 事件每秒触发几十次，如果每次都调 `chart.resize()`，会导致明显卡顿
2. **数据轮询刷新**：接口每 5 秒返回新数据，但如果用户正在切换时间范围，可能短时间内触发多次请求，每次响应都调 `setOption` 会导致图表闪烁

### 解决方案

**resize 防抖：**

```js
const debouncedResize = debounce(() => {
  chartInstance.resize()
}, 200)

resizeObserver = new ResizeObserver(debouncedResize)
resizeObserver.observe(containerRef.value)
```

容器尺寸变化后等 200ms 稳定了再 resize，用户感知不到延迟，但避免了高频重绘。

**数据更新节流：**

```js
const throttledSetOption = throttle((option) => {
  chartInstance.setOption(option, { notMerge: false })
}, 300)
```

用 `throttle` 限制 `setOption` 的调用频率，300ms 内最多执行一次。配合 `notMerge: false` 做增量更新而不是全量替换，减少渲染开销。

**懒加载：**

用 `IntersectionObserver` 监听图表容器是否进入视口，只有可见时才初始化 ECharts 实例和请求数据：

```js
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      initChart()
      observer.unobserve(entry.target)
    }
  })
})
```

大屏虽然是全屏展示，但有些模块在 Tab 切换后才可见，懒加载避免了一次性初始化十几个图表实例的性能压力。

## 问题：响应式适配是怎么做的？大屏分辨率差异大怎么处理？

大屏的分辨率差异很大，从 1920×1080 到 3840×2160 甚至更高都有。

### 整体布局：rem + vw 方案

根据设计稿基准宽度（1920px），动态计算根字体大小：

```js
function setRootFontSize() {
  const width = document.documentElement.clientWidth
  const fontSize = (width / 1920) * 100  // 基准 100px
  document.documentElement.style.fontSize = fontSize + 'px'
}
```

所有尺寸用 rem 单位，这样不同分辨率下等比缩放。

### 图表适配：ResizeObserver

每个 BaseChart 内部用 ResizeObserver 监听自身容器的尺寸变化（而不是 window.resize），这样不管是窗口变化还是布局调整导致的容器大小变化，图表都能自动适配。

### 主题统一

定义了一套 JSON 主题配置：

```js
const theme = {
  colorPrimary: '#00d4ff',
  bgColor: 'transparent',
  textColor: '#fff',
  fontSize: 12,
  // ...
}

echarts.registerTheme('darkScreen', theme)
```

所有图表实例初始化时传入主题名，保证风格一致。支持动态切换主题（比如白天/夜间模式），只需要重新注册主题再 `setOption` 即可。

## 问题：这个项目有什么技术难点？

### 1. 内存泄漏

十几个图表实例 + 定时轮询，如果组件卸载时没有正确清理，内存会持续增长。我在 BaseChart 的 `onUnmounted` 里做了完整的清理：

- `chart.dispose()` 销毁实例
- `resizeObserver.disconnect()` 移除监听
- `clearInterval` 清除轮询定时器

### 2. 大数据量渲染

有些图表（比如近一年的日粒度趋势）数据点有 365 个，直接渲染会卡。用了 ECharts 的 `sampling: 'lttb'`（最大三角形三桶算法）做数据降采样，视觉上几乎无损但渲染性能好很多。

### 3. 数据刷新时的视觉连续性

轮询刷新数据时，如果直接 `setOption` 替换，图表会闪一下。用 ECharts 的动画过渡配置，让新旧数据之间有平滑的过渡动画，用户感知是"数据在流动"而不是"图表在闪烁"。
