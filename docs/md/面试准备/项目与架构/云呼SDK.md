# 云呼 SDK

---

## 问题：说说你们项目的架构，业务逻辑层是怎么设计的？

我们做的是一个呼叫中心 SDK，它会以 iframe 或者脚本的形式嵌入到客户的 CRM 系统里，提供拨号、接听、通话管理这些能力。

整体上我们分了几层：

- **UI 层**：最上面是 React 的 UI 层，负责渲染拨号盘、通话卡片这些界面。
- **BLL 业务逻辑层**：中间有一个 BLL 业务逻辑层，承载所有的校验和策略。
- **Bridge 通信层**：再往下是一个 bridge 通信层，通过 SharedWorker 做跨 tab 的单例管理。
- **协议实现层**：最底层才是真正的电话协议实现，比如 SIP、硬件话机接口、双向回拨接口这些。

---

## 问题：BLL 层具体做了什么？

BLL 层对外暴露两个核心方法：

- **`callPhone`**：单次拨号。
- **`batchCallPhone`**：批量连续拨打。

调用方传进来号码、联系人数据、呼叫类型这些参数之后，BLL 层会先做一系列**前置校验**，全部通过才会真正发起呼叫。

校验主要有三类：

### 1. 号码合法性

- 号码不能为空。
- 正在通话中不能再打。
- 正在批量拨打任务里也不能插进来单独打。

### 2. 线路和坐席状态

- 检查当前选择的呼叫类型有没有配置对应的坐席。
- 坐席有没有过期。
- 当前选的线路是不是在维护中。
- 对于云呼还有一个额外限制：**必须是 HTTPS 协议**，因为浏览器的麦克风权限在非 HTTPS 下是拿不到的。

### 3. WebRTC 媒体流检测（仅云呼路径）

用了一个叫 `detectrtc` 的库，检查浏览器是不是支持 WebRTC、有没有接麦克风，然后还会真正调一次 `getUserMedia` 去请求音频流，确保麦克风权限是真的开着的。

**三关全过了才会去建 SIP 连接、发起呼叫。**

## 问题：四种呼叫类型是怎么区分的？

四种类型是后端接口下发的，字段叫 `default_type`，值是 0 到 3：

| type | 名称 | 说明 |
| --- | --- | --- |
| 0 | 硬件外呼 | 坐席用实体话机，SDK 只负责把参数发过去，让话机那边自动拨出 |
| 1 | 云呼 | 纯软件 SIP 电话，浏览器直接用 WebRTC 跑音频，不需要任何硬件设备 |
| 2 | 双向回拨 | 系统先呼销售的手机，接通后再呼客户，两端都通过电话网络连通，不走浏览器音频 |
| 3 | 直呼 | 直接用手机打出去，不经过服务器中转 |

BLL 层里有一个 `switch`，根据这个 type 把请求分发到对应的 action 函数，然后通过 bridge 层广播到 SharedWorker 里去执行。

**云呼比较特殊**：它在主线程里就把 SIP 对象建好了；其他三种都是丢给 Worker 处理的。

## 问题：加密号码是怎么处理的？

有些平台传过来的号码是加密的，加密方式是我们和后端约定的一种**差分编码**——把每一位字符的 ASCII 码和前一位做差，编码成另一个字符串。解密的时候反过来推回去就行了。

这个解密逻辑在一个独立的工具函数 `decryptNumber` 里，BLL 层和 UI 层都有用到它。

批量拨打的时候，会对 `contactList` 里每一条联系人数据单独判断有没有 `hasEncrypt` 标志，有的话先解密再做格式化，比如去掉 `+86` 前缀、去掉横线和点这些。

## 问题：SharedWorker 的作用是什么？

用 SharedWorker 主要是为了解决**多标签页**的问题。

如果用户同时开了好几个 CRM 的 tab，我们不希望每个 tab 都各自维护一条 SIP 连接，那样资源浪费而且状态会乱。所以我们把电话的核心状态和连接都放在 SharedWorker 里，所有 tab 共享同一个实例，状态通过 bridge 层同步回来。

这样不管用户在哪个 tab 接的电话，其他 tab 的 UI 也能实时更新。

## 问题：简历上写的"深度定制 sip.js"体现在哪？

sip.js 是一个开源的 SIP 协议库，我们在它的基础上打了一个 patch，主要改了两个地方。

### 1. mDNS 地址问题

Chrome 在局域网里，有时会把 WebRTC 的 SDP 里的 IP 地址替换成 `.local` 结尾的 mDNS 主机名，对端如果没有 mDNS 解析能力就会媒体不通——通话虽然接通但没声音。

我们在 sip.js **接受来电、构造 200 OK** 的那个时机，加了一段 `dealMdns` 的逻辑，把含 `.local` 的行替换成固定 IP，绕开这个问题。

### 2. RTCPeerConnection 代理注入

把 `new RTCPeerConnection()` 替换为：

```js
window.tungeeRTCPeerConnection || window.RTCPeerConnection
```

这样我们可以在外部挂载一个**代理对象**，用于通话过程中的 WebRTC 质量监控——劫持原生 `RTCPeerConnection` 之后，就能拿到 `getStats` 的数据做丢包率、RTT 这些指标的上报，而不需要侵入通话主流程。

## 问题：RxJS 构建状态观察体系是怎么做的？

这里要说清楚，我们**没有直接用 RxJS 这个库**，而是自己实现了两套响应式机制，理念上借鉴了响应式编程的思路。

### 第一套：基于 ES6 Proxy 的状态劫持（Observer.ts）

核心思路和 Vue 2 的响应式原理类似——对 ViewModel 做 Proxy，任何属性赋值都会触发 watcher，watcher 再把变更的 key 路径和新值广播出去，驱动 UI 更新和 SharedWorker 的状态同步。

### 第二套：myRx —— 顺序 Promise 编排器

名字是向 Rx 致敬。它解决的问题是：初始化阶段有多个**串行异步操作**，比如先注册 SIP、再拿坐席信息、再建连接，用普通 Promise 嵌套太深。

`myRx` 让你传入一组函数，自动把前一步的结果注入到下一步，还支持 `.every()` 钩子在每一步回调，用起来像 RxJS 的 `pipe` 操作符但更轻量。

## 问题：SharedWorker + 消息总线解决了什么问题？怎么做的？

我们的 SDK 是以 iframe 形式嵌入到客户系统里的，用户打开 CRM 可能会有多个标签页。如果每个 tab 各自维护一条 SIP 连接，会有几个问题：

1. **资源浪费**：SIP 注册多次。
2. **状态抢占**：一个 tab 的操作会影响另一个 tab 的通话状态。
3. **切 tab 丢状态**：通话时用户切换 tab，新 tab 看不到当前通话的状态。

### 解决方案

用 **SharedWorker** 作为所有 tab 共享的单例，所有真正的电话操作都在 Worker 里执行，各个 tab 只是发消息过去。

**Safari 降级**：SharedWorker 在 Safari 不支持，所以我们在 `createBus` 里做了降级——检测到没有 SharedWorker 就动态加载一个兼容版本，用内存事件模拟同样的行为。

### 消息总线（BusAbstract）

自己实现了一个 `BusAbstract`，它维护了：

- **pub/sub 订阅表**
- **心跳检测**
- **广播队列**
- **两步页面销毁协议**

### 两步销毁协议

1. tab 在关闭前先发一个 **`beforeUnloadPass`**，告知其他 tab「我准备关了」，这时候其他 tab 可以接管状态。
2. 等 tab 真正 `beforeunload` 的时候，Worker 再把这个 tab 从注册表里清掉；如果它是最后一个 tab，就挂断电话。

这样解决了直接关 tab 导致通话状态残留的问题。

### 状态同步闭环

Worker 侧维护了：

- `tabs: Set` —— 存活的标签页集合
- `globalPortsMap: Map<tabId, port>` —— 每个 tab 的通信端口

状态变更时用 `sendMsgToEveryPort` 把最新状态推给所有存活的 tab，每个 tab 本地的 ViewModel 收到之后通过 **Proxy 劫持自动触发 UI 重渲染**，整个链路闭环。

## 问题：多端兼容体现在哪？

主要是三个场景的兼容。

### 1. iframe 嵌套的跨域通信安全

我们的消息总线要在 Tab 页和隐藏 iframe 之间用 `postMessage` 通信，iframe 在接收消息之前必须校验来源，防止恶意页面注入消息。

我们在 iframe 壳里实现了一个 `validateOrigin`，用正则检查 `tungee.com` / `tangees.com` 两个域名。

还有一个坑：某些 iframe 嵌套场景下 `event.origin` 会拿不到，需要从 `event.originalEvent.origin` 兜底——这是调试了好几个环境才发现的问题。

### 2. 钉钉容器兼容

钉钉有自己的 webview 环境，有几个问题要解决：

- **域名不标准**：不是 `tungee.com`，而是 `dingtalkcloud.com` 或 `-dd.tungee.cn` 这类，所以 API 请求的 origin 要单独解析。
- **SharedWorker 初始化数据可能为空**：普通环境我们是拿到数据才渲染，但钉钉里必须**强制渲染**，不然会白屏。
- **JSAPI 调用**：某些功能要调钉钉的 JSAPI（比如部门人员选择），需要判断 `dd.env.platform` 来走不同的逻辑。

### 3. 乾坤微前端兼容

SDK 有可能运行在乾坤微前端框架下，这时候 API 的 origin 由主应用通过 `window.__INJECTED_SDK_PATH_BY_QIANKUN__` 注入，我们也处理了这个情况。

## 问题：可插拔扩展 API 是怎么设计的？

我们的 SDK 是给多个不同的宿主系统用的，不同宿主对 UI 的要求不一样，但我们不可能为每个宿主改一套代码，所以设计了一套**双向的扩展协议**。

### 核心：`window.TUNGEE.setProperty`

这是宿主系统提供的一个全局注册接口。SDK 启动的时候，会往上面挂两个方法：

- **`initEngageModule`**：宿主调它来告诉 SDK 要激活哪些功能模块。
- **`initSDKConfig`**：传入 token、版本号等配置。

反过来，SDK 也会往 `TUNGEE` 上挂操作句柄，比如 `SmartPhone.onOpenCreateAITaskTap`，宿主调这个方法就能打开智能外呼的弹窗，并且可以传配置进来。

这样宿主不需要关心 SDK 内部的 React 树长什么样，两边通过约定好的接口解耦。

## 问题：宿主注入 React/DOM 节点怎么实现的？

这个是通过 **slots 机制**做的。

### 类型定义

```ts
type SlotType = ReactNode | Node
```

同时支持 React 组件和原生 DOM 节点。

### 使用方式

模块配置对象里有 `slots` 字段，宿主可以传入自己的组件。以智能外呼任务为例：

- `slots.Content` —— 替换「号码筛选」步骤的内容
- `slots.ResultContent` —— 替换创建结果页
- `slots.Footer` —— 替换底部按钮区

SDK 内部会检查 `if (slots?.ResultContent)` 决定渲染宿主的组件还是默认组件，并且把任务状态、结果数据作为 props 传给宿主组件。宿主只管写 UI，数据由 SDK 提供。

### 向后兼容：normalizeConfig 胶水层

早期的 API 是把 `content`、`footer`、`resultContent` 这些字段直接放在配置对象顶层的，后来统一到了 `slots` 命名空间下。

为了不破坏老用户，我们加了一个 **`normalizeConfig`** 胶水层，在进入组件之前把旧的扁平字段自动转成 `slots.Content`、`slots.Footer` 这种新格式，新老调用方式都能透明地工作。

## 问题：单呼和群呼的状态机是怎么统一的？

我们定义了一个 **CallState 枚举**：

```
ready → dial → ring → turnOn → break
```

SIP、硬件话机、双呼、直呼四种类型**全部走这一套状态**，通过 `Channel.setState()` 统一驱动。

### Channel 层的两个保护

1. **状态只能向前推进，不能回退**：`state < preState` 直接 return，解决多个异步状态源同时更新时的竞争问题。
2. **一旦进入 break，后续任何状态变更都被拒绝**：通话已经结束了，不能再被「拉活」。

接通时自动开始录音计时，挂断时自动停止计时并销毁通话实例。

### 群呼的独立状态

群呼有自己独立的 **BatchCallState**：

```
init → running → paused → finished
```

**BatchCallQueue** 把这两套状态串联起来——每一通单呼必须等 `channel.wait(state === break)` 解决之后，群呼才推进到下一通。这样群呼的进度和单呼的生命周期是完全同步的。

## 问题：限频和合规是怎么做的？

分三个层面。

### 1. 前端前置 guard

在 `callPhone` 里做了一堆前置校验：

- 号码不能为空
- 通话中不能再呼
- 连拨任务进行中不能插入单呼
- 必须 HTTPS
- 坐席不能过期
- 线路不能在维护中

任何一条不通过直接 reject 并返回错误码，错误文案统一在 `constant.ts` 里维护。

### 2. 服务端合规预筛

批量拨打前还有一道服务端合规预筛，把号码列表送到 `filterContactList` 接口，服务端检查：

- 过多拨打
- CRM 已成交客户
- 地区限制

不合格的号码从列表里剔除后再开始任务。

### 3. 群呼执行层面的限频

- SIP 和硬件话机两种类型有 **3 秒的强制冷却间隔**，每通之间必须等够 3 秒。
- 间隔期间会推倒计时消息给 UI 展示「3s 后拨打下一通」，用户也可以随时暂停。
- 服务端返回特定错误码（比如 `45009` 使用人数过多、`45006` 线路暂停）时，群呼任务会**立刻终止**，不会继续跑。

## 问题：埋点和监控是怎么做的？

### 链路埋点（全自动）

业务代码不需要手动插入。我们在呼叫链路的关键节点调 `linkPoint`：

- checkrtc 开始 / 结束
- SIP 注册成功
- 接口返回响铃
- 接口返回接通
- 异常挂断

每次打点都带**时间戳、号码 ID 和当前阶段**，5 条聚合成一个 batch 通过 **`sendBeacon`** 上报。`sendBeacon` 的好处是页面关闭时也不会丢数据。

### WebRTC 质量监控

我们在应用初始化时用一个**透明代理**替换了原生 `RTCPeerConnection`，然后每秒调 `getStats()` 采集 7 类指标：

| 指标 | 说明 |
| --- | --- |
| 音频电平 | 判断是否静默 |
| 发送丢包率 | 上行网络质量 |
| 接收丢包率 | 下行网络质量 |
| 往返延迟（RTT） | 通话延迟感知 |
| 网络抖动 | 音频卡顿预警 |
| 发送码率 | 带宽占用 |
| 接收码率 | 带宽占用 |

**阈值判断**：延迟超过 0.5 秒或者连续丢包，触发防抖 5 秒的网络提示。

**上报时机**：通话结束或者连续 20 秒检测到静默，批量上报全部指标。

这样线上出了通话质量问题，可以直接从日志里查到那通电话的完整网络表现。
