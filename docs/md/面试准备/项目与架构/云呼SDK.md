## 面试回答（开场 · 建议 2～3 分钟）

### 一句话

这是一个**嵌在 CRM 里的打电话 SDK**。销售不用离开客户列表，点一下就能拨号、接听、批量外呼、建任务。对外就是一个全局对象，比如 `window.TUNGEE.getProperty('SmartPhone').onMakeCall(...)`。

### 解决什么问题

销售常同时开好几个 CRM 标签页。如果**每个页面各自连电话**，会出现：

- 重复注册，浪费资源  
- A 页在通话，B 页还以为空闲，又能拨一通  
- 换了个 Tab，看不到当前通话状态  

所以我们做的不是「一个拨号按钮」，而是：**全浏览器只有一套电话后台 + 所有页面状态实时同步 + 统一的通话进度模型**。

### 架构（从上到下，口述用这个）

```
CRM 页面（React 界面）
    ↓  用户点按钮、看响铃/接通
页面数据层          管「界面该显示什么」，改数据自动触发更新
    ↓
业务校验层          管「能不能打」：号码、坐席、线路、麦克风、HTTPS
    ↓
Bridge（翻译官）    传「指令」和「状态」，连接页面和后台
    ↓
SharedWorker        浏览器里唯一的「电话机房」：真正发起呼叫、维护进度
    ↓
具体实现            云呼(WebRTC) / 硬件话机 / 双呼 / 直呼
```

**分工记住两句：**

- **页面侧**：展示 + 校验 + 发指令  
- **后台侧**：真正打电话 + 推进状态 + 通知所有 Tab  

### 四种外呼方式（后端配置决定）

| 类型 | 编号 | 大白话 |
| --- | --- | --- |
| 硬件外呼 | 0 | 实体话机打，浏览器只传参数 |
| 云呼 | 1 | 浏览器里直接说话，走 WebRTC |
| 双向回拨 | 2 | 平台先呼销售手机，再呼客户，声音在手机里 |
| 直呼 | 3 | 同样由平台调度，SDK 传参和同步状态 |

### 三个能体现深度的点

1. **多 Tab + 跨域名**  
   电话后台必须全局唯一，但 CRM 和 SDK 往往不在同一个域名。我们用**隐藏 iframe + postMessage** 当中转，让页面能连上 SharedWorker；钉钉等不支持的环境再降级成单页模式。

2. **单次 + 批量共用一套通话进度**  
   每一通电话都经历：拨出 → 响铃 → 接通 → 挂断。批量不是 for 循环瞎打，而是**等上一通彻底结束**才打下一通，云呼/硬件还有 3 秒间隔，遇到平台错误会直接停任务。

3. **线上能排障**  
   改过开源电话库，解决「接通了但没声音」；通话过程中采集网络延迟、丢包等指标，配合各阶段耗时埋点，线上出问题能定位是网络、设备还是服务端。

### 30 秒压缩版（时间不够背这个）

> 嵌入式云呼 SDK，支持硬件、云呼、双呼、直呼。难点是多标签页、跨域名还要共享一条电话连接，用 iframe 中转 + SharedWorker 单例解决。批量外呼用队列等上一通挂断再打下一通。改过 sip.js 修无声问题，并用 WebRTC 统计做通话质量监控。

## 问题：说说项目架构？

### 口述版

SDK 以**脚本 + 隐藏 iframe** 嵌进客户 CRM。销售在 CRM 里操作，我们负责电话能力和 UI。

从上到下：

1. **React 界面**：拨号盘、通话中卡片、任务弹窗。  
2. **页面数据层**：把「打开拨号盘、填入号码」变成数据变化，不直接在组件里写打电话逻辑。  
3. **响应式监听**：类似 Vue，改数据会自动触发同步（不用每个按钮手动 dispatch）。  
4. **业务校验层**：所有「能不能打」的规则放这里，组件只负责展示。  
5. **Bridge**：页面和 SharedWorker 之间的翻译官（详见 Bridge 专节）。  
6. **SharedWorker**：维护四套电话能力（云呼/硬件/双呼/直呼）和一份全局状态。

**补充（加分）：** 有的 Tab 是「主操作页」，能发起呼叫；有的 Tab 只同步状态、不重复建连接——避免多入口冲突。

### 代码对照（备查）

`Model/Ui`、`Obsever.ts`、`BLL`、`bridge/index.ts`、`powerSingleton/coreLogic.ts`

## 问题：业务层（BLL）具体做了什么？

### 口述版

可以把它理解成**打电话前的总安检 + 分流台**。

对外主要两个入口：

- **单呼**：打一通  
- **批量**：连续打多通  

用户点拨打后，先做**前置检查**，任何一项不过都直接提示，不会真正发起：

| 检查项 | 说明 |
| --- | --- |
| 号码 | 不能为空 |
| 并发 | 正在通话时不能再打；批量任务进行中也不能插单呼 |
| 坐席 | 有没有配置、有没有过期 |
| 线路 | 是否在维护、是否被禁用 |
| 云呼专属 | 必须 HTTPS；浏览器支持 WebRTC；麦克风能拿到 |

全部通过后，看后端配置的**外呼类型编号**（0～3），把请求交给 Bridge，由后台执行对应方式。

### 30 秒版

> 业务层就是安检口。查号码、查坐席、查线路、云呼还要查 HTTPS 和麦克风。过了才发指令给后台，并根据配置走硬件、云呼、双呼或直呼。

### 代码对照（备查）

`BLL/index.ts` → `callPhone`、`batchCallPhone`、`checkrtc`

## 问题：四种呼叫类型怎么区分？

### 口述版

**谁决定：** 后端接口返回当前企业默认用哪种方式（字段 `default_type`，0～3）。

**谁执行：** 都在 SharedWorker 里，页面只发指令。

| 编号 | 名称 | 声音在哪 | 浏览器干什么 |
| --- | --- | --- | --- |
| 0 | 硬件 | 话机 | 把号码、客户信息发给硬件 |
| 1 | 云呼 | 浏览器耳机/麦克风 | SIP + WebRTC |
| 2 | 双呼 | 销售和客户各自手机 | 调平台接口，不传浏览器音频 |
| 3 | 直呼 | 手机 | 同样调平台，SDK 同步状态 |

**完整路径（口述一句）：**  
用户点击 → 页面校验 → Bridge 发指令 → 后台真正呼叫 → 状态变（响铃/接通/挂断）→ 推回所有 Tab 更新界面。

## 问题：加密号码怎么处理？

### 口述版

部分系统传来的号码是**加密字符串**，不是明文。我们和后端约定了一种**差分编码**：每一位和前一位做运算得到密文，解密时反向推回。

- 工具函数统一解密，单呼、批量共用。  
- 批量时逐个看「是否加密」标志，加密则先解密，再去掉 `+86`、横线等格式。

### 代码对照（备查）

`utils/decryptNumber.ts`、`hasEncrypt` 标志

## 问题：SharedWorker 是干什么的？

### 口述版

**SharedWorker = 浏览器里单独跑的一份「电话机房」**，同一个浏览器所有 Tab 共用。

**不用它会怎样：**

- 每个 Tab 各注册一次 SIP → 浪费、易冲突  
- A Tab 在通话，B Tab 不知道  
- 关 Tab 后连接可能 orphaned  

**用了之后：**

- 全浏览器**只有一套**电话连接和通话状态  
- 任意 Tab 的操作，其他 Tab 界面跟着变  

**跨域名怎么办：** CRM 和 SDK 不同源，不能直接连 Worker → 用隐藏 iframe + postMessage（下节和 Bridge 节详述）。

### 30 秒版

> SharedWorker 让全浏览器只有一份电话后台。所有 CRM 标签页共享通话状态，避免重复注册和双开通话。跨域名时用 iframe 中转连上去。

## 问题：Bridge 层是怎么进行数据转发的？

> 本节已口语化，可直接背。

### 一句话

Bridge 是**页面和电话后台之间的翻译官**：不发 UI、不直连电话网，只负责**把操作发出去、把状态收回来**。

### 前台 + 机房类比

| 角色 | 是什么 |
| --- | --- |
| CRM 页面 | 前台：点拨打、看状态 |
| Bridge | 内线：传话、对账 |
| SharedWorker | 机房：唯一总机 |
| 隐藏 iframe | 跨域名时的中转窗口 |

### 传两类东西

**① 指令** — 「去做一件事」：拨打、挂断、批量下一通、连硬件  

**② 状态** — 「现在什么样」：是否在通话、响铃还是接通、选的哪条线  

> **口诀：指令管动作，状态管展示；动作在后台做，结果再以状态推回页面。**

### 按时间讲链路

1. **打开页面**：连后台 → 拉当前全局快照 → 再渲染界面 → 登记本 Tab 在线  
2. **改界面数据**：监听变化 → 攒一批再发给后台 → 其他 Tab 同步  
3. **点拨打**：校验 → 发指令 → 后台呼叫 → 状态推回 → React 刷新  
4. **关 Tab**：先发「我要关了」→ 若是最后一个 Tab，后台挂断通话  

### 跨域名

```
CRM 页面  ←→  postMessage  ←→  隐藏 iframe（SDK 域名）  ←→  SharedWorker
```

### 两个加分细节

- **静默写回**：后台推来的状态写进页面时，暂时不要再次上报，避免死循环。  
- **攒批发送**：一次操作可能改多个字段，合并成一条消息更高效。

### 30 秒版（推荐背）

> Bridge 在页面和 SharedWorker 之间。传两类东西：指令（拨打、挂断）和状态（是否在通话、响铃/接通）。页面改数据会同步到后台和其他 Tab；点拨打发指令，后台执行完再把状态推回来。CRM 和 SDK 不同域名，用隐藏 iframe + postMessage 中转。

### 和消息总线的区别

Bridge 管**发什么、收到怎么更新**；iframe 里的消息总线管**怎么送到 Worker、心跳、排队**。一个管含义，一个管快递。

### 代码对照（备查）

`initBridge`、`createBridge`、`sipAction` 系列、`noWatcherSet`、`changeMap`

## 问题：多 Tab + 跨域名是怎么做的？

### 口述版

**三个痛点：** 重复连电话、Tab 互相抢状态、换 Tab 看不见通话。

**核心矛盾：** SharedWorker 要求**同源**；CRM 在 `crm.xxx.com`，SDK 在 `engage.xxx.com`。

**解法：** 每个 CRM 页里藏一个 **SDK 域名的 iframe**，页面和 iframe 用 **postMessage**（浏览器标准跨窗通信），iframe 和 Worker **同源直连**。

**为什么不用 BroadcastChannel？**  
也要同源，且**没有独立线程**，跑不了长连接和 SIP，只能广播字符串，不够当「机房」。

**安全：** iframe 收到消息时**校验来源域名**白名单；发送方有时用 `*`，靠接收方过滤。嵌套 iframe 时 `origin` 可能丢失，要兜底取 `originalEvent.origin`。

**降级：** 不支持 SharedWorker（如钉钉）→ 退化成 iframe 模拟后台，**只能单 Tab 同步**，但单页能打。钉钉还要**强制先渲染界面**，否则白屏。

**关 Tab 的两步：**

1. 关之前：通知其他 Tab「我要关了」  
2. 真关闭：从在线列表移除；若是最后一个 Tab，**挂断正在进行的通话**

**状态怎么回到界面：** 后台维护「在线 Tab 列表 + 每个 Tab 的通信口」，有变化就推给所有 Tab → 页面静默写入数据 → React 刷新。

### 30 秒版

> 多 Tab 共享一份 SharedWorker。跨域名用隐藏 iframe + postMessage 中转。关 Tab 分两步通知，最后一个 Tab 关掉会挂断电话。钉钉等环境降级为单页模式。

### 代码对照（备查）

`BusAbstract`、`validateOrigin`、`beforeUnloadPass`、`compatibleSharedWorker`

## 问题：深度定制 sip.js 体现在哪？

### 口述版

云呼基于开源库 **sip.js**，我们用 **patch-package** 打补丁（安装依赖后自动生效）。

**问题 1：接通了但没声音**  
Chrome 有时把网络地址写成 `.local` 结尾的名字，对方解析不了 → 信令成功、音频不通。  
**改法：** 在**接听、回复接通**时，把 SDP 里这类地址替换成可用 IP。

**问题 2：想监控通话质量，又不想改业务代码**  
**改法：** 创建 WebRTC 连接时，优先用我们挂好的**代理构造函数**，外面包一层原生 API，里面照常连电话，同时能调 `getStats` 采延迟、丢包等。

### 30 秒版

> patch 了两处：一是修 Chrome mDNS 导致接通无声音；二是 WebRTC 连接走代理，方便采网络质量，不侵入拨号主流程。

### 代码对照（备查）

`patches/sip.js+0.17.1.patch`、`dealMdns`、`window.tungeeRTCPeerConnection`

## 问题：页面数据怎么自动同步？（常被问 RxJS）

### 口述版

**没有用 RxJS 管全局状态**（项目里虽有依赖，但核心是自研的）。

**方案一：响应式数据（主方案）**  
页面侧一份「全局数据对象」，用 **Proxy** 监听属性修改——思路和 Vue 2 类似。一改数据，就通知 Bridge 同步到后台和其他 Tab，再驱动 React 刷新。

**方案二：初始化时的 Promise 链**  
开机时要串行做多步：拉配置 → 注册电话 → 拿坐席信息……写了一个轻量工具把多步 Promise 串起来，避免回调套娃。名字借用了 Rx 的概念，但不是 RxJS 本身。

### 30 秒版

> 全局状态靠 Proxy 监听数据变化，自动同步到后台和所有 Tab。初始化多步异步用自研的 Promise 链工具，没有用 RxJS 做状态流。

## 问题：多端兼容怎么做？

### 口述版

| 场景 | 会遇到什么 | 怎么处理 |
| --- | --- | --- |
| iframe 跨域通信 | 恶意页面伪造消息 | 接收时校验域名白名单 |
| 钉钉 WebView | 没有 SharedWorker、域名不同、初始数据可能空 | 降级单页、改白名单、**必须先渲染再等业务数据** |
| 乾坤微前端 | 接口域名由主应用注入 | 读主应用注入的全局路径配置 |

**总思路：** 启动时**探测环境**，走不同分支，保证「至少能打电话」。

## 问题：怎么让不同 CRM 复用同一套 SDK？

### 口述版

SDK 要卖给很多宿主，不能每个客户改一套 UI。

**约定一个全局对象 `window.TUNGEE`：**

**宿主 → SDK：**

- 告诉 SDK 要加载哪些模块（只要拨号盘、还是也要任务中心）  
- 传入 token、环境等配置  

**SDK → 宿主：**

- 挂出 `SmartPhone.onMakeCall` 等方法，宿主调 API 就能打开能力  
- 还有建客户、发短信、诊断任务等独立入口  

宿主**不需要知道 SDK 内部 React 长什么样**，只调约定好的方法。

**UI 可替换（插槽）：**  
比如智能外呼任务，宿主可以传入自己的 React 组件，替换「结果页」「底部按钮」等区块；SDK 把任务数据当 props 传进去。

**兼容老接口：**  
早期参数写在配置对象顶层，后来统一到 `slots` 里；进组件前有一层转换，老宿主不用改代码。

### 30 秒版

> 通过 window.TUNGEE 双向约定：宿主调 SmartPhone 打开能力，SDK 也提供 init 接口收配置。UI 用 slots 让宿主替换部分区块，老 API 用适配层兼容。

## 问题：单呼和批量怎么统一？

### 口述版

**每一通电话**都有相同的生命周期：

```
拨出 → 响铃 → 接通 → 挂断
```

硬件、云呼、双呼、直呼**共用这套进度**，方便 UI 和批量队列统一处理。

**防止状态乱跳：**

- 进度只能往前走，不能从「响铃」倒回「拨出」  
- 一旦「挂断」，这通彻底结束，不能再被拉活  
- 接通后开始计时、录音；挂断后清理  

**批量任务**另有任务级状态：准备 → 进行中 ⇄ 暂停 → 结束  

**和单通怎么串：**  
队列里**等上一通完全挂断**，才发起下一通。云呼/硬件自动连拨时，两通之间**强制等 3 秒**（界面可显示倒计时）；双呼/直呼批量没有这 3 秒。  
平台返回「线路暂停」「使用人数过多」等错误时，**整批任务直接停**，不继续硬打。

### 30 秒版

> 单通统一走拨出→响铃→接通→挂断。批量用队列，上一通挂断才打下一通，云呼/硬件间隔 3 秒，支持暂停，遇到平台错误熔断。

### 代码对照（备查）

`CallState`、`BatchCallQueue`、`channel.wait(break)`、错误码 45006/45009

## 问题：限频和合规怎么做？

### 口述版

| 层级 | 做什么 |
| --- | --- |
| 按钮层 | 拨打等操作 **3 秒内只能点一次**，防手抖连点 |
| 业务层 | 通话中不能再打、批量中不能插单呼等（见 BLL 节） |
| 批量合规 | 开打前把号码列表送服务端**预筛**，去掉超限、已成交、地区不允许等 |
| 队列层 | 云呼/硬件批呼 3 秒间隔；致命错误码立刻停任务 |

错误提示文案集中维护，前端各处的提示一致。

### 30 秒版

> 三层：按钮防抖、业务层并发校验、批量前服务端合规预筛；队列里还有间隔和错误熔断。

## 问题：埋点和监控怎么做？

### 口述版

分两条线：**通话走到哪了** + **网络好不好**。

**① 链路埋点（过程追踪）**  
在关键节点打时间点：例如开始查麦克风、注册电话成功、响铃、接通、挂断。  
带上时间戳、号码、阶段名称，多条合并上报，方便看「卡在哪一步」。  
（具体上报通道以线上配置为准。）

**② WebRTC 质量（网络追踪）**  
通话建立后，周期性读取浏览器提供的**连接统计**：

| 看什么 | 说明 |
| --- | --- |
| 延迟 | 超过约 0.5 秒可提示用户网络差 |
| 丢包 | 上行/下行分开看 |
| 抖动 | 预测卡顿 |
| 音量电平 | 判断是否长时间静默 |

网络差时弹提示（防抖，避免一直弹）；通话结束或长时间静默后，用 **sendBeacon** 上报（页面关了也不容易丢）。

**排障怎么用：**  
链路看「阶段耗时」→ 质量看「RTT/丢包」→ 区分是权限、网络还是服务端问题。

### 30 秒版

> 链路埋点看各阶段耗时；WebRTC 统计看延迟丢包抖动。两条线合起来定位是设备、网络还是平台问题。

### 代码对照（备查）

`linkPoint`、`webRtcMonitor.ts`、`sendBeacon`、`getStats`

## 踩坑速查（用故事讲，比背表好）

| 用户现象 | 原因（口语） | 我们怎么修 |
| --- | --- | --- |
| 接通了没声音 | 浏览器把地址写成了 `.local`，对方听不懂 | 改 sip.js，替换 SDP 里的地址 |
| 多 Tab 状态对不上 | 每个页面各自连电话 | SharedWorker + iframe 中转 |
| 关页面电话还在 | 后台不知道 Tab 没了 | 关页面前先发通知，最后一个 Tab 关就挂断 |
| 状态闪回乱跳 | 多个地方同时改进度 | 通话进度只允许向前 |
| 钉钉打开白屏 | 等数据才渲染，钉钉里数据可能空 | 钉钉环境先渲染界面 |
| 收不到跨窗消息 | 嵌套 iframe 里 origin 丢了 | 从 originalEvent 兜底取 |

## 高频追问（全口语简答）

### 为什么要单独的业务层？

组件只该管展示。能不能打、走哪条线、什么错误提示，规则很多且会变，集中在一层好维护，也避免 React 里堆满 if-else。

### 为什么用 SharedWorker 而不是每个 Tab 自己连？

要**全浏览器唯一**的电话实例，还要在**独立线程**里跑 SIP 和长连接。BroadcastChannel 只能传消息，没有这种「机房」能力。

### 批量怎么保证不乱序、可暂停？

队列 + 「上一通挂断才下一通」+ 云呼/硬件 3 秒间隔 + 暂停/恢复 + 平台错误熔断。

### Bridge 和消息总线区别？

Bridge 决定**传什么、更新什么**；消息总线决定**怎么 postMessage、心跳、排队**。

### 通话质量问题怎么查？

先看链路埋点卡在哪一步；再看延迟、丢包——网络问题还是注册/权限问题，大致能分开。
