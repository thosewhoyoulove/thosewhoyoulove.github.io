# 企业级高级筛选（进阶检索）系统

---

## 问题：简历上说的企业级高级筛选系统，体现在哪里？

高级筛选系统是整个获客产品的核心功能，用户量和使用频率都很高。我负责的是**企业维度**这条线的前端架构设计和落地。

整体上我做的是一个统一的编排层，叫 **EnterpriseFilter**，它本身不耦合具体的筛选维度，而是把条件面板、结果列表、操作工具栏这些能力模块**插件式**地接进来。这样不管后续加什么维度，或者换业务场景，框架不用动，只需要配置或替换对应的子组件。

核心设计思路就是把「变」的东西收到 prop 和配置里，把「不变」的数据流编排留在框架里，新业务接进来的成本很低。

---

## 问题：条件组装的数据流是怎么设计的？

我设计了一条完整的数据流：

### 1. 条件选择与持久化

用户在 **DimensionSelector** 里选条件，每次变更都会实时序列化并持久化到 `localStorage`，防止刷新丢失。

### 2. 条件下发

点击搜索时，把条件 JSON 序列化后带上上下文参数一起下发给后端：

- `mode`：全部 / 可跟进 / 未解锁等
- `industry`：金融场景标识
- 其他业务上下文参数

### 3. 条件口令功能

用户可以把当前条件序列化成一个**口令字符串**复制分享，其他人粘贴进来就能还原一模一样的搜索条件。这个功能用于团队内的最佳实践分发——比如销售主管配好一套高转化的筛选条件，生成口令发到群里，团队成员直接粘贴就能用。

## 问题：分页是怎么处理的？卡片和表格两种视图怎么协调？

因为我们有**卡片视图**和**表格视图**两种展示模式，它们的分页参数完全独立——各自的 `currentPage`、`pageSize` 是分开维护的。

### 统一抽象

我抽象了两个方法：

- **`getActivePage()`**：获取当前视图的页码
- **`getActivePageSize()`**：获取当前视图的每页条数

调用方不用关心当前是哪种视图，统一调这两个方法就行。

### 切换视图的"新鲜度"判断

切换视图时有个**新鲜度判断**逻辑：如果两边数据都是基于同一套筛选条件请求过的，就不重复打接口，避免浪费。只有当条件变更过、或者目标视图还没有请求过当前条件的数据时，才会触发新的请求。

## 问题：「高筛卡片视图 V2」是怎么建设的？为什么要引入独立数据通道？

### 需求背景

原来高筛只有一个表格视图，用的是 `/api/advance-search` 接口，返回的是工商维度的基础信息字段。产品要做一个**商机卡片视图**，后端提供了新接口 `/api/advance-search-v2`，返回额外的内容：

- 关键词命中摘要
- 商机时间排序
- 提取的关键词标签

这些字段表格接口是完全没有的。

### 第一个决策：独立数据通道

两个视图**各自请求对应接口，不能混用**。如果卡片刷新时用旧接口，返回的字段缺失会直接出 bug。

这个坑我在 `successCallBack` 里踩过——早期回调没区分视图类型，导致卡片视图拿到了表格接口的响应去渲染，字段全是 undefined。后来专门做了**按视图分发**的处理：

```js
successCallBack(data, viewType) {
  if (viewType === 'card') {
    this.handleCardData(data)   // 走卡片数据处理逻辑
  } else {
    this.handleTableData(data)  // 走表格数据处理逻辑
  }
}
```

### 第二个核心设计：双视图状态隔离

卡片和表格各有自己的一套完整状态，**完全分开**：

| 状态字段 | 表格视图 | 卡片视图 |
| --- | --- | --- |
| 当前页 | `currentPage` | `cardCurrentPage` |
| 每页条数 | `pageSize` | `cardPageSize` |
| 加载态 | `loading` | `cardLoading` |
| 是否已搜索 | `hasSearched` | `cardHasSearched` |
| 搜索标识 | `searchKey` | `cardSearchKey` |

统一访问抽象：

- **`getActivePage()`** / **`getActivePageSize()`**：调用方不用判断当前是哪种视图

切换视图时用 **`hasFreshCardData()`** 判断数据是否新鲜：新鲜就直接用缓存数据，不新鲜再发请求。这样来回切换不会重复打接口。

### 排序兼容历史逻辑

表格和卡片的排序参数结构完全不一样：

- **表格排序**：`{ contactNumber: 0 }` 这种 key-value 结构，存在 `this.sort`
- **卡片排序**：`business_sort` 商机时间排序，存在 `this.cardSortValue`

转参数时用各自的方法，确保两边排序互不影响。切换视图时不会把表格的排序参数带到卡片请求里，反之亦然。

### 权限开关灰度上线

这个功能不是一次全量上线的，而是通过**权限开关灰度**。

服务端在 `feature_permission` 里下发 `can_use_advance_search_v2` 字段，前端在父组件初始化时读一次。

**权限关闭时，三处联动保证完全降级：**

1. **视图切换 Tab 不渲染**：用户看不到切换入口
2. **初始视图强制表格**：`defaultView = 'table'`
3. **切换事件 early return**：即使通过其他途径触发切换，也直接 return 不执行

已有用户**零感知**，体验和之前完全一样。

**权限开启时：**

- 用户偏好按 `userId` 持久化到 `localStorage`
- 下次进来自动还原上次的视图选择
- 默认是卡片视图（新功能引导用户体验）

```
localStorage key: `filter_view_preference_${userId}`
value: 'card' | 'table'
```

### 整体流程图

```
初始化
  │
  ├── 读取 feature_permission.can_use_advance_search_v2
  │     │
  │     ├── false → 强制表格视图，隐藏切换 Tab
  │     │
  │     └── true → 读取 localStorage 用户偏好
  │                  │
  │                  ├── 有记录 → 还原上次视图
  │                  └── 无记录 → 默认卡片视图
  │
  ▼
用户切换视图
  │
  ├── hasFreshCardData() ?
  │     │
  │     ├── true → 直接展示缓存数据
  │     └── false → 请求对应接口
  │                   │
  │                   ├── 卡片 → /api/advance-search-v2
  │                   └── 表格 → /api/advance-search
  │
  ▼
successCallBack → 按 viewType 分发处理
  │
  ▼
持久化用户偏好到 localStorage
```

## 问题：卡片视图里的命中摘要高亮和折叠交互是怎么做的？

这个场景是卡片视图 V2 里的**命中摘要展示**。后端会返回关键词命中列表，每条命中里包含用 `<em>` 标签高亮的 HTML 片段，比如「2024年和<em>科技</em>公司签约」这样的文本，关键词部分带 `<em>` 包裹，需要在卡片上高亮显示。

问题有两个维度：**安全**和**交互**。

### 安全：最小必要 HTML 注入

不能直接把后端 HTML 塞进 `dangerouslySetInnerHTML`（XSS 风险），但也不能无脑用 `innerText` 把标签都当文本显示（高亮就没了）。

我的处理分两步：

**第一步：清洗截断实体**

先过一遍 `removeKnownCardHitEntities`，把后端经常出现的截断 HTML 实体（比如 `&#8226;`、`&#160;`）清洗掉，避免乱码。这些实体是后端做文本截断时切到了实体中间产生的。

**第二步：按需选择渲染方式**

```js
if (hasHtmlTags(content)) {
  // 确实含有 <em> 等标签 → 走 dangerouslySetInnerHTML
  return <span dangerouslySetInnerHTML={{ __html: cleanedContent }} />
} else {
  // 纯文本 → 走普通文本节点，零风险
  return <span>{content}</span>
}
```

用正则检测是否含有 HTML 标签，**只有确实需要渲染标签的内容才走 innerHTML**，不做过度信任。

**标题高亮是另一套逻辑**：企业名是结构化字段，客户端自己用正则切割关键词位置，再用 `<mark>` 标签渲染，完全不依赖服务端 HTML——这样即使后端没返回高亮标记，标题也能正确高亮。

### 交互：阈值折叠 + Popover 展开

命中条目可能非常多（几十条），全部展开会把卡片撑得很高，用户在列表页扫描效率很低。

**折叠阈值设计：**

| 区域 | 最大显示行数 | 超出处理 |
| --- | --- | --- |
| 关键词命中区 | 4 行 | 折叠 + Popover |
| 其它命中区 | 5 行 | 折叠 + Popover |

超出的折叠成一个 hover 触发的 **Popover**，显示「查看更多 N 条」。用户鼠标悬停就能看完所有命中，完全不需要进详情页再返回——这就是「降低无效详情点击成本」的含义。

**Popover 宽度自适应：**

Popover 的宽度要和卡片洞察区的实际渲染宽度对齐，我在 `onMouseEnter` 时用 `getBoundingClientRect()` 实时测量父容器宽度再 `setState`，避免 Popover 内容溢出或过窄：

```js
handleMouseEnter = (e) => {
  const parentWidth = this.insightRef.current?.getBoundingClientRect().width
  this.setState({ popoverWidth: parentWidth })
}
```

### 整体渲染流程

```
后端返回命中列表
  │
  ▼
removeKnownCardHitEntities() → 清洗截断实体
  │
  ▼
按阈值截断（关键词区 4 行 / 其它区 5 行）
  │
  ├── 可见部分 → hasHtmlTags 判断 → innerHTML 或纯文本渲染
  │
  └── 超出部分 → 折叠为 "查看更多 N 条"
                    │
                    └── hover → Popover（宽度 = getBoundingClientRect 实时测量）
                                  │
                                  └── 完整命中列表渲染
```

## 问题：模式切换（全部 / 可跟进 / 未解锁）是怎么做的？

模式切换就是顶部的 Tab 栏，它的选项是**动态生成**的。

### 动态适配逻辑

不同客户的 `lead_management_mode` 配置不同，比如：

| 模式 | 说明 |
| --- | --- |
| 纯探迹模式 | 只有探迹自有的 Tab 选项 |
| CRM 模式 | 只有 CRM 相关的 Tab 选项 |
| 两套并行模式 | 探迹 + CRM 的 Tab 都展示 |
| 钉钉嵌入模式 | 根据钉钉容器特性裁剪选项 |

生成的选项数量和文案都不一样，这个逻辑全在 **`getModeOptions`** 里做适配。

### 状态同步

`mode` 值会同步到两个地方：

1. **请求参数**：作为筛选条件的一部分下发给后端
2. **本地持久化**：存到 `localStorage`，刷新后恢复用户上次的选择

## 问题：多业务场景是怎么支持的？

完全靠 **prop 开关**来区分，框架层不感知具体业务逻辑。

### 金融场景

传 `isFinance: true`，请求就带上 `industry: 'financial'`，操作按钮也整体换成金融专属的「关注 / 解锁」。

### 地产热门版

传 `isEstateHotOpen`，会影响后端返回的筛选维度配置——地产场景有自己独特的维度（比如楼盘、开发商等），后端根据这个标识返回不同的维度列表。

### 工厂 / 渠道版

`isDoncusFinance` 透传给条件面板做**维度裁剪**——工厂渠道场景不需要某些通用维度，通过这个标识在面板层面直接隐藏。

### PaaS 嵌入场景

操作按钮的显示与否、文案，全部从 Redux store 的 **`paasConfig`** 里读取。嵌入方在远端配一下就能定制，不用改代码。

## 问题：EnterpriseFilter 的整体架构是什么样的？

```
┌─────────────────────────────────────────────────────┐
│                  EnterpriseFilter                     │
│  （编排层：数据流调度 + 生命周期管理）                  │
├─────────────────────────────────────────────────────┤
│                                                      │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────┐ │
│  │ ModeSelector │  │ DimensionPanel│  │ Toolbar   │ │
│  │ (模式切换Tab) │  │ (条件面板)    │  │ (操作栏)  │ │
│  └──────┬───────┘  └──────┬───────┘  └─────┬─────┘ │
│         │                  │                │       │
│         ▼                  ▼                ▼       │
│  ┌─────────────────────────────────────────────────┐│
│  │          Condition Assembler（条件组装器）        ││
│  │  serialize / deserialize / persist / share       ││
│  └─────────────────────┬───────────────────────────┘│
│                        │                             │
│                        ▼                             │
│  ┌─────────────────────────────────────────────────┐│
│  │          Request Layer（请求层）                  ││
│  │  mode + conditions + industry + pagination       ││
│  └─────────────────────┬───────────────────────────┘│
│                        │                             │
│                        ▼                             │
│  ┌──────────────┐  ┌──────────────┐                 │
│  │  CardView    │  │  TableView   │                 │
│  │  (卡片视图)   │  │  (表格视图)   │                 │
│  └──────────────┘  └──────────────┘                 │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 各层职责

- **EnterpriseFilter（编排层）**：不做具体业务逻辑，只负责把各模块串起来，管理数据流的流向和生命周期。
- **ModeSelector**：动态生成 Tab 选项，响应 `lead_management_mode` 配置。
- **DimensionPanel**：条件面板，接收业务场景 prop 做维度裁剪。
- **Toolbar**：操作工具栏，根据 `paasConfig` 或业务场景 prop 决定按钮的显隐和文案。
- **Condition Assembler**：条件的序列化、反序列化、持久化、口令生成与解析。
- **Request Layer**：统一的请求出口，组装最终参数并管理分页状态。
- **CardView / TableView**：两种视图各自维护分页，通过 `getActivePage` 统一对外暴露。

## 问题：DimensionSelector 的实现原理能具体说说吗？

DimensionSelector 本质是个**条件树编辑器**，它做了三件核心的事。

### 一、双向格式转换

条件在内部用 **Immutable.js List** 存储，每条记录的结构是：

```ts
{
  type: string       // 维度路径，如 "tungeeIndustries"
  option: object     // 维度配置（操作符类型、输入控件类型等）
  actionValue: string // 操作符，如 "in"、"not_in"、"exists"
  value: any         // 用户输入的值
}
```

**向外传出**时调 `transformFilterData`，把 Immutable List 转成 ES 查询风格的 JSON：

```json
{
  "must": [
    { "tungeeIndustries": [{ "in": ["IT", "金融"] }] }
  ]
}
```

**外部写入**时调 `convertValue`，把 ES 风格的 JSON 还原回 Immutable 状态，回填到 UI 控件。

这层转换让**存储格式和 UI 状态完全解耦**——后端只关心 ES 查询结构，前端只关心 Immutable 状态树，互不侵入。

### 二、两列正交的维度配置系统

每个维度有两个配置维度，全部来自后端下发：

| 配置维度 | 作用 | 示例 |
| --- | --- | --- |
| **ACTION_TYPE**（操作符类型） | 控制操作符下拉的选项 | "包含任一"、"不包含所有"、"有/没有" |
| **INPUT_TYPE**（输入控件类型） | 控制值输入区域渲染什么控件 | 多选树、关键词输入、数值范围 |

前端根据这两个值分别渲染对应的子组件：

- **ConditionAction**：根据 `ACTION_TYPE` 渲染操作符选择器
- **ConditionInput**：根据 `INPUT_TYPE` 渲染输入控件（树形多选、文本输入、范围滑块等）

**新增维度不用改代码**——只要后端在配置里加一条新维度，指定它的 ACTION_TYPE 和 INPUT_TYPE，前端自动就能渲染出来。

### 三、递归渲染的条件树

普通条件是扁平的一行一行，但条件组合（`type` 为 `subFilters`）会在渲染时**递归调用 `renderFilter`**，在 transform 时也**递归调用 `transformFilterData`**，这样就天然支持了嵌套的 AND/OR 逻辑组合。

```
条件组 (AND)
├── 行业 包含任一 [IT, 金融]
├── 营收 大于 1000万
└── 子条件组 (OR)
    ├── 地区 包含 [北京]
    └── 成立年限 大于 5年
```

### 关联维度

还有一种**关联维度**，是挂在某条主条件下的附属筛选行，存在 `correlationValue` 里。

比如用户选了「行业 = IT」之后，可以追加一个关联条件「上市状态 = 已上市」，实现精细化的组合筛选。关联维度在 UI 上表现为主条件行下方缩进的子行，在数据结构上是主条件对象的一个嵌套字段。

### 智能推荐集成

每次用户选了新维度，就把当前已选的维度集合发给**推荐接口**，返回的推荐维度会被动态插到配置树最前面，作为一个**虚拟大类**展示。

效果就是「越选越智能」——用户选了行业和地区之后，系统推荐「注册资本」「成立时间」这些高关联维度，降低用户的选择成本。

### 整体数据流

```
用户操作 UI
    │
    ▼
Immutable List（内部状态）
    │
    ├──→ transformFilterData() ──→ ES 查询 JSON ──→ 发给后端
    │
    └──→ persist to localStorage（持久化）
    
外部写入（口令还原 / 接口回填）
    │
    ▼
convertValue() ──→ Immutable List ──→ UI 自动更新
```

## 问题：条件口令功能具体是怎么实现的？

### 编码流程

1. 用户点击「生成口令」
2. 把当前所有筛选条件（维度选择、mode、自定义参数）序列化成 JSON
3. 对 JSON 做压缩编码（base64 + 简单混淆），生成一个短字符串
4. 复制到剪贴板

### 解码流程

1. 用户粘贴口令字符串到输入框
2. 解码还原出原始 JSON
3. 校验 JSON 结构合法性（防止篡改）
4. 把条件回填到 DimensionSelector 各个控件
5. 自动触发一次搜索

### 设计考量

- **版本兼容**：口令里带版本号，新版本能解析旧口令，旧版本遇到新口令给出友好提示
- **安全性**：口令不包含用户身份信息，只有筛选条件本身
- **容错**：解码失败不会崩溃，给出「口令无效」的提示

## 问题：localStorage 持久化有没有遇到什么坑？

有几个需要注意的点：

### 1. 存储 key 的隔离

不同业务场景、不同用户的条件要隔离存储，key 的设计是：

```
filter_${userId}_${sceneType}_conditions
filter_${userId}_${sceneType}_mode
```

### 2. 数据过期清理

条件数据不能无限堆积，设计了一个**过期时间戳**，超过 7 天的持久化数据在初始化时自动清理。

### 3. 序列化异常兜底

`JSON.stringify` 和 `JSON.parse` 都包了 try-catch，解析失败就用默认空条件，不会阻塞页面渲染。

### 4. 存储容量

筛选条件的数据量通常很小（几 KB），不会触发 localStorage 的 5MB 限制，但还是做了超限检测，超限时降级为仅内存态。

## 问题：这套框架的扩展性怎么体现？新业务接入的成本有多低？

举个实际例子：后来产品要加一个「人脉维度」的筛选，整个接入过程是这样的：

1. **新增一个 DimensionPlugin**：实现 `serialize()` 和 `deserialize()` 接口，告诉框架怎么把这个维度的选择转成请求参数
2. **在场景配置里注册**：把这个 plugin 加到对应场景的 `dimensions` 数组里
3. **完成**

框架层的 EnterpriseFilter、Condition Assembler、Request Layer 一行代码都不用改。分页、持久化、口令、视图切换这些能力自动就支持了新维度。

这就是插件式架构的好处——**开闭原则**，对扩展开放，对修改关闭。

## 问题：和后端的接口协议是怎么约定的？

### 请求结构

```json
{
  "conditions": {
    "dimension_1": { "operator": "in", "values": ["a", "b"] },
    "dimension_2": { "operator": "between", "values": [100, 500] }
  },
  "mode": "all",
  "industry": "financial",
  "page": 1,
  "pageSize": 20,
  "viewType": "card"
}
```

### 响应结构

```json
{
  "list": [...],
  "total": 1234,
  "dimensions": [
    { "key": "region", "label": "地区", "type": "multi-select", "options": [...] },
    { "key": "revenue", "label": "营收", "type": "range", "unit": "万元" }
  ]
}
```

### 关键设计

- **维度配置后端下发**：前端不硬编码有哪些筛选维度，全部由后端根据业务场景动态返回，前端根据 `type` 字段渲染对应的控件（多选、范围、日期等）
- **operator 标准化**：`in`、`not_in`、`between`、`gt`、`lt`、`like` 等，前后端统一枚举
- **viewType 透传**：后端可以根据视图类型返回不同粒度的数据（卡片视图返回摘要，表格视图返回完整字段）

## 问题：工程与性能优化具体做了哪些？

工程优化主要做了四个维度：路由懒加载、公共依赖治理、防抖、重复请求收敛。

### 一、路由懒加载

全站路由统一用 `getComponent` + 动态 `import()` 实现按需加载，每个路由只在真正进入时才加载对应代码。

有 reducer 和 sagas 的重量级页面用 **`Promise.all` 并行加载**，避免串行等待：

```js
getComponent(nextState, cb) {
  Promise.all([
    import('./reducer'),
    import('./sagas'),
    import('./index'),
  ]).then(([reducer, sagas, component]) => {
    injectReducer(store, reducer.default)
    injectSagas(sagas.default)
    cb(null, component.default)
  })
}
```

这样首屏只加载 app 壳和当前页面，其他路由的代码不会占用初始加载时间。

### 二、公共依赖治理（splitChunks）

在 webpack 配置里做了系统性的分组：

| 分组 | 内容 | 策略 |
| --- | --- | --- |
| **vendor** | react / immutable / moment | 框架层，变动频率极低，长期缓存 |
| **antd** | Ant Design 全家桶 | 单独一个包，方便客户端长期缓存 |
| **async-vendor** | echarts / g2 等重型图表库 | 仅在使用到的页面异步加载 |
| **async-biz** | 跨 3 个以上异步路由复用的业务代码 | 提取公共业务逻辑，减少重复 |

**moment 专项优化：**

1. 用 `ContextReplacementPlugin` 只打包 `zh-cn` 语言包，砍掉其余 100+ 语言包
2. 用 `alias` 锁定 moment 的解析路径，防止双份打包（不同版本被不同依赖引入）

**输出文件名用 `contenthash`**：内容不变哈希不变，客户端缓存可以长期命中，部署时只有真正改动的 chunk 才会让用户重新下载。

### 三、防抖

用在两个地方：

**1. 条件选择器搜索输入框**

300ms debounce，避免每次击键都触发过滤计算。用户打字过程中不会有任何计算开销，停下来 300ms 后才执行一次过滤。

**2. AI 推荐接口请求**

请求前做**维度 diff**——把当前已选维度集合和上次请求时的集合做对比，集合不变就不重复发请求。避免用户只是修改了某个维度的值（而没有增减维度）时触发无意义的推荐刷新。

### 四、重复请求收敛（三层防护）

| 层级 | 机制 | 解决的问题 |
| --- | --- | --- |
| 第一层 | **loading 状态拦截** | 搜索请求进行中时再点搜索直接 return，防止并发重复请求 |
| 第二层 | **searchKey 指纹去重** | 把 filter/page/sort/mode 全序列化成一个字符串，视图切换时对比，数据新鲜就跳过请求 |
| 第三层 | **RuntimeCache 内存缓存** | 高筛结果保存在内存里，用户从详情页返回时直接从缓存还原 |

**第一层：loading 拦截**

```js
handleSearch() {
  if (this.state.loading) return  // 请求进行中，直接拦截
  // ... 发起请求
}
```

**第二层：searchKey 指纹**

```js
const searchKey = JSON.stringify({ filter, page, sort, mode })
if (searchKey === this.lastSearchKey) return  // 条件没变，跳过
this.lastSearchKey = searchKey
```

这就是前面提到的「新鲜度判断」的底层实现——视图切换时对比这个字符串，数据新鲜就跳过请求。

**第三层：RuntimeCache**

高筛结果保存在内存缓存里，用户从详情页返回时通过 URL 的 `fromCache` 标志直接从缓存还原，**完全跳过网络请求**：

```js
componentDidMount() {
  if (this.props.location.query.fromCache && RuntimeCache.has('filterResult')) {
    this.restoreFromCache(RuntimeCache.get('filterResult'))
    return
  }
  this.doSearch()
}
```

这对复杂筛选条件下的用户回退体验帮助很大——用户花了很长时间配好条件、翻了好几页找到目标企业、点进详情看完之后按返回，如果没有缓存就要重新请求、重新等待，条件和页码还可能丢失。有了 RuntimeCache，返回瞬间恢复，条件、页码、滚动位置全部保留。
