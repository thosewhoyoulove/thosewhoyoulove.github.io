# 项目架构的整体升级方案

## 面试定位

这类项目最适合用来证明你具备架构演进能力。面试官关注的不是“你把 Vue 2 升到了 Vue 3”，而是你为什么升级、怎么拆风险、如何验证收益、遇到兼容问题怎么处理。

回答时要突出三点：

1. 升级不是追新，而是为了解决真实痛点。
2. 大项目升级要分阶段，不能一次性全改。
3. 每一步都要有验证标准和回滚意识。

## 核心原理

架构升级的本质是“在不影响业务稳定性的前提下，逐步替换落后的技术底座”。

合理路径通常是：

```text
识别痛点 -> 评估收益和风险 -> 拆分阶段 -> 小步迁移 -> 回归验证 -> 灰度上线 -> 清理历史兼容
```

这篇项目可以重点讲“构建工具迁移”和“框架版本迁移”分开做。这样可以把问题边界拆开，避免出了问题不知道是构建层、框架层还是业务代码导致的。

---

## 问题：为什么要做这次升级？背景是什么？

项目原来是 Vue 2 + Vue CLI 的技术栈，跑了两三年，积累了几个痛点：

1. **开发体验差**：Vue CLI 基于 webpack，冷启动要 30-40 秒，改一行代码热更新也要 2-3 秒，开发效率很低
2. **维护成本高**：Vue 2 生态逐渐停止维护，很多依赖库只出 Vue 3 版本的新功能
3. **代码组织混乱**：Options API 在复杂组件里逻辑分散，一个功能的代码散落在 data、methods、computed、watch 各处，维护困难
4. **类型支持弱**：Vue 2 对 TypeScript 的支持是后加的，类型推导经常断裂

综合评估后决定做一次整体升级：构建工具从 Vue CLI 迁移到 Vite，框架从 Vue 2 升级到 Vue 3。

---

## 问题：升级策略是怎么定的？为什么不一步到位？

我定的策略是**分两步走**：先迁构建工具，再升框架版本。

### 为什么不同时做

同时换构建工具和框架版本，出了问题很难定位是 Vite 的兼容问题还是 Vue 3 的 API 变更导致的。分开做的好处是每一步都有明确的验证标准——第一步完成后项目还是 Vue 2，但构建用 Vite，功能完全不变就算成功；第二步再升 Vue 3，构建层已经稳定了，只需要关注框架层面的问题。

### 落地节奏

1. **第一阶段**：Vue CLI → Vite（构建工具迁移，不动业务代码）
2. **第二阶段**：Vue 2 → Vue 3 + @vue/compat 兼容模式（跑通全量功能）
3. **第三阶段**：逐步去掉 compat，新功能用 Composition API，老代码按模块渐进重构

---

## 问题：Vue CLI 迁移到 Vite 具体怎么做的？遇到了什么坑？

### 核心步骤

1. 删掉 `vue-cli-service`，装 `vite` + `@vitejs/plugin-vue2`（因为第一阶段还是 Vue 2）
2. 把 `vue.config.js` 的配置翻译到 `vite.config.js`
3. 入口从 `public/index.html` 改成项目根目录的 `index.html`，加上 `<script type="module" src="/src/main.js">`
4. 更新 `package.json` 的 scripts

### 踩过的坑

**1. 别名配置差异**

Vue CLI 里 `@` 默认指向 `src`，Vite 里要手动配：

```js
resolve: {
  alias: {
    '@': path.resolve(__dirname, 'src')
  }
}
```

还有一些业务代码里用了 `~` 前缀引用 node_modules 里的样式，Vite 不认这个语法，要批量替换。

**2. 环境变量前缀**

Vue CLI 用 `VUE_APP_` 前缀，Vite 用 `VITE_` 前缀，而且访问方式从 `process.env.VUE_APP_XXX` 变成 `import.meta.env.VITE_XXX`。这个改动涉及面很广，我写了个脚本批量替换。

**3. CommonJS 依赖兼容**

Vite 开发模式走 ESM，但很多老的 npm 包只有 CommonJS 格式。大部分情况 Vite 的[依赖预构建](/md/面试准备/技术/依赖预构建.md)能自动处理，但有几个包预构建失败，需要手动加到 `optimizeDeps.include` 里强制预构建。

**4. 动态 require 不支持**

Vite 不支持 `require()`，项目里有些地方用了动态 require 加载图片或 JSON，全部要改成 `import()` 或 `import.meta.glob`。

### 验证标准

迁移完之后跑了一遍全量功能回归，确保页面渲染、接口请求、路由跳转全部正常。冷启动从 35 秒降到 1-2 秒，热更新从 2-3 秒降到毫秒级，开发体验提升非常明显。

## 问题：Vue 2 升级 Vue 3 是怎么做的？

### 第一步：引入 @vue/compat 兼容包

Vue 3 官方提供了 [@vue/compat](/md/面试准备/技术/vue兼容包的作用.md) 兼容包，它能让 Vue 3 运行时兼容大部分 Vue 2 的 API。配置方式：

```js
// vite.config.js
export default {
  resolve: {
    alias: {
      vue: '@vue/compat'
    }
  }
}
```

开启后，Vue 2 的废弃 API 还能用，但控制台会打出 deprecation warning，告诉你哪些地方需要改。这样可以**先跑通，再逐步修**，不用一次性改完所有代码。

### 第二步：升级配套生态

| 依赖 | Vue 2 版本 | Vue 3 版本 | 注意事项 |
| --- | --- | --- | --- |
| vue-router | 3.x | 4.x | 路由守卫写法变了，`beforeRouteEnter` 里拿不到 this |
| vuex | 3.x | 4.x | 基本兼容，但推荐后续迁移到 Pinia |
| element-ui | 2.x | element-plus | 组件 API 有变化，需要逐个检查 |
| vue-i18n | 8.x | 9.x | 初始化方式完全不同 |

### 第三步：处理破坏性变更

**生命周期重命名：**
- `beforeDestroy` → `beforeUnmount`
- `destroyed` → `unmounted`

**事件 API 移除：**
- Vue 3 移除了 `$on`、`$off`、`$once`，项目里用 EventBus 的地方全部要换成 mitt 或自己实现一个发布订阅

**v-model 变更：**
- Vue 2 的 `.sync` 修饰符没了，统一用 `v-model:propName` 替代

**过滤器移除：**
- `{{ value | formatDate }}` 这种写法不支持了，改成方法调用或 computed

### 第四步：渐进式重构到 Composition API

不是一次性全改，而是定了个规则：
- **新功能**：必须用 Composition API + `<script setup>`
- **Bug 修复**：改到哪个组件，如果逻辑不复杂就顺手重构
- **复杂组件**：排优先级，按业务模块逐步重构

重构的核心收益是**逻辑内聚**——同一个功能的响应式数据、方法、生命周期钩子放在一起，用 composable 函数抽取复用逻辑，比 mixin 清晰很多。

## 问题：升级过程中最大的难点是什么？

### 1. 第三方组件库迁移

Element UI 到 Element Plus 不是简单换个包名，很多组件的 API、事件名、slot 写法都变了。我的做法是先全局搜索用到的组件列表，按使用频率排序，高频组件优先适配，低频的先用 compat 模式兜着。

### 2. 全局 API 的变更

Vue 2 里大量用 `Vue.prototype.$xxx` 挂全局方法，Vue 3 改成了 `app.config.globalProperties.$xxx`。项目里有十几个全局挂载，要逐个迁移。

### 3. 自定义指令的适配

Vue 3 的指令钩子名称全部对齐了组件生命周期：
- `bind` → `beforeMount`
- `inserted` → `mounted`
- `unbind` → `unmounted`

项目里有几个自定义指令（权限指令、埋点指令），都要改。

### 4. 测试回归

升级完跑了完整的回归测试，重点关注：
- 路由跳转和守卫逻辑
- 表单校验和提交
- 弹窗/抽屉的生命周期（之前用 `destroyed` 做清理的地方容易漏）
- 第三方组件的事件绑定

---

## 问题：升级的收益是什么？

| 维度 | 升级前 | 升级后 |
| --- | --- | --- |
| 冷启动 | 35s（webpack） | 1-2s（Vite） |
| 热更新 | 2-3s | <100ms |
| 打包体积 | - | Tree-shaking 更彻底，减少约 20% |
| 类型安全 | 弱（Vue 2 + TS 推导经常断） | 强（Vue 3 原生 TS 支持） |
| 代码组织 | Options API，逻辑分散 | Composition API，逻辑内聚 |
| 生态支持 | 停止维护 | 活跃维护 |

最直观的感受是开发效率提升了很多，改代码的反馈循环从秒级变成了毫秒级，而且 Composition API 让复杂组件的可维护性好了一个量级。

---

## 追问：有些第三方插件升级不了怎么办？

实际升级过程中确实遇到过这种情况——某个老旧的图表库或树形组件不兼容 Vue 3，官方已经断更了。我的处理策略是按复杂度从低到高排列，优先选成本最低的方案。

### 策略一：寻找平替方案（成本最低，首选）

Vue 3 生态现在非常成熟，大部分 Vue 2 的老插件都有对应的 Vue 3 版本或更强的替代品。

**实际例子：** 如果 vue-echarts 老版本跑不动，直接换成官方 ECharts 原生调用或者 vue-echarts 的新版本。

评估标准是：替代品的 API 差异大不大、迁移工作量多少、长期维护是否更健康。大部分情况下，换一个活跃维护的库虽然短期有迁移成本，但长期维护压力小很多。

### 策略二：局部混用——Iframe / 微前端隔离

如果这个插件极其复杂（比如一个大型的行业报表插件），重写要花一个月，那就把它**隔离起来**，让它在老环境里继续跑。

**两种做法：**

- **Iframe**：把旧插件留在原先的 Vue 2 环境里，通过 Iframe 嵌入到新项目，用 `postMessage` 通信
- **微前端**：如果公司有微前端架构（如乾坤 qiankun），让这个旧页面跑在子应用里，主应用升级 Vue 3

本质是「空间换时间」——用架构隔离的方式让新旧框架并存，优先保证业务上线，后续再找时间重构。

### 策略三：利用 @vue/compat 的组件级兼容

compat 包提供了 `configureCompat` 配置，可以针对**特定组件**声明它依然运行在 Vue 2 模式下：

```js
import { configureCompat } from 'vue'

configureCompat({
  // 针对特定特性开启 Vue 2 兼容
  COMPONENT_V_MODEL: true,
  RENDER_FUNCTION: true,
})
```

这不能包治百病，但能修补一些断裂的 API 链条，让插件先跑起来，争取时间做后续替换。

### 策略四：直接改源码（最硬核）

如果插件逻辑不复杂，只是个别地方报错（比如访问了 `this.$scopedSlots`、`$on`、`$listeners`），那就自己动手修。

**做法：**

1. 把插件代码从 `node_modules` 拷出来，放进项目的 `src/components` 里，按 Vue 3 语法手动修复报错
2. 如果必须留在 `node_modules`，用 **patch-package** 打补丁：

```bash
# 手动修改 node_modules 里的源码后
npx patch-package <package-name>
```

这样每次 `npm install` 后会自动应用补丁，团队其他人也能同步。

**典型修复：**
- `this.$scopedSlots` → `this.$slots`（Vue 3 合并了两者）
- `this.$listeners` → 直接从 props 里接收事件
- `this.$on / $off` → 用 mitt 替代

### 总结：面试怎么回答

> "如果遇到实在升不上的插件，我不会硬刚。首先看社区有没有更好的替代品，长痛不如短痛。如果没有，我会评估这个插件的复杂度——逻辑简单的，直接拷贝源码出来自己改，用 patch-package 搞定；逻辑太复杂的，采用'空间换时间'的办法，用 Iframe 或微前端把它包起来，让它在老环境里继续跑，确保新老业务平稳对接。核心原则就是：**先保业务运行，再分批次重构**。"

## 面试回答

这个项目我会从“为什么升级”和“怎么控风险”两个角度讲。

当时项目是 Vue 2 + Vue CLI，主要问题是开发体验差、构建慢、生态逐渐老化、TypeScript 支持弱，并且复杂组件里 Options API 导致逻辑分散。我的判断是需要做技术底座升级，但不能一次性把构建工具和框架都换掉，否则问题难定位、回归风险也高。

所以我把升级拆成三步：第一步先从 Vue CLI 迁移到 Vite，业务代码基本不动，只验证构建层稳定性；第二步再通过 `@vue/compat` 把 Vue 2 迁到 Vue 3 兼容模式，先保证全量功能可运行；第三步按模块消除兼容警告，新功能使用 Composition API，老模块逐步重构。

迁移过程中重点处理了别名、环境变量、CommonJS 依赖、动态 require、第三方组件库、全局 API、自定义指令和路由状态管理这些兼容问题。最终冷启动从 35 秒降到 1 到 2 秒，热更新从秒级降到毫秒级，同时 Vue 3 和 Composition API 也提升了复杂组件的可维护性。

## 高频追问

### 为什么不直接一次性升级到 Vue 3？

因为同时更换构建工具和框架版本，风险会叠加，问题也很难定位。

分阶段迁移可以让每一步都有明确验证目标：先验证构建层，再验证框架兼容，最后再做业务代码重构。

### @vue/compat 是不是长期方案？

不是。它只是迁移过渡工具。

目标是先让项目跑起来，再根据兼容警告逐步清理 Vue 2 API，最终移除兼容包，切到标准 Vue 3。

### 升级过程中最容易出问题的地方是什么？

第三方组件库、全局 API、自定义指令、EventBus、过滤器、`v-model`、路由和状态管理。

这些地方影响范围大，需要优先梳理使用清单和回归范围。

### 如何证明架构升级是有价值的？

不能只说“技术更先进”。要用指标证明。

比如启动时间、热更新时间、构建时间、包体积、类型错误减少、组件复用提升、后续需求开发效率提升。
