# 综合能力 & 团队合作

## 面试定位

这类问题没有标准答案，面试官想看的是你的沟通方式、冲突处理、质量意识、取舍能力和复盘能力。

回答时建议使用 STAR 结构：

```text
Situation：当时是什么背景。
Task：你要解决什么问题。
Action：你做了什么。
Result：最终结果是什么。
```

但不要机械背 STAR，要让回答听起来像真实经历。

## 核心原理

综合能力题的核心不是“态度好”，而是“能在不确定和冲突中推进结果”。

优秀回答通常有这些特点：

- 不甩锅。
- 不贬低其他角色。
- 能讲清楚约束条件。
- 能说明取舍依据。
- 有结果或复盘。
- 能体现主动性。

## 技术分歧怎么解决？

### 面试回答

如果团队里出现技术分歧，我不会先判断谁对谁错，而是先回到目标。

比如我们要解决的是性能问题、交付速度问题，还是长期维护问题。目标不同，方案选择也不同。然后我会把几个方案的成本、收益、风险和可逆性列出来。如果分歧比较大，会做一个小范围 PoC，用数据辅助判断，比如包体积、首屏时间、开发成本、维护复杂度。

最后确定方案后，我会把决策记录下来，包括为什么选这个方案、放弃了什么、后续如果要演进怎么做。这样后面复盘时也有依据。

### 高频追问

#### 如果对方坚持自己的方案怎么办？

先确认对方担心的点是什么，是性能、风险、时间，还是维护成本。

如果争论无法收敛，就用数据或小范围验证来判断。必要时让技术负责人基于事实做决策。

#### 你有没有真实遇到过？

可以准备一个和简历项目相关的例子，比如状态管理方案、是否引入新库、CSR 和 SSR 取舍、组件抽象粒度、接口缓存策略。

## 你有过 Code Review 经验吗？

### 面试回答

有。我理解 Code Review 不只是检查代码格式，而是保证质量和共享上下文。

我通常会看几个方面：逻辑是否正确，边界条件有没有处理，异常场景有没有兜底，是否有性能或安全风险，代码是否容易维护，是否影响公共模块，以及关键路径有没有测试。

提意见时我会尽量说明原因，而不是只说“这里不好”。如果是风格问题，就尽量交给工具自动处理；如果是设计问题，会给出替代方案或示例，避免 Review 变成单纯挑错。

### 高频追问

#### Review 和开发效率会冲突吗？

短期看会增加一点时间，但能减少返工和线上问题。

我的做法是小步提交、小范围 Review，让每次变更可读。大型变更先做方案评审，避免代码写完后才发现方向不对。

#### Review 发现严重问题怎么办？

先确认是否影响正确性、安全、性能或线上稳定性。

如果影响核心链路，就阻止合并并补充测试。如果只是可维护性问题，可以评估是否本次修改，还是记录为后续优化。

## 需求有多种实现方式时如何决策？

### 面试回答

我会先看约束条件，包括时间、人力、兼容性、稳定性、后续扩展概率。

然后比较不同方案的成本、收益、风险和可逆性。短期需求不一定要做最复杂的架构，长期核心能力也不能只做临时方案。最后会选择最符合当前阶段的方案，并明确后续演进空间。

比如一个组件抽象，如果只有两个页面用，我可能先保持简单；如果已经覆盖多个业务线，并且变更频繁，就应该抽公共配置、统一状态和测试用例。

### 高频追问

#### 怎么避免过度设计？

看复用是否真实存在，变化方向是否明确，抽象是否能降低复杂度。

如果只是为了“看起来高级”而抽象，反而会增加理解成本。

#### 怎么避免临时方案变成技术债？

临时方案要有边界、有记录、有后续计划。

比如标注影响范围、加监控、补文档，并在后续迭代里安排治理。

## 遇到过最有挑战性的问题，如何解决？

### 面试回答

我会按“现象、定位、方案、结果、复盘”来讲。

先说问题现象，比如页面卡顿、首屏慢、线上报错、接口超时或状态错乱。然后说明我怎么定位，比如看日志、复现路径、Performance 面板、网络请求、错误监控。接着讲方案和取舍，最后给结果数据，以及后续如何防止再次发生。

这类问题一定要结合真实项目。不要只讲一个算法题或工具使用，要体现定位过程和工程闭环。

### 高频追问

#### 没有量化指标怎么办？

可以讲验证方式。

例如通过复现用例、日志对比、Performance 录制、用户反馈、测试回归、线上错误数量来证明问题被解决。

## 跨角色协作要注意什么？

### 面试回答

和不同角色协作时，我会先对齐他们最关心的目标。

和产品协作时，重点是需求边界、验收标准和优先级。和后端协作时，重点是接口契约、错误码、超时、幂等和联调窗口。和测试协作时，重点是测试范围、异常场景、可测性和回归重点。和设计协作时，重点是交互细节、响应式、空状态和异常状态。

跨角色协作最怕信息不对称，所以我会把关键结论沉淀到文档、任务卡或评审记录里。

### 高频追问

#### 产品临时改需求怎么办？

先确认变更原因和优先级，再评估影响范围。

如果影响工期，要同步成本和风险，给出保核心、延期低优先级、拆版本上线等方案。

#### 后端接口延期怎么办？

可以先用 mock、接口契约和联调排期降低阻塞。

同时把依赖风险同步出来，避免前端最后背锅。
