# 团队协作与主导能力

## 面试定位

这类问题主要考察你是不是只会写代码，还是能在真实团队里推动事情落地。

面试官会重点看：

1. 你是否能拆解目标。
2. 你是否能识别风险。
3. 你是否能协调跨角色资源。
4. 你是否能用结果证明推进有效。

回答时建议结合真实项目，不要只说“我积极沟通、按时完成”。

## 核心原理

团队协作的本质是把不确定的目标变成可交付的计划。

一个成熟的协作回答通常包含：

- 目标：要解决什么业务问题。
- 拆解：如何拆任务、定边界。
- 协调：如何和产品、后端、测试、设计配合。
- 风险：如何提前识别阻塞。
- 质量：如何保证代码、测试、发布质量。
- 结果：最终交付了什么，有什么收益。

如果只讲“大家一起开会”，会显得很空。要讲你在关键节点做了什么判断。

## 你是怎么带领团队开发核心产品的？

### 面试回答

我会先把目标拆清楚，而不是直接进入开发。

在需求阶段，我会和产品确认业务目标、验收标准和边界，避免开发中途反复变更。然后根据模块复杂度和人员熟悉度拆分任务，把核心链路、公共组件、接口联调、测试验证这些部分拆成可交付节点。

开发过程中，我会通过看板或任务列表跟踪进度，对依赖后端、设计稿变更、技术难点这些风险提前标出来。如果某个模块可能影响整体排期，我会尽早同步，而不是等到最后一天暴露问题。

质量上，我会推动 Code Review、关键路径测试、上线 checklist 和必要的灰度验证。最终交付后，会根据线上反馈、错误监控或业务指标做复盘。

### 高频追问

#### 你怎么证明是你主导的？

不要只说“我是负责人”，要说清楚你主导了哪些关键动作。

例如：需求拆解、技术方案、模块分工、进度跟踪、风险协调、上线验证、复盘总结。

#### 遇到延期风险怎么办？

先判断延期来源，是需求变更、依赖阻塞、技术难点，还是评估不准。

然后给出方案：砍范围、拆优先级、调整人力、先交付核心链路、把低优先级能力延后。

## 项目如何保证质量？

### 面试回答

我会从开发前、开发中、上线前三个阶段保证质量。

开发前先明确验收标准、接口契约和异常场景。开发中通过类型约束、静态检查、Code Review、单元测试或关键路径自测来减少问题。上线前会走测试回归、发布 checklist、灰度验证和回滚预案。

如果是核心模块，我会优先覆盖关键路径测试，而不是追求表面覆盖率。上线后还会关注错误监控、性能指标和用户反馈，确保问题能被及时发现。

### 高频追问

#### 测试覆盖率越高越好吗？

不一定。

覆盖率只是辅助指标，更重要的是核心路径和高风险逻辑有没有被覆盖。低价值测试可能增加维护成本。

#### Code Review 主要看什么？

重点看边界条件、异常处理、可维护性、性能风险、安全风险、是否符合团队规范，以及是否有必要的测试。

## 如何分配任务与跟踪进度？

### 面试回答

我会先按业务链路和技术模块拆任务，再结合成员经验分配。

任务不能只写“开发页面”，而要拆到可验收粒度，比如接口联调完成、核心交互完成、异常态完成、测试回归完成。对风险高或周期长的任务，我会设置中间检查点，避免最后才发现偏差。

进度跟踪上，我更关注阻塞和风险，而不是每天问“做完了吗”。如果发现依赖不清、方案不稳或范围变更，会及时和相关角色对齐。

### 高频追问

#### 成员能力差异比较大怎么办？

核心链路交给更熟悉的人，低风险任务给新人，同时配合文档、结对和 Review。

新人不是只做边角料，也可以从清晰边界的小模块开始，逐步提高参与度。

## 新人 onboarding 或知识传承怎么做？

### 面试回答

我会把新人上手拆成环境、业务、代码和流程四部分。

环境上提供启动文档和常见问题。业务上给核心流程图和主要角色说明。代码上说明目录结构、模块边界、公共组件和接口规范。流程上说明分支、提交、Review、测试和发布规则。

前几次任务会选择边界清楚的小需求，并安排 Review 或结对，帮助新人理解团队标准。关键技术决策会沉淀到文档，避免只靠口头传递。

### 高频追问

#### 文档没人看怎么办？

文档要和流程绑定。

比如新人入职必须走环境文档，方案评审必须补充设计文档，复杂问题复盘后必须更新 FAQ。文档只有进入工作流程，才不会变成摆设。

## 与更高层预期不一致时怎么办？

### 面试回答

我会先把分歧从“观点不同”转成“目标、成本、风险不同”。

比如上级希望更快上线，我会拆出核心范围和非核心范围，给出几个方案：最快方案、稳妥方案、长期方案。每个方案说明交付时间、风险、后续成本和对业务目标的影响。

最终让决策者基于事实选择，而不是只讨论感觉。确定方案后，我会同步团队调整计划，并把范围和风险记录下来。

### 高频追问

#### 如果你不同意最终决策怎么办？

如果决策已经明确，我会按团队目标推进，但会把已识别的风险、监控点和兜底方案准备好。

不同意不等于消极执行，成熟的做法是保留专业判断，同时保证交付。
