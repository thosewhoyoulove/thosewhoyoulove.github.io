# 高频追问清单

## 面试定位

这份清单用于面试前自测。它不是单纯的问题列表，而是用来检查你是否能承受连续追问。

每个问题建议按这个结构回答：

```text
结论 -> 原理 -> 项目场景 -> 边界取舍
```

如果只能背出概念，但讲不出项目里怎么用，面试官很容易判断为“只背过，没有实践”。

## 核心原理

面试追问通常不是随机的，而是沿着一条链路往下挖。

例如问 Promise：

```text
Promise 是什么
-> 状态如何流转
-> then 为什么是异步
-> 微任务和宏任务顺序
-> 如何手写 Promise.all
-> 项目里怎么处理并发请求和错误兜底
```

所以复习时不要只看单个答案，要把相关知识串起来。

## 面试回答

我会用追问清单来判断自己是不是真的准备好了。

每个问题我不会只背结论，而是会按“结论、原理、项目场景、边界取舍”来组织。比如问首屏优化，我会先说优化方向，再解释浏览器加载和渲染链路，然后结合项目里做过的代码分包、资源压缩、缓存、懒加载或接口优化，最后说明这些方案的代价，比如分包过细会增加请求调度成本，缓存策略不当会带来更新问题。

这样准备的好处是，即使面试官连续追问，也可以沿着原理和项目经验继续展开，而不是只停留在背答案。

## 高频追问

### 清单里的问题都要背下来吗？

不需要逐字背。重点是每个问题都能讲出结构化答案。

至少要做到：能讲结论、能解释原理、能举项目场景、能说取舍。

### 遇到不会的问题怎么办？

不要硬编。

可以先说自己了解的边界，再说明会怎么排查或学习。例如：“这个点我没有在项目里深入落地过，但我知道它主要解决的是……如果在项目里遇到，我会先从……验证。”

### 怎么把八股和项目连起来？

每个技术点都补一句项目场景。

比如缓存不要只讲强缓存和协商缓存，还要讲静态资源 hash、HTML 不强缓存、接口缓存策略、版本更新问题。

### 面试前最后一天看什么？

优先看项目表达、追问清单、容易忘的基础原理。

不要在最后一天大规模学新内容，重点是把已有内容说顺。
