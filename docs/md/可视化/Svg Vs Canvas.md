# SVG 与 Canvas

## 面试定位

SVG 和 Canvas 常用于可视化、图表、大屏项目面试。回答重点是渲染模型、性能差异、交互方式、缩放清晰度和适用场景。

## 核心原理

SVG 是基于 XML/DOM 的矢量图形，每个图形元素都是 DOM 节点。

Canvas 是一块位图画布，通过 JavaScript API 在画布上绘制像素，绘制完成后浏览器不再保留每个图形元素的 DOM 结构。

## 对比

| 维度 | SVG | Canvas |
| --- | --- | --- |
| 渲染模型 | 矢量 DOM 节点 | 位图像素绘制 |
| 元素结构 | 每个图形是节点 | 单个画布 |
| 交互 | 原生支持元素事件 | 需要手动命中检测 |
| 缩放 | 矢量无损 | 可能模糊，需要处理 DPR |
| 大数据性能 | 节点多时变差 | 大量点位更适合 |
| 更新方式 | 可局部改 DOM | 通常重绘画布 |

## 适用场景

适合 SVG：

- 图形数量较少。
- 需要元素级事件。
- 需要无损缩放。
- 流程图、编辑器、图标。
- 导出 SVG/PDF。

适合 Canvas：

- 大量数据点。
- 高频重绘。
- 实时大屏。
- 游戏、粒子、热力图。
- 图表性能优先。

## ECharts 为什么支持双渲染

ECharts 同时支持 SVG 和 Canvas，是为了兼顾不同场景：

- 默认 Canvas 更适合大数据量和高频更新。
- SVG 更适合小数据量、高交互、需要矢量导出的场景。

```javascript
const chart = echarts.init(dom, null, {
    renderer: "svg", // 或 "canvas"
});
```

## 面试回答

可以这样答：

> SVG 是矢量图形，每个图形都是 DOM 节点，优点是缩放清晰、元素事件天然支持，适合图形数量少但交互复杂的场景。Canvas 是位图画布，通过 JS 绘制像素，浏览器不保留每个图形元素的 DOM，优点是大量图形和高频重绘性能更好，适合大数据图表、实时大屏、游戏等。选择时，如果需要元素级交互和矢量导出，我倾向 SVG；如果数据量大、更新频繁，我倾向 Canvas。

## 高频追问

### Canvas 怎么做点击某个图形？

需要自己维护图形数据，监听画布事件后根据鼠标坐标做命中检测，或者使用 ECharts/ZRender 这类库封装好的事件系统。

### Canvas 在高清屏为什么会模糊？

Canvas 的实际像素尺寸和 CSS 尺寸不一致时会被拉伸。需要根据 `devicePixelRatio` 放大画布实际尺寸，再缩放绘制上下文。

### SVG 节点多为什么卡？

每个图形都是 DOM 节点，节点数量大时 DOM 创建、布局、事件和更新成本都会上升。
