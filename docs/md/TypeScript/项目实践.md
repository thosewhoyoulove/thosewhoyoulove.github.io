# TypeScript 项目实践

## 面试定位

这道题主要考察你是否真的在项目里用过 TypeScript，而不是只会语法。

面试官通常会追问：你们项目怎么接入 TypeScript、接口类型怎么管理、组件 props 怎么写、遇到 `any` 怎么治理、老项目怎么渐进迁移。

## 核心原理

TypeScript 在项目里的价值不是“所有地方都写类型”，而是让关键边界更清晰。

常见边界包括：

1. 接口请求和响应。
2. 组件 props 和事件。
3. 业务状态和枚举。
4. 工具函数输入输出。
5. 第三方库和全局对象扩展。

只要这些边界清楚，项目协作成本和重构风险都会下降。

## 接口类型管理

接口类型通常按业务模块管理。

```ts
type User = {
  id: number;
  name: string;
  role: "admin" | "user";
};

type ApiResponse<T> = {
  code: number;
  message: string;
  data: T;
};

type UserResponse = ApiResponse<User>;
```

这样既复用了接口外层结构，也保留了具体业务数据类型。

在真实项目里，接口类型可以手写，也可以由后端接口文档、OpenAPI、Swagger 自动生成。自动生成能减少前后端结构不一致的问题。

## 组件类型设计

组件类型重点是 props、事件、插槽或 children。

```ts
type SelectOption<T> = {
  label: string;
  value: T;
};

type SelectProps<T> = {
  options: SelectOption<T>[];
  value: T;
  onChange: (value: T) => void;
};
```

对于通用组件，泛型可以让组件既复用，又保留准确的值类型。

业务组件则不一定要写得很抽象，优先保证可读和稳定。

## 状态类型设计

复杂状态适合用可辨识联合类型。

```ts
type RequestState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; message: string };
```

这样可以避免“data 可能为空、error 可能为空、loading 又是 true”的状态混乱。

状态越复杂，越应该让类型表达清楚合法状态，而不是用多个布尔值拼接。

## any 治理

项目里完全没有 `any` 不现实，但应该控制它出现的位置。

比较合理的做法是：

1. 外部输入先用 `unknown`。
2. 通过类型守卫或运行时校验确认结构。
3. 内部业务代码尽量不扩散 `any`。
4. 对暂时无法处理的 `any` 加注释说明来源。

`any` 最大的问题不是单点存在，而是沿着函数参数、返回值、状态继续扩散，最后让整条链路失去类型保护。

## 老项目迁移

老项目迁移 TypeScript 应该渐进进行。

常见路径是：

1. 先允许 JS 和 TS 共存。
2. 从工具函数、公共组件、接口层开始改。
3. 对高频变更模块优先补类型。
4. 逐步收紧配置，比如减少隐式 `any`。
5. 最后再处理低频页面和历史代码。

不要一开始就追求一步到位，否则成本高，也容易影响业务迭代。

## 面试回答

我在项目里使用 TypeScript 时，重点不是给每一行代码都加类型，而是先把关键边界定义清楚。

比如接口层会定义统一的 `ApiResponse<T>`，业务数据用具体类型承接；组件层会把 props、事件回调、通用组件泛型设计好；复杂状态会用可辨识联合类型表达，比如 `loading`、`success`、`error`，避免多个布尔值组合出非法状态。

对于 `any`，我一般不会让它在业务代码里扩散。外部数据如果暂时不确定，会先用 `unknown`，再通过类型守卫或运行时校验收窄。老项目迁移时，我会采用渐进方式，优先改接口层、公共组件、工具函数和高频改动模块，再逐步收紧配置。

所以 TypeScript 在项目里的价值是提升协作、重构和边界表达能力，而不是单纯追求类型覆盖率。

## 高频追问

### 老项目怎么接入 TypeScript？

先让 JS 和 TS 共存，不要一次性全量迁移。

优先改公共工具、接口层、公共组件和高频业务模块。等主要边界稳定后，再逐步收紧类型检查规则。

### 接口类型手写还是自动生成？

小项目可以手写，大项目更推荐基于 OpenAPI 或接口文档自动生成。

自动生成可以减少前后端字段不一致的问题，但也要注意生成类型和业务二次封装之间的边界，不要让后端结构直接污染前端所有页面。

### 项目里能不能用 any？

可以少量使用，但不能让它扩散。

比如某些第三方库类型不完整、历史代码迁移中暂时无法补全，可以临时用 `any`。但接口入口、业务状态、组件 props 这些关键位置应该尽量避免。

### TypeScript 能不能替代接口数据校验？

不能。TypeScript 类型在编译后会被擦除，不会在运行时检查数据。

接口返回、表单输入、本地缓存这些外部数据，仍然需要运行时校验。TypeScript 只能保证开发阶段的类型约束。
