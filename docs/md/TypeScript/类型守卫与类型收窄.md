# 类型守卫与类型收窄

## 面试定位

这道题主要考察你是否理解 TypeScript 如何从宽泛类型推导到具体类型。它常和联合类型、`unknown`、接口返回处理、组件 props 分支渲染一起考。

回答时要讲清楚：类型守卫是手段，类型收窄是结果。

## 核心原理

类型收窄是指 TypeScript 根据控制流分析，把一个较宽的类型缩小成更具体的类型。

```ts
function format(value: string | number) {
  if (typeof value === "string") {
    return value.toUpperCase();
  }

  return value.toFixed(2);
}
```

在 `if` 分支里，TypeScript 知道 `value` 是 `string`；在后面的分支里，`value` 就被收窄为 `number`。

类型守卫就是触发这种收窄的判断方式。

## 常见类型守卫

### typeof

`typeof` 适合判断基础类型。

```ts
function handle(value: string | number) {
  if (typeof value === "string") {
    return value.trim();
  }

  return value.toFixed(2);
}
```

### instanceof

`instanceof` 适合判断类实例。

```ts
function handleDate(value: Date | string) {
  if (value instanceof Date) {
    return value.getTime();
  }

  return Date.parse(value);
}
```

### in

`in` 适合判断对象是否存在某个属性，常用于联合类型对象。

```ts
type Admin = {
  role: "admin";
  permissions: string[];
};

type User = {
  role: "user";
  level: number;
};

function getInfo(account: Admin | User) {
  if ("permissions" in account) {
    return account.permissions;
  }

  return account.level;
}
```

### 字面量判别字段

在业务代码里，更推荐使用固定字段作为判别标识。

```ts
type State =
  | { status: "loading" }
  | { status: "success"; data: string[] }
  | { status: "error"; message: string };

function render(state: State) {
  switch (state.status) {
    case "loading":
      return "loading";
    case "success":
      return state.data.join(",");
    case "error":
      return state.message;
  }
}
```

这种模式叫可辨识联合类型，适合表达接口状态、业务状态、组件渲染状态。

## 自定义类型守卫

自定义类型守卫通过 `value is Type` 告诉 TypeScript：如果函数返回 `true`，这个值就是某个具体类型。

```ts
type User = {
  id: number;
  name: string;
};

function isUser(value: unknown): value is User {
  return (
    typeof value === "object" &&
    value !== null &&
    "id" in value &&
    "name" in value
  );
}

function handle(value: unknown) {
  if (isUser(value)) {
    return value.name;
  }

  return "";
}
```

自定义类型守卫适合处理接口返回、缓存读取、本地存储、第三方 SDK 数据等来源不稳定的数据。

## 面试回答

类型守卫是用来帮助 TypeScript 判断具体类型的条件判断方式，类型收窄是判断之后得到的结果。

比如一个参数是 `string | number`，在 `typeof value === "string"` 分支里，TypeScript 会把它收窄成 `string`，于是可以安全调用字符串方法。常见的类型守卫包括 `typeof`、`instanceof`、`in`、字面量判别字段，以及自定义类型守卫。

实际项目里，我比较常用的是可辨识联合类型和自定义类型守卫。可辨识联合类型适合表达业务状态，比如 `loading`、`success`、`error`。自定义类型守卫适合处理外部数据，比如接口返回或本地缓存读取，因为这些数据虽然在 TypeScript 看起来有类型，但运行时仍然可能不符合预期。

所以类型守卫的价值是让宽泛类型在使用前变得更具体、更安全，减少错误访问属性或错误调用方法的问题。

## 高频追问

### 类型守卫和类型断言有什么区别？

类型守卫是通过判断让编译器自然收窄类型，类型断言是开发者直接告诉编译器“我认为它是什么类型”。

类型守卫更安全，类型断言更像绕过检查。除非非常确定，否则不应该滥用类型断言。

### as 能不能替代类型守卫？

不能完全替代。

`as` 只影响编译期判断，不会做运行时校验。如果数据本身不符合结构，运行时仍然会报错。类型守卫通常会包含真实判断逻辑，更适合处理不可信数据。

### 为什么推荐可辨识联合类型？

因为它让业务分支更清晰，也更容易做穷尽检查。

每个类型都有一个固定的判别字段，比如 `status` 或 `type`。代码根据这个字段分支后，TypeScript 能准确知道当前分支对应的数据结构。

### unknown 为什么经常配合类型守卫？

因为 `unknown` 不能直接访问属性或调用方法，必须先判断类型。

这正好符合外部数据的处理方式：先把数据当成未知，再通过类型守卫确认结构，最后安全使用。
