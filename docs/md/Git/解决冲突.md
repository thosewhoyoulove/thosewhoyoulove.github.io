# Git 解决冲突

## 面试定位

Git 冲突题考察团队协作经验。回答重点是：先确认冲突范围，再理解双方改动意图，最后解决、测试、继续合并或 rebase。

## 核心原理

冲突通常发生在两个分支修改了同一文件的同一位置，Git 无法自动判断应该保留哪部分。

冲突标记：

```text
<<<<<<< HEAD
当前分支内容
=======
合入分支内容
>>>>>>> feature
```

## 处理流程

1. 查看状态。

```bash
git status
```

2. 打开冲突文件，理解双方改动。

3. 手动编辑，删除冲突标记，保留正确结果。

4. 标记已解决。

```bash
git add <file>
```

5. 继续流程。

如果是 merge：

```bash
git commit
```

如果是 rebase：

```bash
git rebase --continue
```

6. 运行测试或本地验证。

## 预防冲突

- 小步提交，减少大 PR。
- 经常同步主分支。
- 公共文件、配置文件改动提前沟通。
- 代码模块边界清晰。
- 避免多人长期改同一个大文件。

## 面试回答

可以这样答：

> 遇到 Git 冲突我会先 `git status` 看冲突文件，再打开文件查看 `<<<<<<<`、`=======`、`>>>>>>>` 标记。解决时不是简单选自己的或别人的，而是理解两边改动目的，必要时找相关同事确认。编辑成最终正确代码后 `git add` 标记已解决；如果是 merge 就提交 merge commit，如果是 rebase 就 `git rebase --continue`。解决后一定会跑相关测试或启动项目验证，避免语法上解决了但业务逻辑被破坏。

## 高频追问

### rebase 冲突和 merge 冲突有什么区别？

解决文件冲突的方式类似，但 rebase 是逐个重放提交，可能一个提交解决完后后续提交还会继续冲突。

### 不想继续合并怎么办？

merge 可以用 `git merge --abort`，rebase 可以用 `git rebase --abort`。

### 怎么减少冲突？

保持分支短生命周期，频繁同步主干，拆小 PR，公共代码改动提前沟通。
